# Research: CLI/MCP "Type" Terminology Consistency

## D1: `internal/core.Filter.Kinds` renames to `Types`, `matchKinds` to `matchTypes` ā€” no other change

**Decision**: `Filter.Kinds []string` becomes `Filter.Types []string`; the unexported `matchKinds` method
becomes `matchTypes`; both keep their exact current body (`node.Type == k` comparison, empty-matches-every
short-circuit). `Filter.Match`'s call site updates from `f.matchKinds(node)` to `f.matchTypes(node)`.

**Rationale**: `Filter.Kinds` already compares against `node.Type` (confirmed in the current source ā€” the
field's own doc comment already says "a node matches if its Type is any listed value"), so the field was
never actually about a "kind" concept distinct from "type"; the name was simply never updated when the
node-shape migration (spec 010) replaced `Kind` with `Type` everywhere else. Renaming it removes the last
place in `internal/core` where a reader has to know "Kinds means Types" as tribal knowledge. `Attrs` and
`AttrPatterns` are untouched ā€” they were never named after "kind" and already match against `node.Attrs`'s
`[]Predicate` values correctly (confirmed via `attrStrings`/`matchAttrValue`/`matchAttrPattern`, which already
unwrap `Predicate.Value`); this feature does not touch them.

**Alternatives considered**:
- Leave `Filter.Kinds` named as-is and only rename the CLI-visible `--kind` flag, translating at the
  `optsFilter.build()` boundary ā€” rejected: this would leave exactly the same "kind means type" tribal
  knowledge one layer down instead of removing it, and contradicts spec FR-008 ("no single command may mix
  'kind' and 'type' vocabulary for the same underlying concept") in spirit even if not in literal CLI output.
- Rename to `NodeTypes` or `TypeFilter` for extra clarity ā€” rejected (YAGNI): every sibling field (`Tags`,
  `Attrs`, `AttrPatterns`) already uses the bare concept name with no qualifying prefix; `Types` matches that
  existing convention exactly.

## D2: MCP `node_grep` filter's wire field renames from `kind` to `type`, JSON tag included

**Decision**: `mcpFilter.Kind []string \`json:"kind,omitempty"\`` becomes `mcpFilter.Type []string
\`json:"type,omitempty"\``; `toCoreFilter`'s `Kinds: append([]string(nil), f.Kind...)` becomes `Types:
append([]string(nil), f.Type...)`.

**Rationale**: This is the only remaining externally-facing wire contract still spelled "kind" (confirmed:
`cmd/arc/graph/serve.go`'s `mcpFilter` struct and its one call site). **Correction discovered during
implementation (T012)**: this decision originally assumed plain `encoding/json` unrecognized-field tolerance
would apply, matching spec.md's original Edge Cases wording ("no error, but that criterion has no effect").
That assumption does not hold ā€” `mcp.AddTool` (`github.com/modelcontextprotocol/go-sdk`) validates every
tool's arguments against a JSON Schema generated by `jsonschema.ForType`, which sets
`additionalProperties: false` on every struct-derived object schema unconditionally, for every tool this
server registers (`node_get`, `node_grep`, `subgraph_get` alike) ā€” this predates this feature and is not
something a hand-rolled schema override should special-case away just for `mcpFilter`. A client still sending
`"kind"` therefore gets a tool-call error identifying it as an unrecognized property, before the handler (and
`encoding/json`) ever runs. spec.md's Edge Cases entry and this contract's Ā§3 were corrected to match this
verified behavior rather than the codebase being changed to fit the original (incorrect) assumption ā€” the
corrected behavior is in fact more consistent with FR-010's CLI-side no-alias treatment, not less.

**Alternatives considered**:
- Accept both `kind` and `type` as inputs, coalescing whichever is present ā€” rejected: spec Assumptions
  explicitly rules out keeping a backward-compatible alias, and accepting two field names for one criterion
  reintroduces the exact "two names, one concept" inconsistency this feature exists to remove.
- Hand-craft a permissive `additionalProperties: true` schema override for `mcpFilter` (via a custom
  `Tool.InputSchema` bypassing `mcp.AddTool`'s reflection) to force the originally-assumed silent-ignore
  behavior ā€” rejected: this would special-case `node_grep`'s filter argument against every other tool
  argument in this server's own convention (all struct-derived, all strict), duplicate the schema by hand
  with the drift risk that implies, and expand this feature's scope beyond a rename into adapter-shape
  behavior change territory that plan.md's Constitution Check explicitly disclaims ("adapter's shape is
  unchanged beyond the wire field rename").

## D3: Warning and help-text wording ā€” mechanical noun swap, checked for grammatical fit

**Decision**: `apply.go`'s warning becomes `"%s is not a recognized node type for this graph ā€” auto-registered
with a default schema document"` (only "kind"ā†’"type", rest unchanged). `grep.go`/`subgraph.go`'s flag
description becomes `"Restrict to nodes of this type (repeatable, OR)"`; `Long` help text's `--kind/--tag/--attr`
becomes `--type/--tag/--attr` and the `<kind>` row-format placeholder becomes `<type>`; `Example` blocks'
`--kind source`/`--kind entity` become `--type source`/`--type entity`; `subgraph.go`'s `--stubs` description's
"kind and id only" becomes "type and id only". Every checked string already reads correctly with a direct
one-word substitution ā€” no sentence needed restructuring for grammar (spec.md's Edge Cases anticipated this
possibility, but it did not materialize in the actual current strings).

**Rationale**: Confirmed by reading every current occurrence (`cmd/arc/graph/grep.go` lines 40/46/57/199-200/206;
`cmd/arc/graph/subgraph.go` lines 97/99/107/183; `internal/app/graph/service/apply.go` line 225) ā€” none require
pluralization or article changes beyond the noun itself, since "kind"/"type" are both uncountable-in-context
singular nouns in every sentence they appear in here.

**Alternatives considered**: None ā€” this is a direct, mechanical text substitution with no design choice
beyond confirming (above) that it doesn't produce awkward phrasing anywhere it's applied.

## D4: No `schemaVersion` marker, no `--json` schema change

**Decision**: This feature makes no change to any `--json` output's field names or shape. `ApplyResult`,
`GrepResult`, and `SubgraphResult`'s JSON tags were confirmed to already say `"created"`/`"merged"`/`"type"`
(never `"kind"`) ā€” the Go struct fields these renamed surfaces touch (`Filter.Kinds`ā†’`Types`,
`mcpFilter.Kind`ā†’`Type`) are internal/wire *input* shapes, not output-*Result* JSON payloads. No
`schemaVersion` marker is introduced, since no output schema is changing.

**Rationale**: Directly grounded in reading the current source (`internal/app/graph/kernel/apply.go`,
`grep.go`, `subgraph.go`) ā€” confirms spec.md's own Assumptions section ("the `--json` output field names
produced by `apply`, `grep`, and `subgraph` are already correct... and are unaffected by this feature").

**Alternatives considered**: N/A ā€” this decision is a scope confirmation, not a design choice between
alternatives; it forecloses work an earlier (superseded) technical-approach draft assumed was needed.
