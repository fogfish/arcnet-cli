# Bug Report: Core schema seed data (`CoreTypeDefs`) omits predicates ARCNET-CORE §10 permits on every/multiple types, causing `arc lint`'s new Requires/Optional checks to false-positive on spec-conformant graphs

**Type**: Implementation drift (primary) + Spec gap (for `scoreZ`/`scoreC` only — see Resolution)
**Severity**: High
**Feature**: `014-lint-spec-conformance` (branch `014-lint-spec-conformance`) — bug surfaces *through* this feature but the defect lives in `internal/app/schema/kernel.CoreTypeDefs`, seed data introduced by `011-machine-readable-schema`
**Reported**: 2026-07-09
**Status**: Resolved
**Patched**: 2026-07-09
**Resolved**: 2026-07-10 — T034-T043 implemented and verified: `go test ./...` and `staticcheck ./...` clean; the originally-reported repro (an `entity` node using `conformsTo`) confirmed fixed against a real `arc init`-seeded graph; all six `quickstart.md` scenarios re-verified with no regression.

## Description

`arc lint` itself behaves correctly per its own spec (014) — `checkTypeRequires`/`checkTypeOptional` faithfully enforce whatever a type's `_schema/types/<name>.md` document declares under `## Requires`/`## Optional`. The bug is that the *seed data* those documents are rendered from (`kernel.CoreTypeDefs` in `internal/app/schema/kernel/schema.go`) does not actually list every predicate ARCNET-CORE's own specification (`https://raw.githubusercontent.com/fogfish/arcnet-spec/refs/heads/main/ARCNET-CORE.md`) permits on each type. Before 014 shipped, `arc lint` never checked a node's predicates against `## Requires`/`## Optional` at all, so this gap in the seed data was invisible. Now that 014's `typeRequires`/`typeOptional` checks run on every `arc lint` invocation, a graph authored exactly per the spec's own worked examples gets **false-positive violations**.

**Confirmed by direct reproduction** (see Root Cause Analysis) against a fresh `arc init`-seeded graph: an `entity` node using the `conformsTo` semantic predicate (`ARCNET-CORE.md` §10.5, §11.3's own worked example) is rejected:

```text
entities/Transport Layer Security.md:10 — [typeOptional] predicate "conformsTo" is not permitted by type "entity" (not listed under its Requires or Optional)
```

The user's original report breaks this into five items, verified one by one against the actual spec text (fetched and read directly, not summarized) at commit-time of `arcnet-spec`'s `main` branch:

1. **Content predicates (`tags`, `text`) should be permitted on every type.** Spec §10.2 gives these no `Used by:` restriction, unlike §10.7's explicitly type-scoped predicates — structurally, §10.1–§10.6 predicates are cross-cutting/universal, only §10.7/§10.8 are type-restricted. **Confirmed as a real gap by structural reading of the spec**, though the spec's own worked `_schema/types/*.md` examples (§11.2–§11.5) don't literally list `tags`/`text` under every type's `## Optional` (e.g. `resource`/`timeline` list neither).
2. **Metadata/control predicates `published`, `created`, `updated` should be permitted on every type; `indexed` should also exist.** Same structural argument as (1) applies to `published`/`created`/`updated` (§10.3, no `Used by:` restriction). ~~`indexed` does not exist anywhere in `ARCNET-CORE.md` — confirmed by exhaustive grep of the fetched document; this is either a genuine spec gap, a term from a different/newer spec document, or a local-only extension the user wants added regardless of upstream spec.~~ **Corrected — see Resolution below**: `indexed` is not a spec gap at all; it's a real predicate `specs/009-node-timestamp-attrs` already defines and `arc apply` already stamps on every node, simply never registered in `kernel.CorePredicateDefs`.
3. **`scoreZ`/`scoreC` control predicates are missing; `score-z`/`score-c` (hyphenated) must be renamed.** **Confirmed absent from `ARCNET-CORE.md` entirely** — no heading, no inline mention, nothing (exhaustive case-insensitive grep for "score" found zero matches in §9/§10/§11/anywhere). Also confirmed **absent from the current Go codebase** as a registered predicate — `score-z`/`score-c` appear only as illustrative attribute names in old bug reports (`specs/003-apply-patch/bugs/BUG-003.md`, `BUG-004.md`) describing a hypothetical "graph-analytics score" `entity` attribute for `MergeUnion` conflict-handling test fixtures, never as an actual `_schema/predicates/` entry. **This needs explicit clarification before a fix can proceed** (see Recommended Fix).
4. **Structural predicates (`mentions`, `mentionedIn`) should be permitted on all core classes**, not just `source`(`mentions`)/`entity`(`mentionedIn`) as today. Spec §10.4 annotates each with a specific `from → to` pair (`source → entity` / `entity → source`), which argues *against* universal availability — this is a design tension between the user's request and the spec's own directional framing that needs a decision, not just a data fix.
5. **All Semantic predicates (§10.5: `broader`, `narrower`, `isPartOf`, `hasPart`, `requires`, `replaces`, `isReplacedBy`, `conformsTo`, `related`) should be permitted on entities and resources.** **Confirmed as an unambiguous, direct gap** — `ARCNET-CORE.md` §11.3's own worked `entity` type-schema example literally ends its `## Optional` list with `- any §10.5 semantic predicate, as applicable`, and `entity` is also missing `notes` (§10.7, listed in the spec's own `entity` Optional example) — `kernel.CoreTypeDefs["entity"].Optional` currently contains only `["aliases", "tags"]`, omitting `notes` and all nine semantic predicates. Whether `resource` should *also* list these under its own `## Optional` (making a `resource` node able to *originate* a semantic edge, not just be a valid *target* of one authored on an `entity`) is the user's request but is **not directly supported by the spec's own §10.5 prose** ("written as `edge`-role predicates in the entity body") — needs a decision.

## Artifact Traceability

### spec.md (014-lint-spec-conformance)

- **Affected user story**: User Story 2 ("Catch a node carrying a predicate its type doesn't permit") — FR-002/FR-003. The check itself is correct per spec.md; the *data* it validates against (schema seed) is wrong.
- **Affected requirements**: FR-001/FR-002 indirectly — both assume `index.Types[type].Required`/`.Optional` are a *complete and accurate* reflection of ARCNET-CORE's own conformance rules. spec.md's own Assumptions section says as much: *"A type schema's `## Requires`/`## Optional` sections are themselves well-formed... validating the type schema's own internal well-formedness beyond what's needed to run these node-level checks is out of scope for this feature."* — i.e. 014 explicitly assumed the seed data was already correct and declared validating it out of scope. That assumption was wrong for `entity`'s Optional list.
- **Gap identified**: No requirement in spec.md 014 covers keeping `CoreTypeDefs` synchronized with the upstream ARCNET-CORE specification — that responsibility was never assigned to any spec. This bug report is the first place it surfaces.

## Root Cause Analysis

Two independent root causes:

1. **Incomplete `Optional` seed data (confirmed, unambiguous)**: `kernel.CoreTypeDefs["entity"].Optional` is `["aliases", "tags"]`; ARCNET-CORE.md §11.3's own worked type-schema example lists `["aliases", "tags", "notes"]` plus "any §10.5 semantic predicate, as applicable." This was never validated by any test (011's own test suite only checks `Required`, not `Optional`) and was invisible before 014 shipped (lint never read `Optional` before). **Reproduced directly**: a fresh `arc init` graph, with an `entity` node carrying `conformsTo:: [[...]]` exactly as the spec's own `Transport Layer Security` worked example shows, produces a `[typeOptional]` false positive under the just-shipped `arc lint`.

2. **Ambiguous scope for "universal" predicates (needs a decision, not just a data fix)**: ARCNET-CORE.md structurally distinguishes §10.1–§10.6 (no per-predicate `Used by:` restriction — arguably intended to be available on any type) from §10.7/§10.8 (explicit `Used by:` restriction). The *worked examples* in §11, however, don't literally list every §10.1–§10.6 predicate under every type's `## Optional` — e.g. `resource`/`timeline` list neither `tags` nor `text`, and `mentions`/`mentionedIn` carry an explicit directional `from → to` annotation that argues against blanket availability across all four core types. `scoreZ`/`scoreC`/`indexed` don't appear in the spec at all. This is not a bug in the code — it's an open design question the user's report raises that the spec document itself doesn't unambiguously resolve, and needs an explicit decision (see below) before item 1's fix can be scoped to "every type" vs. "exactly what the spec's worked examples show."

## Resolution (patched 2026-07-09)

Further investigation resolved every open question below without needing to go back to the user — new
evidence turned up for both `indexed` and the "universal predicate" scope question:

- **`indexed` is not actually a spec gap** — it's a real, already-in-use predicate this report initially
  mischaracterized. `specs/009-node-timestamp-attrs/spec.md` FR-004/FR-006 defines it precisely (an ISO
  8601 second-resolution timestamp, stamped once at node creation, never modified thereafter), and
  `internal/app/graph/service/apply.go` already unconditionally stamps it
  (`setAttr(merged.Attrs, "indexed", stamp)`) on every non-stub, non-schema node `arc apply` creates — it
  was simply never added to `kernel.CorePredicateDefs`. Registered as `role: meta`, `merge: immutable`
  (matching FR-006 exactly), `aligned: "arc:indexed"` (tool-native, not an upstream-spec term).
- **Scope for items 1/2/4 (universal predicates)**: resolved as broad/universal exactly as the user's
  original report stated ("allowed to all classes/type" / "allowed to all core classes") — the user's
  wording was already unambiguous on this point; the "open question" in the original report was about
  whether the *spec's own worked examples* supported that scope as literally written, not about what the
  user wanted. Implemented as stated.
- **`scoreZ`/`scoreC`**: confirmed absent from `ARCNET-CORE.md` entirely (genuine spec gap, not
  implementation drift). Added as a local/tool-native extension per the user's explicit request —
  `role: meta`, `merge: validatedOverwrite`, `aligned: "arc:scoreZ"`/`"arc:scoreC"` — reasoned from
  historical bug-report context (`specs/003-apply-patch/bugs/BUG-003.md`/`BUG-004.md` describe `score-c`/
  `score-z` as recomputed by an ingest pipeline on every run, which matches CORE §9.3's own description of
  `validatedOverwrite` almost verbatim). Full reasoning in `research.md` D10.
- **`resource`'s semantic predicates**: the user's original report already explicitly said "allowed to all
  entities and resources" — implemented for both, per that explicit request, despite ARCNET-CORE §10.5's
  own prose reading slightly narrower ("written... in the entity body"). Full reasoning in `research.md`
  D8's table.

See `research.md`'s "Bugfix: BUG-001" section (D8–D10) for the complete, concrete per-type/per-predicate
decision table, `spec.md` FR-014–FR-020/SC-007 for the corresponding requirements, and `tasks.md`'s new
"Bugfix: BUG-001" section (T034-T043) for the implementation tasks.
