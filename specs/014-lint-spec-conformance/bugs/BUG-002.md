**Reported**: 2026-07-10
**Status**: Resolved

## Resolution

The reporter was asked to resolve item 4's naming conflict (`entries` vs. `cites`) and chose to **reuse
`cites`**, replacing `entries` outright. Consequences worked through and recorded in `research.md`'s
"Bugfix: BUG-002" section (D12): `cites`'s own `merge` behavior changes from `union` to `append` to
preserve the chronological-ordering guarantee `entries` carried; `CorePredicateDefs["entries"]` is removed
entirely (no real graph ever produced a genuine `entries`-tagged edge, so nothing depends on the name
remaining registered); `cites`'s `Description` is broadened to cover both its pre-existing usage (a
source's own citation of an external resource) and its new one (a timeline's chronological reference to a
source it contains). This is recorded as a deliberate product decision diverging from
`ARCNET-CORE.md` ¬ß11.5's own naming, not a spec-conformance fix ‚Äî spec.md's FR-022 says so explicitly.

Items 1 (`period`) and 3 (parser trailing-annotation gap) proceed exactly as originally recommended ‚Äî see
spec.md FR-021/FR-023/FR-024 and research.md D11/D13/D14 for the concrete implementation decisions.

## Description

Every timeline period file `arc apply` produces (`timeline/yearly/<YYYY>.md`, `timeline/monthly/<YYYY-MM>.md`) fails `arc lint` with exactly the errors the user reported:

```text
‚ùå timeline/yearly/2026.md:1 ‚Äî [typeRequires] type "timeline" requires predicate "entries", but this node does not carry it
‚ùå timeline/yearly/2026.md:4 ‚Äî [typeOptional] predicate "period" is not permitted by type "timeline" (not listed under its Requires or Optional)
```

**Reproduced directly**, using the user's own example content: a fresh `arc init` graph, `arc apply`'d with a patch for `dmitry-2026-graph` ("Ontologies, Graph Structures, and LLM-Based Knowledge Management", Dmitry, 2026-07-03), produces:

```text
---
"@id": "2026"
"@type": timeline
period: "2026"
granularity: yearly
---
# 2026

- [[dmitry-2026-graph]] ‚Äî *Ontologies, Graph Structures, and LLM-Based Knowledge Management* (Dmitry) ‚Äî 2026-07-03
```

...and `arc lint` reports both errors verbatim, on both the yearly and monthly period files.

Unlike BUG-001, this is **not a schema-seed-data gap** ‚Äî `kernel.CoreTypeDefs["timeline"]` already correctly lists `"entries"` under `Required` (matching ARCNET-CORE ¬ß11.5's own worked example) ‚Äî it never listed `"period"`, but that's the smaller half of this bug. The real defect is that the code producing timeline files (`internal/app/graph/service/apply.go`'s `upsertTimelinePeriod`, backed by `internal/core/timeline.go`'s `TimelineEntry`) **never actually writes an `entries`-tagged edge at all** ‚Äî every entry is rendered as a bare, untyped `[[wikilink]]` with a trailing display annotation, by *deliberate* design (`internal/core/timeline.go`'s own doc comment: *"the timeline node's own Edges carry only the bare target"*, locked in by `TestTimelineEntry`). That design predates `CoreTypeDefs["timeline"].Required` declaring `entries` mandatory (spec 011) and predates this feature's new `typeRequires` check ever validating it ‚Äî so the mismatch between "the schema says every timeline node requires `entries`" and "the code that writes timeline files never produces one" went unnoticed until now, exactly the same discovery pattern as BUG-001.

There is a second, independent, more serious defect discovered while investigating: **the exact annotated-bullet format ARCNET-CORE ¬ß11.5's own worked example shows is silently unparseable** by `internal/core.ParseNode` today. Feeding it the spec's own literal example line ‚Äî

```markdown
- entries:: [[rescorla-2026-tls13]] ‚Äî *TLS 1.3: Design and Rationale* (Eric Rescorla) ‚Äî 2026-04-12
```

‚Äî produces a `Node` with **zero `Edges` and zero `HRefs`** for that line (confirmed by direct reproduction against `core.ParseNode`): the entire bullet is silently dropped, not merely misclassified. The regex `internal/core/markdown.go`'s `listItemPattern` uses (`^(?:(\w+)::\s*)?\[\[([^\]|]+?)(?:\|([^\]]+?))?\]\]$`) anchors `$` immediately after the wikilink's closing `]]`, rejecting any trailing text ‚Äî even though `internal/core`'s own *generic* renderer (`renderEdges`/`renderLinkBullet`) never produces trailing annotation text in the first place (only `upsertTimelinePeriod`'s bespoke writer does), the moment a predicate-tagged, annotated line reaches the parser ‚Äî whether hand-authored per the spec's own documented convention, or once `upsertTimelinePeriod` is fixed to actually write `entries::` ‚Äî it is **silently lost**, not flagged. This is the user's third reported item ("Linter fails to process an edge with annotation"), and it is more accurately a shared-AST-parser gap (`internal/core`, spec 010) than a lint-specific one ‚Äî `arc lint` itself never sees the edge to check because `core.ParseNode` already dropped it upstream.

The user additionally proposes renaming `entries` to `cites` for this relationship. **This conflicts with the authoritative external specification** (`github.com/fogfish/arcnet-spec`'s `ARCNET-CORE.md`, fetched and read directly, not summarized): ¬ß11.5 names the predicate `entries` explicitly, and `cites` is already a distinct, existing predicate (¬ß10.6) for an unrelated relationship ‚Äî a `source`'s own outbound citation to an external `resource` (`## Cites` block, `cites:: [[RFC 8446]]`). Reusing `cites` for timeline membership would collide with its existing, different meaning. This sub-point is flagged as a conflict needing an explicit decision, not silently implemented either way (see Root Cause Analysis and Recommended Fix).

## Root Cause Analysis

Three related but distinct defects, discovered together because they all manifest as the same two `arc lint` error lines:

1. **`period` never registered (confirmed, unambiguous ‚Äî same shape as BUG-001's `indexed`)**: `upsertTimelinePeriod` writes `period: "<value>"` to every timeline node's front matter (deliberately, per spec 003 BUG-007's fix ‚Äî a quoted string so a bare 4-digit yearly period doesn't decode as a YAML integer), but `kernel.CorePredicateDefs`/`CoreTypeDefs["timeline"]` never registered it. The user's own recommended fix (register it, `## Requires`) is correct and low-risk.

2. **`entries` is required by schema but never actually written as a predicate-tagged edge (confirmed, unambiguous)**: `TimelineEntry` renders a bare `[[id]] ‚Äî *title* (authors) ‚Äî date` bullet with no `entries::` prefix, by original design (its own doc comment says so), predating `CoreTypeDefs["timeline"].Required`'s later declaration that every timeline node requires `entries`. The fix is to make `TimelineEntry`/`upsertTimelinePeriod` actually emit `entries:: [[id]] ‚Äî *title* (authors) ‚Äî date`, matching ARCNET-CORE ¬ß11.5's own worked example and what `CoreTypeDefs` has claimed since spec 011.

3. **The predicate-tagged, annotated-bullet format is silently unparseable (confirmed by direct reproduction, most severe)**: fixing #2 alone would not be enough ‚Äî once `entries::` is prefixed, `internal/core/markdown.go`'s `listItemPattern` (`^(?:(\w+)::\s*)?\[\[([^\]|]+?)(?:\|([^\]]+?))?\]\]$`) still rejects the line because of its trailing ` ‚Äî *title* (authors) ‚Äî date` text after the wikilink's closing `]]`. Confirmed directly: feeding `core.ParseNode` the literal line from ARCNET-CORE ¬ß11.5's own worked example produces zero `Edges` and zero `HRefs` for it ‚Äî **silent data loss**, not a lint-detectable violation. This is a gap in the shared AST parser (`internal/core`, spec 010's own domain), not lint-specific, even though the user (correctly, from an observed-symptom standpoint) attributed it to "the linter."

4. **Not a defect, but a genuine conflict needing a decision**: the user's suggestion to use `cites` instead of `entries` for this relationship contradicts ARCNET-CORE ¬ß11.5 (which names it `entries`) and would collide with `cites`'s own already-distinct, already-implemented meaning (¬ß10.6, a source's citation of an external resource). Implementing this literally would require either overriding the authoritative external spec or renaming/aliasing the existing `cites` predicate's semantics ‚Äî neither should happen silently.
