# Feature Specification: Full ARCNET-CORE §16 Conformance Checks for `arc lint`

**Feature Branch**: `014-lint-spec-conformance`

**Created**: 2026-07-09

**Status**: Draft

**Input**: User description: "Extend `arc lint` to validate a graph against ARCNET-CORE v0.7's actual conformance checklist (§16), which today's lint only partially covers. Today's lint does not check whether a node actually carries every predicate its type requires, nor whether a node carries a predicate its type doesn't permit at all. The new spec requires `arc lint` to additionally check, for every node: every predicate listed in its type's \"## Requires\" section is present on the node; every predicate present on the node that isn't in \"## Requires\" is listed in \"## Optional\" (or is one of the two universal identity predicates); the \"@id\"/\"@type\" front-matter keys are present and correctly quoted; a citation predicate's validity is checked against the schema's registered predicates whose alignment is a cito: vocabulary term, rather than a hardcoded Go list, so a domain profile can register additional citation predicates that get recognized automatically; and a node's declared predicates match their schema-declared role (e.g. a predicate registered with role \"text\" should not appear as a body edge, and vice versa). Every new check must report a clear rule name, the offending file and line where determinable, and a human-readable message, consistent with arc lint's existing violation format and its \"collect everything, don't stop at first failure\" behavior. Out of scope: auto-fixing violations; changes to arc apply's own behavior."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Catch a node that's missing a predicate its type requires (Priority: P1)

A graph maintainer's extraction pipeline (or a hand-edited node) produces a node of a registered type but omits a predicate that type's own schema declares mandatory in its "## Requires" section — for example an `entity` node missing its `category` predicate, or a domain-specific type missing a predicate its profile requires. The maintainer wants this caught immediately, since a node silently missing required data is incomplete in a way that's easy to miss by eye but breaks every downstream consumer that assumes the type's contract holds.

**Why this priority**: This is the single gap called out first and most directly in the request — lint today never compares a node's actual predicates against what its own type schema demands. Without this, a graph can accumulate incomplete nodes indefinitely with lint reporting a clean pass.

**Independent Test**: Author a node of a registered type, deliberately omitting one predicate that type's schema lists under "## Requires". Run lint and confirm it reports exactly that missing predicate, naming the type, the predicate, and the offending file — and confirm a node carrying every required predicate produces no such violation.

**Acceptance Scenarios**:

1. **Given** a node whose type's schema lists one or more predicates under "## Requires", and the node is missing at least one of them, **When** the user runs lint, **Then** the tool reports one violation per missing required predicate, naming the predicate, the node's type, and the offending file.
2. **Given** a node that carries every predicate its type's "## Requires" section lists, **When** the user runs lint, **Then** no missing-required-predicate violation is reported for that node.
3. **Given** a node whose type is not itself registered in the graph's schema, **When** the user runs lint, **Then** this check is not applied to that node (the existing unrecognized-type violation already covers it), avoiding a redundant or misleading second violation.

---

### User Story 2 - Catch a node carrying a predicate its type doesn't permit (Priority: P1)

A graph maintainer's node carries a predicate that its type's schema does not list under either "## Requires" or "## Optional" — for example a leftover predicate from a copy-pasted node of a different type, or a typo'd predicate name that happens to already be registered globally. The maintainer wants this flagged, since a predicate outside its type's declared contract signals either a mistake or an undocumented extension of that type's shape.

**Why this priority**: Named as the direct complement of User Story 1 in the request, and equally load-bearing — a type schema's "## Requires"/"## Optional" sections are only a meaningful contract if lint enforces both directions (nothing missing, nothing extra). Both checks together are what "does a node actually conform to its type" means.

**Independent Test**: Author a node of a registered type carrying one predicate that is registered in the graph's predicate schema but not listed under that type's "## Requires" or "## Optional" sections. Run lint and confirm it reports exactly that predicate as not permitted by the node's type — and confirm the two universal identity predicates and any predicate actually listed under "## Requires"/"## Optional" produce no such violation.

**Acceptance Scenarios**:

1. **Given** a node carrying a predicate that its type's schema lists under neither "## Requires" nor "## Optional", **When** the user runs lint, **Then** the tool reports one violation naming that predicate, the node's type, and the offending file and line.
2. **Given** a node carrying only predicates listed under its type's "## Requires" and/or "## Optional" sections, **When** the user runs lint, **Then** no such violation is reported.
3. **Given** any node's front-matter identity predicates ("@id" and "@type"), **When** the user runs lint, **Then** these two are never reported as "not permitted", regardless of whether the node's type schema repeats them in "## Requires"/"## Optional" — they are universally implicit.

---

### User Story 3 - Catch missing or incorrectly quoted `@id`/`@type` front-matter keys (Priority: P2)

A node's front matter was hand-edited or produced by a tool that wrote the reserved JSON-LD-style keys `@id`/`@type` as bare (unquoted) YAML keys instead of quoted strings, or omitted one of them outright. The maintainer wants this caught, since ARCNET-CORE §10.1 requires these two identity keys to be present and quoted specifically to avoid inconsistent behavior across YAML parsers.

**Why this priority**: Identity keys are foundational — every other check in this feature and in existing lint assumes a node's `@id`/`@type` were read reliably. A quoting inconsistency that one YAML implementation tolerates and another rejects is exactly the kind of silent, tool-dependent drift a conformance check exists to catch, but it affects fewer nodes in practice than the Requires/Optional gap (most authoring tools already quote these keys correctly), hence P2.

**Independent Test**: Author a node whose front matter writes `@id` (or `@type`) as a bare, unquoted key, and a second node that omits one of the two keys entirely. Run lint against both and confirm each is reported with a violation identifying the missing or incorrectly-quoted key and its line — and confirm a node with both keys present and quoted produces no such violation.

**Acceptance Scenarios**:

1. **Given** a node's front matter is missing the `"@id"` key, the `"@type"` key, or both, **When** the user runs lint, **Then** the tool reports one violation per missing key, naming which key is missing and the file.
2. **Given** a node's front matter writes `@id` or `@type` as a bare (unquoted) YAML key rather than the quoted form `"@id"`/`"@type"`, **When** the user runs lint, **Then** the tool reports one violation per incorrectly-quoted key, naming the key, the file, and the line it appears on.
3. **Given** a node's front matter writes both `"@id"` and `"@type"` as quoted keys, **When** the user runs lint, **Then** no such violation is reported for that node.

---

### User Story 4 - Recognize citation predicates from the graph's own schema, not a hardcoded list (Priority: P2)

A domain profile registers an additional citation-style predicate in the graph's own `_schema/predicates/` (for example a domain-specific citation relationship), declaring its alignment as a `cito:` vocabulary term. The graph maintainer wants lint to recognize that predicate as a valid citation predicate automatically, without waiting for a new `arc` release that hardcodes it into the tool.

**Why this priority**: Directly requested as a portability fix — the current hardcoded Go list of citation predicates means a domain profile cannot extend the citation vocabulary without a code change to `arc` itself, which contradicts the graph's schema being the authoritative, extensible source of truth. P2 because it's a correctness/extensibility fix to an already-existing check rather than a wholly new gap.

**Independent Test**: Register a new predicate in a graph's schema with a `cito:`-aligned value not in `arc`'s previous hardcoded list, use it as a citation in a node, and confirm lint accepts it without a false-positive violation. Then use an unregistered or non-`cito:`-aligned predicate as a citation and confirm lint still reports it as invalid.

**Acceptance Scenarios**:

1. **Given** a predicate registered in the graph's schema whose declared alignment is a `cito:` vocabulary term, and a node uses that predicate as a citation, **When** the user runs lint, **Then** no citation-predicate violation is reported for that usage, regardless of whether `arc`'s own source code was ever updated to know that predicate's name.
2. **Given** a node uses a predicate as a citation that is either unregistered in the graph's schema or registered without a `cito:`-aligned alignment, **When** the user runs lint, **Then** the tool reports a citation-predicate violation for that usage, naming the file and line.
3. **Given** a graph's schema registers zero predicates with a `cito:`-aligned alignment, **When** the user runs lint against a node that uses any predicate as a citation, **Then** that usage is reported as a citation-predicate violation (there is no fallback to a built-in default vocabulary).

### User Story 5 - Catch a predicate used in a structural position its schema role doesn't permit (Priority: P3)

A node uses a predicate in a way that conflicts with that predicate's own schema-declared role — for example a predicate registered with role `text` (meant to appear as a prose paragraph) instead written as a typed edge bullet, or a predicate registered with role `edge` instead written as prose text. The maintainer wants this caught, since a predicate's role determines how every tool in the ecosystem (not just `arc`) expects to find and interpret it, and a mismatch silently breaks that expectation.

**Why this priority**: A real conformance gap and explicitly requested, but narrower in practical impact than the Requires/Optional contract (User Stories 1-2) — most role mismatches only arise from hand-editing or a non-`arc` tool writing a node, since `arc apply` itself already writes predicates according to their declared role (spec 013). P3 reflects that this check mostly guards against out-of-band edits rather than `arc`'s own normal operation.

**Independent Test**: Author a node with a predicate registered with role `text` written as a typed edge bullet instead of prose, and a second node with a predicate registered with role `edge` written as prose text instead of a bullet. Run lint against both and confirm each mismatch is reported, naming the predicate, its declared role, and where it was actually found — and confirm a node where every predicate's usage matches its declared role produces no such violation.

**Acceptance Scenarios**:

1. **Given** a predicate registered with a specific schema role, and a node's occurrence of that predicate is written in a structural position inconsistent with that role (e.g. role `text` written as an edge, role `edge` written as prose, role `meta` written as a body element), **When** the user runs lint, **Then** the tool reports one violation naming the predicate, its declared role, and the file and line of the mismatched occurrence.
2. **Given** every predicate occurrence in a node is written in the structural position its own schema role declares, **When** the user runs lint, **Then** no role-mismatch violation is reported for that node.
3. **Given** a node uses a predicate that has no registered schema at all, **When** the user runs lint, **Then** this check is not applied to that occurrence (the existing unregistered-predicate violation already covers it), avoiding a redundant or misleading second violation.

### Edge Cases

- What happens when a node's type is registered but its type schema's "## Requires" section is entirely absent or empty? The tool MUST treat this as "requires nothing" — no missing-required-predicate violations are possible for that type, but the "## Optional" check (User Story 2) still applies to every non-identity predicate the node carries.
- What happens when a node's type schema's "## Optional" section is entirely absent or empty? The tool MUST treat this as "no additional predicates permitted beyond ## Requires" — any non-identity predicate the node carries that isn't in "## Requires" is reported as not permitted.
- What happens when the same predicate is listed under both a type's "## Requires" and "## Optional" sections (a malformed type schema)? The tool MUST treat the predicate as satisfying both checks (present-if-required, permitted-if-present) without reporting a spurious violation against the node being linted — this feature validates nodes against their type schema, not the type schema's own internal consistency.
- What happens when a node carries the same predicate more than once (e.g. multiple edge occurrences of the same predicate name)? The Requires/Optional/role checks apply once per distinct predicate name per node, not once per occurrence — but a role mismatch on any individual occurrence is still reported for that specific occurrence's file and line.
- What happens when a node's `@id`/`@type` front-matter keys are missing entirely, such that the node fails to parse at all? The existing front-matter parse-failure violation already covers this case; the new missing/unquoted-key check (User Story 3) applies to nodes whose front matter is otherwise parseable, so the two checks don't produce confusing duplicate reports for the same root cause.
- What happens when a citation predicate check (User Story 4) encounters a graph whose schema registers a predicate's alignment using a vocabulary prefix other than `cito:` (e.g. a plain English label, or a different ontology prefix)? That predicate MUST NOT be treated as a citation predicate — only an alignment recognizable as a `cito:` vocabulary term qualifies.
- What happens when a node's kind (type) is unrecognized (already caught by the existing unrecognized-kind check)? The new Requires/Optional and predicate-role checks (User Stories 1, 2, 5) MUST NOT run against that node's declared-but-unregistered type, since there is no schema to validate against; only the pre-existing unrecognized-kind violation is reported for that gap.
- What happens when a very large graph is linted with all five new checks enabled alongside the existing checks? The tool MUST still complete a full run and report every violation across every applicable check, consistent with existing lint's performance expectations for large graphs.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: For every node whose type is registered in the graph's schema, the tool MUST verify that every predicate listed under that type's "## Requires" section is present at least once on the node, reporting one violation per missing required predicate, naming the predicate, the node's type, and the offending file.
- **FR-002**: For every node whose type is registered in the graph's schema, the tool MUST verify that every distinct predicate the node carries — other than the two universal identity predicates ("@id"/"@type") — is listed under that type's "## Requires" or "## Optional" section, reporting one violation per predicate that is present on the node but listed under neither, naming the predicate, the node's type, the offending file, and line.
- **FR-003**: FR-001 and FR-002 MUST NOT be applied to a node whose type is not itself registered in the graph's schema — that gap is already reported by the existing unrecognized-type check, and re-reporting it here would be redundant.
- **FR-004**: The tool MUST verify that every node's front matter contains both the `"@id"` and `"@type"` keys, reporting one violation per missing key naming which key is missing and the offending file.
- **FR-005**: The tool MUST verify that, where present, the `"@id"` and `"@type"` front-matter keys are written as quoted YAML string keys, reporting one violation per key written as a bare (unquoted) key, naming the key, the offending file, and the line it appears on.
- **FR-006**: The tool MUST determine the set of valid citation predicates dynamically from the graph's own schema — every predicate registered in `_schema/predicates/` whose declared alignment is a `cito:` vocabulary term — rather than from a fixed list built into the tool, so a domain profile's additional citation predicates are recognized without a change to the tool itself.
- **FR-007**: The tool MUST verify that every predicate used as a citation in a node's body is one of the dynamically-determined valid citation predicates (FR-006), reporting one violation per citation using a predicate that is unregistered or whose alignment is not a `cito:` vocabulary term, naming the file and line.
- **FR-008**: For every predicate occurrence on a node where that predicate is registered in the graph's schema, the tool MUST verify that the occurrence's structural position (front-matter scalar/array, prose paragraph, inline untyped link, typed edge bullet, or grouped link block) matches that predicate's own schema-declared role, reporting one violation per mismatched occurrence, naming the predicate, its declared role, and the offending file and line.
- **FR-009**: FR-008 MUST NOT be applied to an occurrence of a predicate that has no registered schema at all — that gap is already reported by the existing unregistered-predicate check, and re-reporting it here would be redundant.
- **FR-010**: Every violation reported by FR-001 through FR-008 MUST identify, at minimum: a clear and distinct rule name for that specific check, the offending file, a line number where one is determinable (or a clear indication that no single line applies), and a human-readable message describing the violation — consistent with `arc lint`'s existing violation format.
- **FR-011**: The tool MUST continue evaluating every check (existing and new) across every node in a single run and MUST NOT stop at the first violation found, consistent with `arc lint`'s existing collect-everything behavior.
- **FR-012**: The tool MUST make no changes to the graph's files, schema, or git history while performing any of the checks introduced by this feature — linting remains a read-only, non-mutating operation.
- **FR-013**: None of the checks introduced by this feature MUST alter `arc apply`'s own behavior — this feature only extends `arc lint`'s validation; how the graph is written or merged is out of scope.
- **FR-014**: The tool's built-in type schema seed (`kernel.CoreTypeDefs`, rendered to `_schema/types/` by `arc init`) MUST list every Content (§10.2: `tags`, `text`) and type-unrestricted Metadata/Control (§10.3: `created`, `updated`; plus `indexed`, FR-019) predicate under `## Optional` on all four core types (`source`, `entity`, `resource`, `timeline`), in addition to whatever type-specific predicates each type already requires or permits — so a node legitimately carrying one of these cross-cutting predicates is never reported as not-permitted by FR-002. *(Added — BUG-001.)*
- **FR-015**: `published` MUST be listed under `## Optional` on `entity` and `resource` — matching spec 009 FR-001's own enumeration of which node kinds `arc apply` auto-stamps `published` on. `source`'s existing `## Requires` entry for `published` is unaffected. `timeline` is deliberately excluded: spec 009 FR-001 does not name it among the kinds `arc apply` stamps `published` on, and ARCNET-CORE's own worked `timeline` type-schema example does not list it either. *(Added — BUG-001.)*
- **FR-016**: The structural predicates `mentions`/`mentionedIn` (§10.4) MUST be listed under `## Optional` on all four core types, in addition to their existing `## Requires` status on `source` (`mentions`) and `entity` (`mentionedIn`) respectively — so any core type may originate either direction of this relationship, not only the one ARCNET-CORE's own `from → to` annotation names as canonical. *(Added — BUG-001.)*
- **FR-017**: `entity`'s `## Optional` list MUST include `notes` and all nine §10.5 semantic predicates (`broader`, `narrower`, `isPartOf`, `hasPart`, `requires`, `replaces`, `isReplacedBy`, `conformsTo`, `related`) — matching ARCNET-CORE §11.3's own worked example verbatim ("any §10.5 semantic predicate, as applicable"), which the tool's current seed omits entirely. This is the confirmed, unambiguous defect underlying this bugfix: an `entity` node using `conformsTo` exactly as CORE's own worked example shows is rejected by `arc lint` today. *(Added — BUG-001.)*
- **FR-018**: `resource`'s `## Optional` list MUST also include all nine §10.5 semantic predicates, per this bugfix's explicit scope decision that they be usable when authored on either an `entity` or a `resource` node, not only an `entity` node as ARCNET-CORE §10.5's prose ("written as `edge`-role predicates in the entity body") most narrowly reads. *(Added — BUG-001.)*
- **FR-019**: The tool's built-in predicate schema seed (`kernel.CorePredicateDefs`) MUST register `indexed` (`role: meta`, `merge: immutable`, `aligned: arc:indexed`) — spec 009's node-provenance timestamp, currently absent from `CorePredicateDefs` entirely despite `arc apply` unconditionally stamping it on every non-stub, non-schema node it creates (spec 009 FR-004/FR-006). Without this entry, every node `arc apply` produces carries an attribute no schema check can recognize as legitimate. *(Added — BUG-001.)*
- **FR-020**: The tool's built-in predicate schema seed MUST register two new control predicates, `scoreZ` and `scoreC` (`role: meta`, `merge: validatedOverwrite`, `aligned: arc:scoreZ`/`arc:scoreC` respectively — neither is part of ARCNET-CORE's own upstream vocabulary, confirmed absent from `github.com/fogfish/arcnet-spec`'s `ARCNET-CORE.md` by direct inspection), and list both under `## Optional` on all four core types. `validatedOverwrite` is chosen to match these predicates' documented nature (a graph-analytics score recomputed by a validation/ingest pass on every run, per historical usage in `specs/003-apply-patch/bugs/BUG-003.md`/`BUG-004.md`), consistent with CORE §9.3's own description of that merge behavior ("used by profile predicates whose value is computed after the fact"). Any prior ad hoc usage of the hyphenated forms `score-z`/`score-c` was never a formally registered predicate (confirmed by repository-wide search) and is superseded by these two camelCase names going forward. *(Added — BUG-001.)*
- **FR-021**: The tool's built-in predicate schema seed MUST register `period` (`role: meta`, `merge: immutable`, `aligned: arc:period` — not part of ARCNET-CORE's own upstream vocabulary; `arc apply`'s own timeline-period mechanism writes it as a duplicate, explicitly-quoted copy of a timeline node's own `@id`, solely so a bare 4-digit yearly period decodes as a string rather than a YAML integer, spec 003 BUG-007), and `CoreTypeDefs["timeline"]`'s `## Requires` list MUST include it. *(Added — BUG-002.)*
- **FR-022**: `timeline`'s chronological reference to each `source` node whose `published` date falls in its period MUST be declared under `## Requires` as `cites`, not `entries` — reusing the existing `cites` predicate (§10.6) rather than the `entries` name `CoreTypeDefs["timeline"]` previously declared. `cites`'s own `merge` behavior changes from `union` to `append`, to preserve the chronological-ordering guarantee `entries` previously carried ("ordered by date") — `append`'s own semantics (CORE §9.3: "grow the predicate's content without discarding what is already there... an ordered, uniquely-keyed list gets a new entry") are compatible with, and no weaker than, `cites`'s pre-existing usage on `source` nodes for their own outbound citations. The `entries` predicate registration itself is removed — no real graph ever produced a genuine `entries`-tagged edge (FR-023 below explains why), so no existing content depends on it remaining registered. *(Added — BUG-002 — this is a deliberate scope decision reusing an existing predicate for two related-but-distinct relationships, not a spec-conformance fix: ARCNET-CORE §11.5's own worked example names this relationship `entries`, and §10.6's own `cites` describes a source's citation of an external resource specifically — this FR knowingly diverges from the fetched upstream spec text on this one point, per explicit product decision.)*
- **FR-023**: `arc apply`'s timeline-period-file writer (`internal/core.TimelineEntry`, `internal/app/graph/service.upsertTimelinePeriod`) MUST render each chronological reference as a `cites::`-tagged bullet (`- cites:: [[id]] — *title* (authors) — date`), not a bare, untyped `[[id]] — ...` bullet as it does today — the current bare-link form is why `arc lint`'s FR-001 check (already correctly requiring this predicate under `timeline`'s own `## Requires`, per spec 011) has always failed against every timeline file `arc apply` produces: the predicate was required by schema but never actually written to any file. *(Added — BUG-002.)*
- **FR-024**: The tool's shared node parser (`internal/core.ParseNode`, specifically its list-item recognizer) MUST accept a predicate-tagged wikilink bullet (`predicate:: [[target]]`) followed by additional display-only annotation text on the same line (e.g. `— *title* (authors) — date`, ARCNET-CORE §11.5's own worked example convention), extracting the predicate/target pair and discarding the trailing annotation — today it silently discards the *entire bullet*, producing neither an edge nor an href, whenever any text follows the wikilink's closing `]]`. This is independent of FR-022/FR-023's naming decision: the parsing gap exists for any predicate, not just a timeline's own references, and was confirmed by feeding the parser ARCNET-CORE's own literal worked example line. *(Added — BUG-002.)*

### Key Entities

- **Type Requirement Contract**: The "## Requires" and "## Optional" predicate lists declared in a registered type's own schema node — together they define the complete set of predicates a conforming instance of that type may carry, with "## Requires" being the mandatory subset.
- **Universal Identity Predicate**: Either of the two front-matter keys every node carries regardless of type ("@id"/"@type") — never itself subject to the Requires/Optional contract and never reported as "not permitted" by a type's schema.
- **Citation Predicate Vocabulary**: The set of predicates registered in the graph's own schema whose declared alignment is a `cito:` vocabulary term — determined dynamically per graph, replacing a fixed built-in list, so it grows with whatever a domain profile registers.
- **Predicate Role**: A predicate's own schema-declared structural position (front-matter scalar/array, prose paragraph, inline untyped link, typed edge bullet, or grouped link block) — every occurrence of that predicate on any node is expected to appear in that position.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A graph maintainer can determine, from a single `arc lint` run, whether every node in their graph carries exactly the predicates its own type's schema requires and permits — no missing-required or not-permitted predicate goes unreported.
- **SC-002**: 100% of a representative graph's deliberately introduced violations — one per new check (missing required predicate, not-permitted predicate, missing/unquoted identity key, schema-driven citation-predicate mismatch, predicate-role mismatch) — are each individually detected and reported, with no check silently skipped.
- **SC-003**: A domain profile can register a new `cito:`-aligned citation predicate in the graph's own schema and see `arc lint` recognize valid uses of it on the very next run, with zero changes to the `arc` tool itself.
- **SC-004**: 100% of graphs with zero introduced violations across the new checks continue to produce a clean pass on those checks, alongside all pre-existing checks.
- **SC-005**: Running `arc lint` with these new checks enabled never modifies any graph file, schema, or git history — verified by the graph's state being byte-for-byte identical before and after any run.
- **SC-006**: A graph maintainer can distinguish, from the reported rule name alone, which of the five new checks fired for any given violation, without needing to inspect the tool's source code.
- **SC-007**: 100% of ARCNET-CORE-conformant nodes using a cross-cutting Content, Metadata/Control, Structural, or (for `entity`/`resource`) Semantic predicate (FR-014–FR-020) on any of the four core types pass `arc lint`'s Requires/Optional checks with zero false positives — verified directly against a real `arc init`-seeded graph, including the specific `entity`-using-`conformsTo` case this bugfix was reported against. *(Added — BUG-001.)*
- **SC-008**: 100% of timeline period files (`timeline/yearly/<YYYY>.md`, `timeline/monthly/<YYYY-MM>.md`) `arc apply` produces pass `arc lint`'s Requires/Optional checks with zero false positives, verified end-to-end (`arc init` → `arc apply` a real patch → `arc lint`) against a real graph — the exact `dmitry-2026-graph` scenario this bugfix was reported against produces zero `[typeRequires]`/`[typeOptional]` violations on either generated period file. *(Added — BUG-002.)*
- **SC-009**: `internal/core.ParseNode` correctly extracts a predicate-tagged wikilink bullet's predicate and target when followed by display-only trailing annotation text, for 100% of lines matching ARCNET-CORE §11.5's own worked-example convention — verified by parsing that exact line and confirming a populated `Edges` entry results, not silent data loss. *(Added — BUG-002.)*

## Assumptions

- The graph's schema-resolution mechanism that already decodes each registered type's "## Requires"/"## Optional" sections and each registered predicate's role/alignment fields (used today by other `arc` commands) is the authoritative source this feature validates nodes against; this feature does not change how those sections are authored or parsed, only how lint consumes them.
- "Present on the node" for a predicate means at least one occurrence exists anywhere in the node (front matter, prose, inline link, edge bullet, or grouped link block) that is attributed to that predicate name.
- ~~A type schema's "## Requires"/"## Optional" sections are themselves well-formed (each entry names a predicate that is, in turn, registered in `_schema/predicates/`); validating the type schema's own internal well-formedness beyond what's needed to run these node-level checks is out of scope for this feature.~~ **Superseded — BUG-001**: this assumption does not hold for the tool's own built-in seed data. `kernel.CoreTypeDefs["entity"].Optional` was missing `notes` and all nine §10.5 semantic predicates relative to ARCNET-CORE §11.3's own worked example — a real, reproduced conformance defect this feature's own new checks (FR-002) exposed for the first time, addressed by FR-014–FR-020. The narrower principle still holds going forward — this feature's *checks* do not themselves re-validate a type schema's internal well-formedness on every run — but "the seed data is already correct" was a false premise, not a scope boundary this feature is entitled to assume without verification.
- "Correctly quoted" for `"@id"`/`"@type"` means the front-matter key is written as a YAML string literal (single- or double-quoted) rather than a bare/plain scalar key, per ARCNET-CORE §10.1.
- Fixing a reported violation from any of these five new checks is out of scope, consistent with `arc lint`'s existing report-only scope — a future auto-fix capability, if built, would be a separate feature.
- These checks apply uniformly to every registered node type (built-in and domain/extension-registered alike) — there is no per-type opt-out from the Requires/Optional contract, identity-key quoting, citation-predicate vocabulary, or role-conformance checks.

## Notes

**Bugfix**: 2026-07-09 — BUG-001 (implementation drift, discovered via `arc lint` itself post-implementation): `kernel.CoreTypeDefs`'s `Optional` lists (spec 011's own seed data, `internal/app/schema/kernel/schema.go`) never fully reflected ARCNET-CORE's own §10 vocabulary — before this feature (014) shipped, `arc lint` never read `## Requires`/`## Optional` at all, so the gap was invisible; FR-002's new check (this feature's own primary deliverable) is what first exposed it. Added FR-014 through FR-020 and SC-007, and struck through the now-falsified "type schema seed data is well-formed" assumption. Confirmed by direct inspection of `github.com/fogfish/arcnet-spec`'s `ARCNET-CORE.md` (fetched and read in full, not summarized) and by reproducing the false positive against a real `arc init`-seeded graph: an `entity` node using `conformsTo` — verbatim from CORE §11.3's own worked example — was rejected under `[typeOptional]`. `scoreZ`/`scoreC` (FR-020) and the broader "permit on all four core types" scope (FR-014/FR-016) are not directly specified by `ARCNET-CORE.md` (confirmed absent by exhaustive grep) — this bugfix's scope decisions for those items are recorded in `research.md`'s own Bugfix note, not re-derived here.

**Bugfix**: 2026-07-10 — BUG-002 (implementation drift spanning three pre-existing specs — 003, 009, 010 — discovered the same way as BUG-001: via this feature's own checks running against a real `arc apply`-produced graph). Every timeline period file `arc apply` produces failed `arc lint` with `[typeRequires]`/`[typeOptional]` on `entries`/`period`, reproduced verbatim against the reporter's own example content. Root causes: (1) `period` (a real front-matter field `upsertTimelinePeriod` has written since spec 003's BUG-007 fix) was never registered in `kernel.CorePredicateDefs`/`CoreTypeDefs["timeline"]`; (2) `entries` was declared `## Requires` by spec 011's seed data, matching ARCNET-CORE §11.5's own worked example, but `internal/core.TimelineEntry` never actually wrote a predicate-tagged edge — only a bare, untyped wikilink, by original design (its own doc comment said so) — so the predicate schema and the code producing timeline files disagreed since before this feature existed; (3) independently, and more seriously, `internal/core.ParseNode`'s list-item parser was confirmed (by direct reproduction) to silently drop ARCNET-CORE §11.5's own literal worked-example bullet shape entirely — zero `Edges`, zero `HRefs` — whenever a predicate-tagged wikilink is followed by any trailing text, which the spec's own convention always includes. Added FR-021 through FR-024 and SC-008/SC-009. Per an explicit product decision (not a spec-conformance reading — see FR-022's own callout), this bugfix reuses the existing `cites` predicate for a timeline's chronological reference to its source nodes, replacing `entries` outright, rather than keeping `entries` as ARCNET-CORE §11.5 names it — `cites`'s own `merge` behavior changes from `union` to `append` to preserve the ordering guarantee `entries` carried. The parser fix (FR-024) is independent of this naming decision and would be needed regardless of which predicate name timeline entries use.
