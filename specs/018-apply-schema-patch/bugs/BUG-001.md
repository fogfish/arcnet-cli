# Bug Report: `arc apply schema` rejects a valid published extension because `merge` is wrongly mandatory on `Class` (type) definitions

**Type**: Spec conflict
**Severity**: High
**Feature**: 018-apply-schema-patch (symptom); root conflict originates in 011-machine-readable-schema / 012-predicate-merge-policies / 017-subclass-of-predicate
**Reported**: 2026-07-19
**Status**: Patched
**Patched**: 2026-07-19

## Description

Running `arc apply schema -v ../arcnet-spec/schema/domain-article.md` ‚Äî the
command's own headline use case, importing a published arcnet extension ‚Äî
fails outright:

```
../arcnet-cli/arc apply schema -v ../arcnet-spec/schema/domain-article.md
Reading patch (2ms)
‚ùå Applying schema nodes failed: [github.com/fogfish/arcnet-cli/internal/app/schema/service.planSchemaNode 150] schema document Hypothesis has a missing or invalid merge:

 ‚ùå schema document Hypothesis has a missing or invalid merge
   Run `arc help apply schema` for guidance.
```

The extension's `Hypothesis` `Class` definition is otherwise well-formed
(ARCNET-CORE ¬ß9.2 shape: `@id`, `@type: Class`, a descriptive body, and
`## Requires`/`## Optional`), but it carries no `merge` field ‚Äî and
`arc apply schema` refuses the entire patch because of it.

As the user identified: **`merge` is only operationally mandatory at the
`Property` level.** A `Class` (type) document's own whole-node `merge`
field has had no effect on reconciliation since spec 012 shipped ‚Äî `arc
apply` reconciles every predicate individually via its own declared merge
behavior (`core.Index.Predicates[name].Merge`), never by a node's type.
Requiring a functionally-meaningless field to be present, and rejecting an
entire externally-published extension when it's absent, blocks the
command's own primary scenario.

## Artifact Traceability

### spec.md

- **Affected feature/user story**: `018-apply-schema-patch`, User Story 1 ‚Äî
  "Import a published extension's schema in one step" ‚Äî specifically
  **Acceptance Scenario 2**: "Given a well-formed patch document containing
  only `Class` node sections, When the maintainer runs `arc apply schema`
  against that document, Then a schema type definition is created for each
  `Class` node it carries." A `Class` document with no `merge` field is
  "well-formed" per ARCNET-CORE's own documented shape (research.md D3 of
  spec 011 explicitly calls the type-level `merge` field an
  arcnet-cli-specific *deviation* from CORE's own Schema Index shape, not
  part of it) ‚Äî the command should create the type definition, not reject
  the whole patch.
- **Affected requirement**: 018's own **FR-008** ("For each valid `Class`
  node section, the command MUST create a new type definition ... following
  that definition's declared merge behavior") implicitly assumes `Class`
  always declares one. It should instead read: "declared merge behavior, if
  any" ‚Äî merge is genuinely optional/inert at the type level.
- **Root conflict, spec 011** (`machine-readable-schema`) ‚Äî **FR-015**:
  "`_schema/types/<name>.md` MUST additionally declare that type's
  whole-node merge behavior ... so `arc apply`'s existing whole-node merge
  dispatch continues to read each type's merge behavior." Spec 011's own
  Assumptions section (research.md/spec.md line ~123) already flags this as
  *temporary*: "a deliberate, temporary deviation from the core
  specification family's own documented Schema Index shape ... kept only so
  `arc apply`'s existing whole-node merge dispatch has zero regression
  during this feature, and expected to be retired once a future feature
  moves `arc apply` to genuine per-predicate merge dispatch."
- **Root conflict, spec 012** (`predicate-merge-policies`) ‚Äî line 120: "The
  whole-node `merge` field on a type's schema document ... is no longer
  consulted by `arc apply` once this feature ships. Removing that field
  from the schema-document shape itself is out of scope; it simply becomes
  unused by reconciliation." Spec 012 retired the field's *function* but
  explicitly scoped out revisiting its *mandatory-ness* in validation ‚Äî
  that follow-up was never filed as its own task, and 018 was built on top
  of the still-mandatory validation without re-examining it.
- **Gap identified**: No spec ever reconciled "the field is functionally
  retired" (012) with "the field stays mandatory in `decodeTypeDef`'s
  validation, used by `Resolve` ‚Äî and therefore by `arc apply`, `arc lint`,
  and `arc apply schema` alike" (011/017, unchanged since). This is a
  genuine spec conflict, not just a code bug: two specs shipped
  requirements that are individually correct for their own time but
  contradict each other once both are in effect.

### plan.md

- **018/plan.md**: Data-model.md and plan.md for 018 describe `decodeTypeDef`

  still correct post-spec-012 ‚Äî the plan inherited the mandatory-merge

- **False completions**: T025 is legitimately done for what it specified,

- **Missing tasks**: A new task (in whichever spec directory

  describe `merge` as mandatory on a Type Schema Node.

## Root Cause Analysis

This is a **spec conflict** that shipped in two increments:

1. Spec 011 introduced a type-level `merge` field as a deliberate, explicitly
   *temporary* bridge (FR-015) so `arc apply`'s pre-012 whole-node merge
   dispatch kept working, and made it mandatory in `decodeTypeDef`
   (`internal/app/schema/service/schema.go`) so the schema index would
   fail fast on a malformed document.
2. Spec 012 retired the field's actual function (whole-node dispatch is
   gone; every predicate reconciles by its own declared merge behavior
   instead) but explicitly scoped out removing/relaxing the field from the
   schema-document shape ‚Äî leaving `decodeTypeDef`'s mandatory-field check
   untouched.
3. Spec 018 was built afterward on top of `decodeTypeDef` as an existing,
   presumed-still-correct dependency (T025), inheriting the now-stale
   mandatory-merge-on-Class requirement without re-deriving it ‚Äî so a
   `Class` definition from a real, CORE-conformant external catalog (which
   has no reason to carry a functionally inert field) is rejected outright.

No single spec is "wrong" in isolation; the conflict is between spec 011/017
(merge is mandatory on a Type Schema Node) and spec 012 (merge is
functionally unused at the type level), never reconciled by a follow-up
task.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to update spec artifacts ‚Äî most likely
   spec 012 (`predicate-merge-policies`), since it is the spec that made the
   field vestigial and is the natural owner of "finish retiring it from
   validation too." Amend it (or file the follow-up against 018, whichever
   `/speckit.bugfix.patch` determines is the better fit) to state: a
   `Class` document's `merge` field becomes optional and unvalidated (or is
   dropped from the mandatory-field check in `decodeTypeDef`), while a
   `Property` document's `merge` field remains mandatory.
2. Run `/speckit.bugfix.verify` to confirm consistency across specs
   011/012/017/018 and ARCHITECTURE.md's Type Schema Node glossary entry.
3. Resume `/speckit.implement` to relax `decodeTypeDef` in
   `internal/app/schema/service/schema.go` (stop treating `merge` as
   mandatory/invalid-if-absent for `Class` nodes; keep `decodePredicateDef`'s
   `Property`-level check unchanged), re-run `resolveTypes`/`Resolve`'s own
   tests, and add a regression test importing a `Class` definition with no
   `merge` field via both `arc apply schema` and hand-authored
   `_schema/types/` content read by `arc lint`/`arc apply`.
