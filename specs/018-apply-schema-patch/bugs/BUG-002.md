# Bug Report: `arc apply schema` rejects an incremental `Property`/`Class` re-declaration that omits mandatory fields already supplied by the existing definition

**Type**: Spec gap (with implementation drift)
**Severity**: High
**Feature**: 018-apply-schema-patch

**Reported**: 2026-07-19
**Status**: Patched
**Patched**: 2026-07-19

## Description

Re-applying an extension patch that adds new `Optional` predicates to an
already-registered `Class` (here, the built-in `Source` type) fails, even
though `Source`'s existing schema document already has a valid
description:

```
❌ Applying schema nodes failed: [github.com/fogfish/arcnet-cli/internal/app/schema/service.planSchemaNode 150] schema document Source has a missing or invalid description
```

The offending patch section:

```
## Source
```yaml
```

**Optional**
- optional:: [[proposes]]
- optional:: [[raises]]
```

The incoming section carries no description (and an empty yaml fence) — it
is a pure *delta*: "add these two predicates to `Source`'s `Optional`
list," relying on the existing `_schema/types/Source.md` to still supply
the description, `merge` (now optional per 018/BUG-001), and every other
field the incoming section doesn't restate. As the user identified, this is
exactly the shape spec 018's own User Story 3 describes as a supported
capability: "inject only properties from the patch."

## Artifact Traceability

### spec.md

- **Affected user story**: `018-apply-schema-patch`, **User Story 3** —
  "Refresh an already-imported extension's schema." The story's own
  narrative states the capability directly: "An extension the maintainer
  previously imported publishes an update — a revised description, **an
  added optional predicate on an existing type**, or a new predicate
  altogether. The maintainer re-applies the extension's updated patch
  document, and the local schema picks up the changes for definitions it
  already knows about, **without needing to be told which specific
  definitions changed**." Adding `proposes`/`raises` to `Source`'s
  `Optional` list without restating its description is precisely "an added
  optional predicate on an existing type" — the story's own headline
  example.
- **Affected requirements**: **FR-008** ("For each valid `Class` node
  section, the command MUST create a new type definition ... or merge into
  the existing one if it does") never defines what "valid" means for a
  section that is merging into an *existing* definition versus creating a
  *new* one — the spec gap. **Acceptance Scenario 1** (under User Story 3)
  only covers a `Property` re-declaration "with an added or changed field,"
  never a `Class` re-declaration that omits a field the existing document
  already supplies; **Acceptance Scenario 2** only covers a no-op
  re-apply. Neither scenario exercises "existing document supplies a
  mandatory field the incoming delta omits" for either node kind.
- **Gap identified**: The spec never distinguishes two readings of "valid
  ... node section": (a) *self-contained-complete* — every mandatory field
  (`role`/`merge` for `Property`; `description` for `Class`) must be
  present in the incoming section itself, independent of any existing
  document, or (b) *valid-once-merged* — the section may omit any field an
  existing definition already supplies, since the whole point of "merge
  into the existing one" (FR-007/FR-008's own words) is that the section is
  a delta, not a full restatement. User Story 3's narrative clearly intends
  (b); the implementation (see below) does (a).

### plan.md

- **Affected section**: 018/plan.md's Project Structure lists
  `internal/app/schema/service/apply.go` as reusing `decodePredicateDef`/
  `decodeTypeDef` "for validation" without specifying *when* in the
  create-vs-merge flow that validation runs relative to reading back the
  existing document — the plan inherited the same assumption FR-008 left
  implicit.
- **Impact**: Any future plan touching `planSchemaNode` must explicitly
  decide validation ordering: validate the raw incoming section, the merged
  result, or both (with different, narrower rules for each).

### tasks.md

- **Affected task**: 018/tasks.md **T035** — "Verify/extend the T024/T025
  create/merge loop in `internal/app/schema/service/apply.go` so an
  existing `_schema/predicates/<name>.md`/`_schema/types/<name>.md`
  document is read back ... and merged via `core.Merge` rather than
  overwritten" — implemented the read-back-and-merge step, but the
  pre-existing `decodePredicateDef`/`decodeTypeDef` validation call (from
  T024/T025) was never moved to run *after* that read-back, so it still
  validates the patch's own raw section in isolation.
- **False completions**: T035 is marked done and its own stated scope (wire
  up read-back-and-merge) is genuinely complete; the gap is that T024/T025's
  validation call was never revisited once T035 introduced the
  existing-document read-back it should have validated against instead.
- **Missing tasks**: A task to move `decodePredicateDef`/`decodeTypeDef`'s
  validation call in `planSchemaNode`
  (`internal/app/schema/service/apply.go`) to run against the *merged*
  result (`final`) rather than the raw incoming `node`, so a section that
  omits a field an existing document already supplies is validated
  correctly — while a section for a *brand-new* definition (no existing
  document, `final == node` unchanged) still requires every mandatory field,
  preserving FR-008's original "new type definition" guarantee. A new E2E
  test for User Story 3 Acceptance Scenario 1's `Class` counterpart (an
  already-registered `Class` gains a new `Optional` predicate via a section
  that omits `description`/`merge`) is also missing — the existing T013
  suite only exercises this shape for `Property`.

## Root Cause Analysis

`planSchemaNode` (`internal/app/schema/service/apply.go:139-183`) validates
`decodePredicateDef(node)`/`decodeTypeDef(node)` against the **raw incoming
patch section** before ever reading back the existing document or
attempting the merge:

```go
switch node.Type {
case propertyType:
    if _, invalid := decodePredicateDef(node); invalid != "" { ... }
case classType:
    if _, invalid := decodeTypeDef(node); invalid != "" { ... }
}
path := dir + "/" + node.ID + ".md"
existing, existed, err := readExistingSchemaNode(store, path)
...
final := node
if existed {
    merged, _, _, merr := core.Merge(existing, node, metaIndex, sourceID)
    ...
    final = merged
}
```

This is the same architectural pattern already identified and fixed in
018/BUG-001 (the `merge` field was validated as mandatory on the raw
section, ignoring that it has no operational meaning) — except this time it
affects `description` (`Class`) and, by the same logic, `role`
(`Property`): any mandatory field the incoming section omits fails
validation immediately, even when an existing document on disk would
supply it once merged. The command's own "validate every node before
writing any" design (`schemaNodePlan`, FR-012) is sound and doesn't need to
change — only *what* gets validated does: the merged result for an
already-existing definition, the raw section (unchanged) for a brand-new
one.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to update spec artifacts — spec 018
   (`apply-schema-patch`), since `planSchemaNode`'s validation-ordering
   design is this feature's own implementation, not a shared dependency
   like 018/BUG-001's `decodeTypeDef` was. Clarify FR-008 (and FR-007 for
   symmetry) to state explicitly: a `Property`/`Class` section merging into
   an *existing* definition is validated against the merged result, not the
   raw incoming section alone; a section creating a *new* definition must
   independently satisfy every mandatory field. Add an Acceptance Scenario
   under User Story 3 covering "an existing `Class` gains a new `Optional`
   predicate via a section that omits `description`."
2. Run `/speckit.bugfix.verify` to confirm consistency.
3. Resume `/speckit.implement` to reorder `planSchemaNode`
   (`internal/app/schema/service/apply.go`): compute `final` (existing
   read-back + `core.Merge`) first, then run
   `decodePredicateDef(final)`/`decodeTypeDef(final)` against it instead of
   `node` — `final` already equals `node` unchanged when nothing existed
   beforehand, so new-definition completeness is preserved automatically.
   Add regression tests: a `Class` section adding an `Optional` predicate to
   an existing type with no `description`/`merge` in the section succeeds
   and preserves the existing description; a brand-new `Class`/`Property`
   section with a missing mandatory field still fails.
