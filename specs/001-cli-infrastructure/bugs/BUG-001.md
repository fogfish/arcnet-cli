# Bug Report: `build` workflow's GoReleaser step fails — config uses `version: 2` schema, CI installs GoReleaser v1 by default

**Type**: Dependency issue
**Severity**: High
**Feature**: 001-cli-infrastructure (branch `001-cli-infrastructure`)
**Reported**: 2026-07-01
**Status**: Patched
**Patched**: 2026-07-01

## Description

The `build` workflow's release step fails every time it runs, because `goreleaser/goreleaser-action@v5` was added without a pinned `version:` input. The action's own log confirms it silently falls back to a `~> v1` constraint when unpinned, installing GoReleaser `1.26.2`:

```
GoReleaser ~> v1 installed successfully
/opt/hostedtoolcache/goreleaser-action/1.26.2/x64/goreleaser release
  • starting release...
  ⨯ release failed after 0s   error=only configurations files on version: 1 are supported, yours is version: 2, please update your configuration
Error: The process '/opt/hostedtoolcache/goreleaser-action/1.26.2/x64/goreleaser' failed with exit code 1
```

`.goreleaser.yaml` declares `version: 2` (the current, non-deprecated GoReleaser config schema — added during implementation to clear `goreleaser check` deprecation warnings for `archives.format` and `snapshot.name_template`). GoReleaser 1.x cannot parse a `version: 2` config file at all, so the job fails immediately on every push to `main`, blocking every release.

The local `goreleaser release --snapshot --clean` dry-run (quickstart.md Step 4 / tasks.md T025) passed cleanly during implementation because the local machine has GoReleaser 2.x installed via Homebrew — a different runtime than what CI's unpinned action resolves to. The dry-run validated the config against the wrong GoReleaser major version, masking this defect until a real CI run.

## Artifact Traceability

### spec.md
- **Affected user story**: User Story 3 — Automated versioned release on merge (Priority: P3)
- **Affected requirements**:
  - FR-009: "The default-branch pipeline MUST invoke the release configuration automatically after a new version tag is created, publishing the resulting artifacts" — currently never succeeds
  - FR-013: "automated checks and pipelines MUST require no ... manual local setup beyond what is provisioned in the CI environment" — the config only works if the CI environment happens to resolve the same GoReleaser major version used locally, which is not guaranteed and is not currently pinned
- **Gap identified**: spec.md does not pin a GoReleaser major version (nor did it need to at the requirements level), but neither does any downstream artifact — the CI/release contract is silent on which config schema version the action must install, which is exactly the seam this bug fell through.

### plan.md
- **Affected sections**: Project Structure (`.goreleaser.yaml`), Technical Context (no line pins a GoReleaser version, only Go 1.26)
- **Impact**: No plan change needed for scope; the plan already treats `.goreleaser.yaml` and `build.yml` as one coupled unit. The bug is that the two were edited without cross-checking that the workflow's action installs a GoReleaser binary compatible with the config's declared schema version.

### research.md
- **Affected section**: "Decision: GoReleaser configuration shape" — documents mirroring the reference project's config shape (builds/archives/checksum/changelog/brews) but never states a target GoReleaser major version or an action `version:` pin. This decision was made before `version: 2` was added to `.goreleaser.yaml` during implementation (to resolve `goreleaser check` deprecation warnings), so research.md is now silent on a choice that materially affects CI behavior.

### tasks.md
- **Affected tasks**:
  - T021 — created `.goreleaser.yaml`; the task description (and research.md) never specified a schema version, so `version: 2` was added later, ad hoc, without updating T024
  - T024 — added the `goreleaser/goreleaser-action@v5` step but did not set a `version:` input compatible with `version: 2` in the config
  - T025 — ran the local snapshot dry-run and reported success; this is a **false completion** in the sense that it validated the config against a locally-installed GoReleaser 2.x binary, not against what CI's unpinned action actually installs, so it did not catch this defect
- **Missing tasks**: A task to pin `goreleaser/goreleaser-action@v5`'s `version:` input to a GoReleaser 2.x release (or constraint, e.g. `"~> v2"`), matching `.goreleaser.yaml`'s `version: 2` schema declaration

## Root Cause Analysis

Implementation drift introduced mid-task: `.goreleaser.yaml` was upgraded from the unversioned/v1 config shape to the `version: 2` schema during implementation polish (to silence `goreleaser check` deprecation warnings surfaced by the locally installed GoReleaser 2.x), but the corresponding `goreleaser-action` step in `build.yml` was never updated to pin a matching major version. `goreleaser-action@v5` defaults to installing GoReleaser under a `~> v1` constraint when no `version:` input is given, so the config (v2 schema) and the CI-installed binary (v1) are incompatible by construction. The local dry-run task (T025) used a different, newer GoReleaser binary than CI ever will, so it validated the wrong pairing and reported a false pass.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to:
   - Update research.md's "GoReleaser configuration shape" decision to explicitly record the target GoReleaser major version (2.x) as a stated choice, not an implicit side effect of the local dev machine's installed CLI
   - Add a task (or amend T024) requiring `goreleaser/goreleaser-action@v5`'s `with.version:` input to be pinned to a GoReleaser 2.x release/constraint in `.github/workflows/build.yml`
2. Run `/speckit.bugfix.verify` to confirm spec/plan/tasks consistency
3. Resume `/speckit.implement` to apply the code fix: add `version: "~> v2"` (or an explicit pinned version, e.g. `"2.5.0"`) to the `goreleaser/goreleaser-action@v5` step's `with:` block in `.github/workflows/build.yml`, then re-verify the snapshot dry-run still matches what CI will run
