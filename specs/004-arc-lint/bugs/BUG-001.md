# Bug Report: `checkDerivedProvenance` false-positives on the tool's own timeline index files, and can't recognize `applyTimeline`'s rendered bullet as a link at all

**Type**: Spec gap
**Severity**: Medium
**Feature**: 004-arc-lint (`arc lint`)
**Reported**: 2026-07-04
**Status**: Resolved
**Patched**: 2026-07-04
**Resolved**: 2026-07-04 — see tasks.md T057-T058

## Description

Running `arc lint` against a graph produced by `arc apply` reports:

```text
❌ timeline/monthly/2026-07.md — [derivedProvenance] does not link to any source node
```

for a file that plainly does reference a source document:

```markdown
---
kind: timeline
period: 2026-07
granularity: monthly
---
# July 2026

- [[dmitry-2026-graph-thought]] — *Ontologies, Graph Structures, and LLM-Based Knowledge Management* (Dmitry) — 2026-07-03
```

This is two distinct, compounding defects:

1. **`checkDerivedProvenance` should never run against a `timeline`-kind
   node at all.** `internal/app/lint/kernel`'s `checkDerivedProvenance`
   ([rules_links.go:67-88](../../../internal/app/lint/service/rules_links.go#L67-L88))
   exempts only `node.Kind == "source"`; every other kind, including the
   format's own reserved `timeline` index kind (CORE §12.2 — "index
   kinds are not carried," `specs/003-apply-patch/research.md` D8/D8b),
   is treated as "a node distilled from a document" that must link back
   to a source. A timeline period file is not that — it is the tool's
   own chronological index *referencing* many documents' entries over
   time, not content synthesized *from* one document.
2. **Even if the check should apply, `core.ParseNode` cannot recognize
   `applyTimeline`'s own rendered bullet as a link at all.**
   `internal/core`'s list-item link regex
   (`listItemPattern`, [markdown.go:488](../../../internal/core/markdown.go#L488))
   requires the *entire* list item to be exactly `[[Target]]` (or
   `predicate:: [[Target]]`), anchored with `$`. `applyTimeline`'s own
   rendered bullet (`core.TimelineEntry`) appends trailing content after
   the link — `— *title* (author) — date` — so the whole line never
   matches, `collectListLinks` silently produces zero `Edges` for this
   bullet, and `checkDerivedProvenance` (even fixed for point 1) would
   still see no resolvable links for a *different* reason: the file's
   own recorded entries are simply invisible to `core.ParseNode`'s
   generic link recognition.

Confirmed directly: `core.ParseNode` on this exact monthly period file
content returns `Edges: []`, `HRefs: []`, `Links: map[]` — the bullet's
`[[dmitry-2026-graph-thought]]` reference is completely lost to any
generic, `core.Node`-based consumer, not just to this one lint rule.

## Artifact Traceability

### spec.md

- **Affected requirement**: **FR-006** — "For every derived node (any
  node distilled from a document, i.e. every node other than a `source`
  node itself), the tool MUST verify it links to at least one `source`
  node..."
- **Gap identified**: the requirement's own parenthetical — "i.e. every
  node other than a `source` node itself" — over-generalizes "derived
  node" (CORE §3.4's actual phrase, quoted correctly in research.md D8's
  own rationale: "a node distilled from a document") to include every
  non-source kind, including the format's own reserved `timeline` index
  kind, which is categorically not "distilled from a document" — it's
  the tool's own derived index over many documents, explicitly excluded
  from ordinary patch-carried content by CORE §12.2. FR-006 needs a
  carve-out for `timeline` (and, by the same reasoning, any other
  future tool-owned index kind, should one ever exist) alongside the
  existing `source` exemption.
- **Related requirement**: **FR-015** ("Every violation names rule +
  file + line (or 'not applicable')") is not itself wrong here, but a
  correctly-scoped FR-006 would prevent this violation from being
  generated in the first place, rather than relying on FR-015 to merely
  format it clearly.

### plan.md / research.md

- **Affected section**: research.md **D8** ("For every node whose
  `Kind != 'source'` ... lint checks whether at least one of its
  `HRefs`, `Edges`, or `Links[*].Seq` entries has a `Target` that
  resolves ... to a node whose `Kind == 'source'`") — inherits FR-006's
  own over-generalization verbatim; needs a decision explicitly
  exempting `timeline` (the only other CORE built-in kind besides
  `source`, per `internal/core/rules.go`'s `CoreMergeRules`) from this
  check, alongside a design note on point 2 above (the link-recognition
  gap).
- **Related section**: research.md **D3** — the raw-text line locator —
  is unaffected; this bug is about which nodes the *rule* runs against
  and what `core.ParseNode` recognizes as a link, not about line
  attribution once a violation exists.

     and is not required to be, recognized by `core.ParseNode`'s generic
     link grammar — `arc lint`'s node enumeration should not assume
     every reference inside every node file is a structurally
     recognized `Link`, and no other CORE §14 rule should be designed
     to depend on that assumption for `timeline`-kind files specifically.

## Root Cause Analysis

`research.md` D8's decision (and FR-006 before it) were written and
verified against `entity`/`resource`/domain-kind fixtures — real content
nodes CORE §3.4 actually describes as "distilled from a document" — and
never checked against a real `timeline`-kind period file, since no
E2E/unit test fixture for `004-arc-lint`'s Phase 3 work included one.
`internal/core.CoreMergeRules` already singles out `timeline` as one of
exactly four format-fixed kinds (alongside `source`/`entity`/`resource`),
and `003-apply-patch/research.md` D8b (as revised by BUG-006) already
establishes that a `timeline`-kind node is never ordinary, patch-carried
content — `004-arc-lint`'s own design should have inherited that same
distinction when scoping FR-006, but source/entity/resource were the
only kinds exercised during `checkDerivedProvenance`'s own test-fixture
design.
