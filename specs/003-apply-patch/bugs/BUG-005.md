# Bug Report: A patch-carried `kind: timeline` node section lands in a bogus `timelines/` folder, colliding with the tool's own derived timeline index

**Type**: Spec gap
**Severity**: High
**Feature**: 003-apply-patch (`arc apply`)
**Reported**: 2026-07-04
**Status**: Superseded — this bug's own fix (T097-T099: refuse the entire patch) was itself wrong; see [BUG-006](BUG-006.md) for the corrected resolution
**Patched**: 2026-07-04
**Resolved**: 2026-07-04 — see tasks.md T097-T101 (reopened same-day by BUG-006; corrected via T102-T106)

## Description

`arc lint` (specs/004-arc-lint) surfaced a real basename collision in an
applied graph:

```text
❌ [uniqueBasename] basename "2026-07" is used by more than one file: timeline/monthly/2026-07.md, timelines/2026-07.md
```

`timeline/monthly/2026-07.md` is the graph's own, correctly-derived monthly
timeline period file (CORE §9.4, research.md D8,
[apply.go:213](../../../internal/app/graph/service/apply.go#L213),
`applyTimeline`). `timelines/2026-07.md` (plural, no `yearly`/`monthly`
subfolder) is a second, bogus file — produced by `service.Apply`'s
**generic** per-node-contribution path, not by `applyTimeline`.

The generic path is `nodePath`/`nodeFolder`
([apply.go:31-50](../../../internal/app/graph/service/apply.go#L31-L50)):

```go
var coreKindFolders = map[core.Kind]string{
	"source":   "sources",
	"entity":   "entities",
	"resource": "resources",
}

func nodeFolder(kind core.Kind) string {
	if folder, ok := coreKindFolders[kind]; ok {
		return folder
	}
	s := string(kind)
	if strings.HasSuffix(s, "s") {
		return s
	}
	return s + "s"
}

func nodePath(node core.Node) string {
	return nodeFolder(node.Kind) + "/" + node.ID + ".md"
}
```

`coreKindFolders` has no entry for `"timeline"`. If a patch's body contains
a `# Timeline` H1 section (`internal/core.ParsePatch`'s generic H1-kind/
H2-node grammar has no rule against it — any kind name is accepted), the
`Apply` loop
([apply.go:117-168](../../../internal/app/graph/service/apply.go#L117-L168))
treats it as an ordinary node contribution: `nodeFolder("timeline")` falls
through to the generic pluralization branch (`"timeline"` doesn't end in
`"s"`, so it becomes `"timelines"`), and the node is created at
`timelines/<published-period-id>.md` — landing right next to, and
colliding with, whatever real `timeline/monthly/<period>.md` or
`timeline/yearly/<period>.md` file `applyTimeline` derives independently
for the same publication date. Two distinct code paths in the same
`Apply` call write conflicting files for the same conceptual period, with
no error, no warning, and no idempotency check against each other.

This is exactly `research.md` D6's own documented invariant, already
stated but never enforced:

> CORE §12.2 excludes index kinds (`timeline`) from patches entirely
> ("Index kinds (`timeline`) are not carried")

and spec.md's own FR-022 aside:

> the tool's own built-in timeline index (FR-010): that index is never
> itself patch-carried

Both documents assert the invariant as *true by construction*. Nothing in
`internal/core.ParsePatch` or `internal/app/graph/service.Apply` actually
checks or enforces it — a patch is free to declare `# Timeline` regardless,
and when one does, the tool doesn't reject it (per CORE §12.2, it should
be invalid patch structure), doesn't route it consistently with
`applyTimeline`'s own two-tier `timeline/yearly/`+`timeline/monthly/`
layout, and doesn't even use a folder name (`timelines/`, plural) that
matches the format's real, singular `timeline/` folder — a second,
independent naming inconsistency layered on top of the missing guard.

  - **FR-005**/**FR-006** (create-or-merge for "every reconstructed

    section from being created/merged through the generic path.

    not state that a patch attempting to carry a `timeline`-kind section

    closest existing textual anchor for this invariant, but it appears
    only as clarifying context for `append`'s semantics, not as its own
    standalone, enforceable MUST.
- **Gap identified**: spec.md never states that `timeline` is a
  format-reserved kind a patch MUST NOT declare, and never states what the
  tool MUST do if a patch violates that (refuse the whole application, per
  FR-002's existing "patch body does not follow the expected node-section
  structure" refusal — the most consistent existing precedent — or
  silently ignore/drop the section). Without that requirement, the
  generic per-node reconstruction path (FR-004) has no reason to special-
  case `timeline` at all, and doesn't.

## Root Cause Analysis

CORE §12.2's "index kinds are not carried" rule was correctly understood
and documented twice (research.md D8, spec.md FR-022's aside) but never
promoted to an actual functional requirement with a corresponding guard.
Because `internal/core.ParsePatch`'s H1-kind grammar accepts any kind name
structurally (it has no fixed-kind allowlist — by design, since domain/
extension kinds must also parse), nothing distinguishes `# Timeline` from
`# Hypothesis` at parse time. `internal/app/graph/service.Apply`'s generic
per-node loop then treats a `timeline`-kind section exactly like any other
node contribution, and `nodeFolder`'s fallback pluralization (built for
genuinely novel domain/extension kinds, e.g. `"hypothesis"` →
`"hypothesis"`, `"logentry"` → `"logentrys"`) produces `"timelines"` for
the one kind name that should never reach this function at all. The
result is silent: no error, no warning, just a second file that happens to
collide with the tool's own derived index the moment the arbitrarily-
chosen node ID matches a real period code (e.g. `"2026-07"`) — which is
exactly what happened here, and would just as easily *not* collide (and
go completely unnoticed) for a differently-named `timeline`-kind node ID.
`arc lint`'s `uniqueBasename` check (specs/004-arc-lint) is what surfaced
this only after the fact; nothing in `arc apply` itself catches it at
ingest time, which is the actual defect.
