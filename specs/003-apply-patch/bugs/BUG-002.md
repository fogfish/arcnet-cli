# Bug Report: `arc apply` crashes with "append is not a recognized merge operation"

**Type**: Implementation drift
**Severity**: High
**Feature**: 003-apply-patch (`arc apply`)
**Reported**: 2026-07-03
**Status**: Resolved
**Patched**: 2026-07-03
**Resolved**: 2026-07-03 — see tasks.md T084-T087

## Description

`arc apply` fails outright when it needs to merge a node whose resolved
merge behavior is `append`:

```
❌ Applying node contributions failed: [github.com/fogfish/arcnet-cli/internal/core.Merge 43] append is not a recognized merge operation:

 ❌ append is not a recognized merge operation
```

`append` is one of CORE §10's four fully-documented merge operations
("insert the contribution into an ordered list, keyed for uniqueness so
re-insertion is a no-op") — it is not exotic or out-of-spec. `internal/core.Merge`'s
`switch` statement, however, has no `case MergeAppend` at all; every call
with `op == core.MergeAppend` falls through to the `default` branch and
returns `ErrUnknownMergeOp`. This is reachable in two ways given the
current implementation:

1. A patch's H1 section names the built-in `timeline` kind (`core.CoreMergeRules["timeline"] = MergeAppend`
   always resolves via `rules.Lookup`), and a node under it already exists in
   the graph — nothing in `core.ParsePatch`/`service.Apply` currently
   rejects a patch that carries a `# Timeline` section, even though CORE
   §12.2 says index kinds "are not carried."
2. A graph registers a **domain/extension kind** with `mergeRules: <kind>: append`
   in `.arc/config.yml` (spec.md FR-019 permits any of the format's fixed
   merge behaviors, and `.arc/config.yml`'s own load path — `internal/app/config/service.Resolve` —
   validates only against the fixed `MergeOp` vocabulary, which includes
   `"append"`), then applies a second patch that re-mentions a node of that
   kind.

Both paths hit the exact same unimplemented `case` and crash with the raw,
un-humanized internal error text shown above.

## Artifact Traceability

### spec.md

- **Affected requirements**: **FR-019** ("The tool MUST provide a way for a
  graph to register a domain/extension node kind together with its
  declared merge behavior — one of the format's fixed merge behaviors
  (none / union / union-first-writer / other declared equivalent)") and
  **FR-020** ("...the tool MUST create or merge it using that kind's
  registered merge behavior, with the same creation/merge/conflict-flagging
  guarantees (FR-005 through FR-013) that apply to the format's built-in
  kinds"). FR-020 makes an unqualified promise — *any* registered behavior
  gets full support — that the implementation does not keep for `append`.
- **Gap identified**: unlike `none` (FR-007), `union` (FR-008), and
  `union-first-writer` (FR-009), spec.md has **no functional requirement
  at all describing what `append` means for a domain-registered kind that
  is not the built-in `timeline` index**. FR-010 only describes timeline's
  own, hardwired append behavior (yearly/monthly period files). This is a
  genuine spec gap, not just an implementation oversight: even with a code
  fix, there is currently no requirement text saying what "append" should
  do when merging two ordinary (non-index) node contributions — e.g.,
  whether it behaves like `union` (dedupe, order-preserving), or something
  narrower.

### plan.md / research.md

- **Affected section**: research.md **D6** ("Merge algebra (CORE §10) lives
  in `internal/core`"), which states: *"`MergeAppend` is not handled by
  `Merge` at all — CORE §12.2 excludes index kinds (`timeline`) from
  patches entirely... so no patch-carried `Node` is ever merged with this
  operation."*
- **Root cause of the drift**: this reasoning is valid **only** for the
  format's one fixed index kind (`timeline`), which CORE §12.2 explicitly
  excludes from patches. It does not hold for a domain/extension kind a
  graph registers via FR-019 with `append` as its chosen behavior — such a
  kind's nodes *are* ordinary patch-carried content, not an index, and
  FR-020 promises them the same guarantees as any other registered
  behavior. D6 over-generalized a true, narrow fact (timeline is
  patch-excluded) into a false, broad one (append is never reachable by
  `Merge`), and `internal/core/merge.go` was implemented to match that
  overly broad (and incorrect) conclusion.
- **Impact**: D6 needs a revised decision defining generic, non-timeline
  `append` semantics for `internal/core.Merge`, and `contracts/ast-contract.md`'s
  `Merge` documentation (currently silent on `append`) needs the same
  update.

- **False completions**: T036 and T038 are both marked `[X]` but did not

- **Missing tasks** (for `/speckit.bugfix.patch` to add):

     `unionLinks`/`unionScalarSlice` helpers `mergeCore` already has), with
     scalar fields treated first-writer-silent (no conflict flagging,
     consistent with "keyed for uniqueness so re-insertion is a no-op"
     having no concept of a scalar-field conflict at all).

  3. Add spec.md coverage (a new FR, or an amendment to FR-019/020)
     stating explicitly what `append` means for a non-timeline,
     domain-registered kind.
  4. Add an E2E test in `cmd/arc/graph/apply_test.go` registering a
     domain kind as `append` and applying two patches that both mention a
     node of that kind, asserting no crash and correct list-insertion

## Secondary, incidentally-discovered issue (same crash, not the main bug)

The pasted error shows the raw, bracket-prefixed internal fault text
(`[github.com/fogfish/arcnet-cli/internal/core.Merge 43] ...`) **twice** —
once from `internal/app/graph/service.Apply`'s `--verbose` progress
reporting (`reporter.Error(label, err)`, added by BUG-001's fix, which
prints `err.Error()` raw, unlike `cmd/arc/main.go`'s `humanize()` which
strips the `[pkg.Func line]` prefix for the final CLI error line), and once
correctly humanized via `cmd/arc/main.go`. This is a minor DS-07/Principle
XII regression from BUG-001 (raw internal error text should never reach
the terminal, even under `--verbose`) — flagged here since it is visible
in the exact reproduction, but it is independent of the `append` gap above
and should be tracked/fixed separately if the user wants it addressed
(e.g., `stderrReporter.Error` should call the same `humanize`-equivalent
sanitization `cmd/arc/main.go` already has, ideally by moving that helper
into `internal/bios` so both call sites share it).

## Root Cause Analysis

A design-time over-generalization in research.md D6: the true, narrow fact
that CORE §12.2 excludes the *fixed* `timeline` kind from patches was
mistakenly extended to "no patch-carried node ever uses `append`," ignoring
that FR-019 lets any domain/extension kind opt into `append` too.
`internal/core.Merge` was implemented to that flawed premise (append is
simply unreachable, so treat it as an error), and the gap went undetected
because T038's table-driven tests never actually added the `MergeAppend`
case its own task description promised. `.arc/config.yml`'s load path
(`internal/app/config/service.Resolve`) has no additional validation
narrowing which of the four `MergeOp` values a domain kind may choose, so
nothing stops a user from registering `append` and hitting this at
`arc apply` time instead of at registration time.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to:
   - Revise research.md D6 with concrete, non-timeline `append` semantics
     (recommended: identical list/edge union behavior to `MergeUnion`, with
     no scalar-conflict flagging).
   - Add a spec.md requirement (new FR, or amend FR-019/FR-020) stating
     this explicitly.
   - Reopen T036/T038 and add new tasks for the `Merge` case, its unit
     test, and a new E2E test.
2. Run `/speckit.bugfix.verify` to confirm consistency.
3. Resume `/speckit.implement` to add the `case MergeAppend` to
   `internal/core.Merge`, its unit test, and the E2E test — and, if
   desired, fix the secondary raw-error-leak noted above in the same pass.
