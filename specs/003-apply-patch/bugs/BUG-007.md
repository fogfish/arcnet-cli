# Bug Report: A yearly timeline period file (`period: 2026`) fails to parse via `core.ParseNode` because YAML decodes the bare year as an integer, not a string

**Type**: Implementation drift
**Severity**: Medium
**Feature**: 003-apply-patch (`arc apply` / `internal/core`)
**Reported**: 2026-07-04
**Status**: Resolved
**Patched**: 2026-07-04
**Resolved**: 2026-07-04 — see tasks.md T107-T108

## Description

Running `arc lint` (specs/004-arc-lint) against a graph produced by `arc apply`
reports:

```text
❌ timeline/yearly/2026.md:1 — [frontMatter] [github.com/fogfish/arcnet-cli/internal/core.ParseNode 103] patch manifest is missing a mandatory field (kind: patch, document, published):
```

despite `timeline/yearly/2026.md` being a perfectly well-formed file `arc
apply` itself wrote:

```markdown
---
kind: timeline
period: 2026
granularity: yearly
---
# 2026

- [[dmitry-2026-graph-thought]] — *Ontologies, Graph Structures, and LLM-Based Knowledge Management* (Dmitry) — 2026-07-03
```

The error message is misleading (it's `ErrManifestInvalid`'s shared text,
written for the `ParsePatch` case — see `internal/core/errors.go`) but the
underlying cause is real: `internal/core.ParseNode` calls `deriveNodeID`
([markdown.go:135-142](../../../internal/core/markdown.go#L135-L142)):

```go
func deriveNodeID(manifest map[string]any) string {
	for _, key := range []string{"id", "title", "period"} {
		if v, ok := manifest[key].(string); ok && v != "" {
			return v
		}
	}
	return ""
}
```

A timeline period file has no `id`/`title` field, so `deriveNodeID` falls
back to `period`. For a **monthly** file, `period: 2026-07` decodes as a
YAML string (it doesn't match a pure integer or a full ISO date), so the
`.(string)` assertion succeeds and `ParseNode` works. For a **yearly**
file, `period: 2026` is a bare, unquoted 4-digit scalar — YAML's implicit
typing decodes this as an **integer**, not a string. The `.(string)`
assertion then fails, `deriveNodeID` returns `""`, and `ParseNode` returns
`ErrManifestInvalid` for a file that is not actually malformed — just
serialized with a type-ambiguous scalar.

Confirmed directly:

```go
core.ParseNode(strings.NewReader("---\nkind: timeline\nperiod: 2026\ngranularity: yearly\n---\n# 2026\n\n- [[foo]]\n"))
// → error: patch manifest is missing a mandatory field ...
core.ParseNode(strings.NewReader("---\nkind: timeline\nperiod: 2026-07\ngranularity: monthly\n---\n# July 2026\n\n- [[foo]]\n"))
// → nil error, id: "2026-07"
```

This does not break `arc apply` itself — re-inserting into an existing
period file goes through `upsertTimelinePeriod`'s own bespoke
`parseTimelineEntries` regex scan
([apply.go:392-406](../../../internal/app/graph/service/apply.go#L392-L406)),
never `core.ParseNode` — but it does break every *other*, generic
consumer of graph node files that relies on `core.ParseNode` succeeding
for every `*.md` file in the graph, which `contracts/ast-contract.md`
documents as the whole point of keeping `internal/core` shared: "Public
surface any future graph-reading command (`lint`, `retract`, `index
build`) can depend on." `arc lint` is exactly such a consumer, and every
yearly period file in every graph will trip this the same way.

## Artifact Traceability

### spec.md

- **Affected section**: no functional requirement in `003-apply-patch/spec.md`
  directly addresses timeline-period-file YAML serialization (FR-010
  only requires the period files exist and carry a chronologically-
  ordered entry — it does not specify field types at the YAML level).
- **Gap identified**: neither `spec.md` nor `contracts/ast-contract.md`
  ever states that every node file `arc apply` writes must remain
  parseable by the *generic* `core.ParseNode`/`deriveNodeID` contract —
  an implicit expectation given `ast-contract.md`'s own stated purpose,
  but never made an explicit requirement, which is exactly how this
  gap went unnoticed until a second consumer (`arc lint`) exercised it.

### plan.md / research.md

- **Affected section**: research.md **D8** (Timeline derivation) —
  describes `applyTimeline`'s period-file writing but does not specify
  how the `period` front-matter field must be serialized to remain
  unambiguously a string on read-back.
- **Root cause location**: `internal/app/graph/service/apply.go`'s
  `upsertTimelinePeriod`
  ([apply.go:462-497](../../../internal/app/graph/service/apply.go#L462-L497))
  hand-writes YAML front matter via raw string concatenation
  (`buf.WriteString("period: " + period + "\n")`), bypassing the
  codebase's own YAML-safe encoding path (`internal/core`'s
  `renderFrontMatter`/`encodeYAMLNode`, used by `core.RenderNode` for
  every other node file, which encodes each value through `yaml.Node.Encode`
  and would tag a numeric-looking string with an explicit
  string style/quote, never leaving it to bare, implicit YAML typing).
  This is why every *other* node file `arc apply` writes doesn't have
  this problem — `RenderNode` is the only other file-writing path, and
  timeline period files are the one node-shaped file in this codebase
  not written through it.
- **Impact**: research.md needs a decision either quoting `period`
  explicitly when writing timeline period files (`period: "2026"`), or
  making `internal/core.deriveNodeID`'s `period` fallback tolerant of a
  non-string YAML scalar (e.g. `fmt.Sprint` on whatever underlying type
  came back, not a strict type assertion).

### tasks.md

- **Affected tasks**: **T039** (`internal/core/timeline.go`) and **T063**
  (`upsertTimelinePeriod`'s implementation) — both correctly implemented
  research.md D8 as written; D8 itself never specified the string-safety
  requirement.
- **Missing tasks** (for `/speckit-bugfix-patch` to add):
  1. Either quote `period` in `upsertTimelinePeriod`'s front-matter
     write, or make `deriveNodeID`'s `period` case tolerant of any
     scalar type.
  2. Add a unit test in `internal/core/markdown_test.go` (or
     `internal/app/graph/service/apply_test.go`) asserting a yearly
     timeline period file (bare 4-digit `period` value) round-trips
     through `core.ParseNode` successfully.

## Root Cause Analysis

`upsertTimelinePeriod` is the one place in this codebase that writes a
node-shaped file's front matter by hand-built string concatenation
instead of through `internal/core`'s own YAML-safe encoder
(`RenderNode`/`encodeYAMLNode`), which every other node-writing path
uses and which would have avoided this exact class of bug by explicitly
encoding the Go string value with YAML string semantics regardless of
its lexical shape. Because a monthly period code (`"2026-07"`) happens
to not be a valid bare YAML scalar for int/float/bool/null/date, the bug
was invisible for monthly files and only surfaces for yearly ones (a
bare 4-digit number, always numeric-looking) — this asymmetry is why it
went unnoticed through `003-apply-patch`'s own test suite (which never
exercises the yearly file through `core.ParseNode`, only through
`arc apply`'s own bespoke `parseTimelineEntries`) and only surfaced once
`004-arc-lint`'s node enumeration (which does call `core.ParseNode`
generically on every `*.md` file) walked a real yearly period file.
