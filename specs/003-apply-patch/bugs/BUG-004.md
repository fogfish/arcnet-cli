# Bug Report: `MergeUnion` flags near-certain false-positive conflicts on regenerated scalar fields

**Type**: Spec gap
**Severity**: High
**Feature**: 003-apply-patch (`arc apply`)
**Reported**: 2026-07-03
**Status**: Resolved
**Patched**: 2026-07-03
**Resolved**: 2026-07-03 â€” see tasks.md T092-T096

## Description

`entity`'s registered merge behavior, `MergeUnion` (`core.CoreMergeRules["entity"]`,
[rules.go:26](../../../internal/core/rules.go#L26)), treats every scalar field
(`Text`, `Notes`, non-list `Attrs`) the same way: identical or empty-incoming
changes nothing, and any other divergence between two non-empty values is
**unconditionally** flagged as a conflict â€” the existing value is kept but
wrapped in a git-style `<<<<<<< / ======= / >>>>>>>` marker
([merge.go:92-104](../../../internal/core/merge.go#L92-L104),
[merge.go:196-207](../../../internal/core/merge.go#L196-L207)).

That rule implicitly assumes a scalar field is written once and then edited
rarely, by independent contributors â€” a real conflict is meant to be the
exceptional case. For `entity` nodes carrying **derived/regenerated** scalar
content, that assumption doesn't hold, and the observed real-world patch
below demonstrates it:

```yaml
kind: entity
id: LLM
score-c: |-
    <<<<<<< existing
    0.13432835820895522
    =======
    0.28125
    >>>>>>> dmitry-2026-graph
score-z: |-
    <<<<<<< existing
    2.2522964920476682
    =======
    2.8783652519773235
    >>>>>>> dmitry-2026-graph
```

- **`score-c`/`score-z`** are graph-analytics scores (e.g. centrality)
  recomputed from scratch on every ingest run. As the graph evolves between
  runs, the recomputed float is essentially never bit-identical to the
  stored one, and the comparison is exact string equality
  (`fmt.Sprint(ev) == fmt.Sprint(iv)`, [merge.go:197](../../../internal/core/merge.go#L197))
  with no tolerance for expected numeric drift.
- **`Text`** on an `entity` is wholesale-regenerated prose from an LLM
  synthesis pass each run, not an incremental hand-edit of the existing
  description. Two independently-generated paraphrases of the same entity
  will almost never be textually identical, even when they describe exactly
  the same thing.

Because both fields are recomputed/regenerated on essentially every apply
(not independently-and-rarely edited facts), a conflict marker on them is
the **default outcome**, not the exception the mechanism was designed for.
In the user's words: "these conflicts always happen, meaning each apply
would not be successful" â€” `MergeUnion` for `entity` nodes is effectively
unusable for repeat ingestion of the same evolving subject.

## Proposed Fix (from user)

1. **Scalar `Attrs`** merged under `union`: first-writer-wins with **no**
   conflict flagging at all (same numeric/attribute drift across runs is
   expected, not a disagreement to surface for manual review).
2. **`Text`** merged under `union`: not a single scalar comparison.
   - Split `existing.Text` into paragraphs.
   - For each incoming paragraph, compute Jaccard similarity on shingles
     (word n-grams) against each existing paragraph.
   - If the best match is **> 0.8** similar, treat the incoming paragraph as
     same-content and skip it (no duplication, no conflict).
   - Otherwise, append the incoming paragraph as a new paragraph onto the
     existing `Text` (an additive merge, not an overwrite-or-flag).

This replaces conflict-marker emission for `entity`-shaped content with two
different silent-merge strategies matched to what each field actually is:
recomputed metadata (attrs) vs. independently-regenerated prose (text).

    is deterministically recomputed/regenerated every run (no single

    itself is not wrong for the fields it was written for (e.g. `resource`

    inherited FR-013's flagging unconditionally, with no carve-out for

    companion concept â€” a similarity-tolerant, additive text merge â€” that
    does not exist in the spec today; `Text` is currently lumped in with
    generic "scalar Attrs."
- **Gap identified**: spec.md defines exactly four merge behaviors (none /
  union / union-first-writer / append) and treats "scalar field" as one
  undifferentiated bucket per behavior. There is no requirement
  distinguishing a **stable, human-set** scalar (where a genuine conflict
  is meaningful and rare) from a **recomputed/regenerated** one (where
  divergence is the expected steady state and conflict-flagging is noise),
  nor any requirement describing paragraph-level, similarity-based text
  combination as an alternative to whole-field flagging.

## Root Cause Analysis

`MergeUnion`'s scalar-field policy (research.md D6, implemented in
`internal/core/merge.go`) was designed around FR-013's "one authoritative
value" model: a scalar diverges rarely, and when it does, a human should
decide. That model was never validated against a node kind whose scalar
fields are **routinely and legitimately regenerated** by the same
ingestion pipeline on every run â€” `entity`'s `score-c`/`score-z`
(recomputed graph-analytics metrics) and `Text` (LLM-resynthesized prose,
not incrementally hand-edited). For such fields, "divergence" is the
expected steady state, not a signal of two writers disagreeing. Applying
FR-013's flagging rule unconditionally to every scalar, regardless of
whether the field is human-authored-once or pipeline-regenerated-always,
turns `MergeUnion` into a mechanism that guarantees a conflict on nearly
every re-application of a previously-ingested `entity` â€” the opposite of
the "safe union default" spec.md FR-018 and research.md's own "always
succeeds" framing promise.

This was not caught earlier because none of T038's table-driven test cases
nor T017's US2 E2E scenarios exercise re-applying a patch for a node whose
kind (`entity`) carries scalar content that changes on every run; the
existing "first-writer scalar" and "conflict-marker embedding" test cases
both use static, hand-picked divergent values, which reads as a legitimate
conflict rather than the systemic, every-run case a recomputed float or
regenerated paragraph represents.
