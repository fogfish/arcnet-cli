# Bug Report: `arc apply` silently drops edges from patch node bodies

**Type**: Spec conflict
**Severity**: Critical
**Feature**: 003-apply-patch (`arc apply`)
**Reported**: 2026-07-03
**Status**: Resolved
**Patched**: 2026-07-03
**Resolved**: 2026-07-03 — see tasks.md T088-T091

## Description

Applying a real-world patch whose node body groups predicate edges under
**bold labels** (CORE §12.2's own canonical convention: "node bodies use
bold labels, never headings") silently loses every predicate-grouped block
except the first list encountered. Given a patch node:

```markdown
## Arcnet-spec

```yaml
category: [independent, abstract, occurrent, script]
score-c: 0.3283582089552239
score-z: 1.0650681152742885
```

A lightweight ontology specification ... defining core graph structures,
[[Article Extension]]s, and thought extensions ...

**Mentioned In**
- mentionedIn:: [[dmitry-2026-graph]]

**Referenced By**
- referencedBy:: [[Core Thoughts Extension]]

**Related**
- related:: [[Article Extension]]
```

the graph node `arc apply` writes ends up as:

```markdown
---
kind: entity
id: Arcnet-spec
category: [independent, abstract, occurrent, script]
score-c: 0.3283582089552239
score-z: 1.0650681152742885
---
# Arcnet-spec

A lightweight ontology specification ... defining core graph structures,
Article Extensions, and thought extensions ...

**Mentioned In**

- mentionedIn:: [[dmitry-2026-graph]]

**Referenced By**

**Related**
```

**Both `referencedBy:: [[Core Thoughts Extension]]` and `related:: [[Article Extension]]`
are gone entirely** — not merely unlinked, but absent from the file and
therefore from the graph. `mentionedIn` survives only because it happens
to be the first list after the leading prose (misclassified as an
ungrouped bare edge rather than a `Mentioned In`-titled block, though this
is not currently observable since `RenderNode` does not re-emit a
`## Mentioned In` heading for it — the bold label paragraph text itself
survived only because it was independently swept into `Notes`).

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 1 — Ingest a brand-new document into
  the graph (P1), Acceptance Scenario 1 ("the tool creates a new file for
  every node the patch carries... each written in the canonical node
  format").
- **Affected requirements**: **FR-004** ("For every node section the patch
  carries, the tool MUST reconstruct the full node — its kind, identity,
  scalar attributes, and body content — from the patch's declared
  structure") is directly violated: declared body content (two of three
  predicate-grouped blocks) is not reconstructed, it is discarded. The
  **Key Entities** section's "Node Contribution" definition ("its kind,
  identity, attributes, and body — as reconstructed by the tool") implies
  the same completeness FR-004 states explicitly.
- **Gap identified**: none in spec.md itself — FR-004 is unambiguous and
  correct. The defect is entirely in how lower artifacts (research.md) and
  the implementation interpret "the patch's declared structure."

### plan.md / research.md

- **Affected section**: research.md **D3** ("Markdown parsing — goldmark,
  confined to `internal/core`"), which states: *"...whose remaining body
  (prose, `predicate:: [[Target]]` list/body-form bullets, `## <Predicate>-headed
  blocks`) is walked to populate `Text`/`Edges`/`Links`/`HRefs`..."*
- **Conflict identified**: this decision recognizes predicate-grouped body
  blocks **only** via literal `##` (H2) Markdown headings. This directly
  contradicts CORE §12.2, which this same research.md section is supposed
  to implement: *"Markdown headings are reserved for kind and identity;
  node bodies use bold labels, never headings."* The project's own
  reference copy of the canonical format (`github.com/fogfish/arcnet-spec`'s
  `examples/patches/rescorla-2026-tls13.md`) confirms this: every
  predicate-grouped block in that real, spec-authored example patch uses
  `**Mentions**`, `**Proposes**`, `**Raises**`, `**Cites**` — bold labels,
  never `##` headings. `quickstart.md`'s own patch fixtures (Scenario 1's
  `## Mentions`) independently repeat the same non-canonical convention,
  so this feature's own documentation is internally consistent but
  disagrees with the upstream authority it implements.
- **Impact**: research.md D3, `data-model.md`, `contracts/ast-contract.md`
  (which mirrors D3's "## <Predicate>-headed blocks" language), and every
  patch fixture in `quickstart.md` need revision to recognize a
  **bold-label paragraph immediately followed by a list** as a
  predicate-grouped block — the CORE §12.2-canonical form — in addition to
  or instead of the current H2-heading form.

## Secondary, incidentally-discovered issue (same example, not the main bug)

The original patch's inline mention `[[Article Extension]]s` (a bracketed
wikilink immediately followed by a plain `s` to form a plural — a common,
legitimate wiki-link convention, and present verbatim in the user's real
patch) is rendered back as plain `Article Extensions`, with the bracket
markup never reconstructed. This traces to `internal/core.followedByBoundary`
(research.md D3b's "ends... immediately followed by whitespace/punctuation"
rule): a match immediately followed by a bare letter (here, the plural
`s`) is correctly treated as *not eligible* per that rule's literal text,
so `reconstructHRefs` leaves it unlinked by design, not by crash — but
this makes the rule too strict for an extremely common real-world pattern
(pluralizing a wikilink target immediately after its closing brackets).
The `Link` itself is not lost from `HRefs` — only the visual bracket
markup on re-render is — so this is lower severity than the edge-loss bug
above, but likely worth relaxing (e.g. permit a trailing lowercase letter
run with no intervening space, mirroring Obsidian's own "unlinked suffix"
convention) in the same pass if the user wants it addressed. Flagged here
since it is visible in the exact reproduction; tracked separately from the
primary edge-loss bug.

## Root Cause Analysis

Two distinct defects compound into the single observed symptom:

1. **Spec conflict (primary, structural cause)**: research.md D3 designed
   `ParsePatch`'s body walk around H2 (`##`) headings for predicate-grouped
   blocks, directly contradicting CORE §12.2's explicit "node bodies use
   bold labels, never headings" rule — likely because this feature's own
   `quickstart.md` fixtures were themselves written with the same
   non-canonical `## Mentions` convention (an error introduced during
   `/speckit-plan`/`/speckit-tasks` that was never checked against the
   actual `github.com/fogfish/arcnet-spec` reference examples). A bold-label
   block is therefore never recognized as a link block at all.
2. **Implementation bug (independent, compounding cause)**: even setting
   aside (1), `walkNodeBody`'s trailing-content loop silently discards any
   non-`Paragraph` sibling instead of processing or erroring on it. This
   is what actually deletes the two edges — a bold-label paragraph gets
   swept into `Notes` as inert text, but its immediately-following `List`
   is dropped on the floor with no trace, not even a warning.

Neither defect was caught by T031's unit tests or T016's E2E tests because
none of them exercised a node body with more than one predicate-grouped
block, and none used the CORE-canonical bold-label form at all.
