# Bug Report: `arc apply --verbose` reports no per-step or per-node detail

**Type**: Implementation drift
**Severity**: Medium
**Feature**: 003-apply-patch (`arc apply`)
**Reported**: 2026-07-03
**Status**: Resolved
**Patched**: 2026-07-03
**Resolved**: 2026-07-03 — see tasks.md T079-T083

## Description

`arc apply --verbose` currently produces the exact same stderr output as
`arc apply` without `--verbose` (nothing — `cmd/arc/graph/apply.go`
constructs its git adapter with an unconditionally silent `bios.Reporter`:
`git.New(bios.NewReporter(true, true))`, and `internal/app/graph/service.Apply`
does not accept or use a `bios.Reporter` at all). This contradicts both the
feature's own design docs and `arc init`'s already-established `--verbose`
convention (BUG-001 in `specs/002-arc-init/`).

The user's specific ask: under `--verbose`, `arc apply` should report **each
node's title and its status** (created / merged / conflict) as it processes
the patch — not just coarse phase labels.

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 1 — Ingest a brand-new document into
  the graph (P1), whose Acceptance Scenario 4 is "the tool reports what was
  created... so the user can confirm the outcome without a separate
  inspection step" — and, by extension, US2/US3/US4, since they all share
  `service.Apply`'s one pipeline.
- **Affected requirements**: FR-016 ("On success, the tool MUST report to
  the user what happened — counts of nodes created versus merged, by kind,
  and a reference to the resulting commit"). FR-016 is satisfied for the
  final summary line, but spec.md itself never mandates a per-node
  `--verbose` progress trace — that expectation lives entirely in the
  design layer (plan.md/research.md/contracts), not spec.md.
- **Gap identified**: spec.md has no functional requirement at all for
  `--verbose` progress reporting (it's silent on this by design — verbose
  diagnostics are a cross-cutting ADR 002 UX concern, not a per-feature
  FR). The gap is real but lives in the design artifacts below, not here;
  no spec.md edit is needed to fix this bug, only a plan/research
  clarification of the per-node granularity the user now wants.

### plan.md

  `cmd/arc/ctrl/init.go`'s existing entry style.

### research.md

- **Affected section**: D9 ("Command grammar, UX (ADR 002 compliance)"),
  which specifies: *"Reporter steps (DS-06, flat mode...): `"Reading patch
  file"`, `"Checking idempotency"`, `"Applying node contributions"`,
  `"Updating timeline"`, `"Committing"`."*
- **Gap identified**: D9 documents five **phase-level** Reporter steps but
  says nothing about per-node granularity within "Applying node
  contributions" — the exact detail the user is asking for (node title +
  created/merged/conflict status per node). D9 needs a new paragraph
  specifying that "Applying node contributions" additionally emits one
  `Reporter.Step(...)` line per node, naming the node's ID/title and its
  outcome (created / merged / merged-with-conflict).

### data-model.md

- **Affected section**: "Reporter events (`internal/bios.Reporter`, ADR
  002 DS-06)" table, which lists the same five phase labels as research.md
  D9 with "Each label gets one Start/Done (or Error) pair" — it does not
  document the `Reporter.Step(...)` sub-lines this bug asks for.
- **Impact**: The table needs a new row or footnote for the per-node
  `Step` lines nested under "Applying node contributions."

### contracts/cli-contract.md

- **Affected section**: stderr contract — *"Reporter progress (research.md
  D9 labels) shown ONLY under `--verbose`/`-v`, styled faint/gray..."*
- **Gap identified**: This is the one place the contract is unambiguous
  that `--verbose` progress is a committed part of `arc apply`'s CLI
  surface. The current implementation violates this contract outright —
  it shows no Reporter progress under `--verbose` at all, coarse or
  per-node.

## Root Cause Analysis

Implementation-time scope-narrowing, not a spec oversight. While
implementing T062, the pinned function signature listed in tasks.md
(`Apply(ctx, mounter, vcs, rules, dir, patchPath)`) omits a `bios.Reporter`
parameter, even though the task's own prose ("reporting each step via
`bios.Reporter`") and data-model.md's Reporter events table both require
one. Faced with that inconsistency between the literal signature and the
prose/data-model requirement, the implementation followed the literal,
narrower signature and treated per-step progress reporting as out of scope
for this pass — silencing the git adapter's reporter unconditionally in
`cmd/arc/graph/apply.go` instead of threading a real one through. This was
a deliberate but under-documented judgment call, not caught before
`/speckit.implement` reported completion, and it also left the CLI
contract's explicit `--verbose` promise unmet. The user's ask for per-node
(not just per-phase) granularity is a genuine refinement beyond what
research.md D9/data-model.md ever specified, not just a missed
implementation detail — it needs a small design addendum before the code
fix, not just a code fix alone.
