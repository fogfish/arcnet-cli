# Bug Report: `arc apply` silently drops patch body content that isn't wikilink-shaped, violating FR-009's verbatim-preservation guarantee

**Type**: Implementation drift (root cause: a capability explicitly deferred across three specs, never closed)
**Severity**: Critical
**Feature**: 010-predicate-node-model (violated requirements); 011-machine-readable-schema and 013-predicate-role-rendering (each explicitly deferred the closing work to the other, and neither did it)
**Reported**: 2026-07-20
**Status**: Patched
**Patched**: 2026-07-20

## Description

`arc apply` silently drops patch body content that doesn't structurally look
like a wikilink, instead of consulting the schema to know how to handle it тАФ
causing real, silent data loss on ingest, with no error and no warning even
under `--verbose`.

**Reproduction**: applying `../bots/dmitry-2026-article.md` to a graph via
`arc apply -v ../bots/dmitry-2026-article.md` produces `Hypothesis` nodes
missing entire sections present in the source patch. The patch's
`## Ontology-Driven Multi-Purpose Knowledge Representation` section has a
`**Assumptions**` block (3 bullets of plain prose, no wikilinks) and a
`**References**` block (3 standard `[Title](url) - description` markdown
links, not `[[wikilinks]]`) тАФ both vanish entirely from the resulting node
file. Meanwhile `**Related Aporias**` (a `[[wikilink]]` bullet) and
`**Assumes**`/`**Derived From**` (`predicate:: [[wikilink]]` bullets) in the
same sections survive correctly. This happens even though
`_schema/predicates/assumptions.md` is explicitly registered in the target
graph with `role: text, merge: append` тАФ the parser never consults it, so
the registration has zero effect.

## Artifact Traceability

### spec.md (010-predicate-node-model)

- **Affected user story**: User Story 1 тАФ "Author a node with open,
  predicate-keyed identity and content."
- **Affected requirements тАФ currently violated, not merely underspecified**:
  - **FR-005**: "Arc MUST support an open, unbounded set of named prose
    fields per node ... rather than a fixed pair of slots." The parser only
    ever populates two prose slots (leading/trailing); a third named field
    like `assumptions` has nowhere to go and is dropped.
  - **FR-006**: "Arc MUST preserve each prose field's content verbatim under
    its own name across a read/write cycle." `Assumptions`/`References`
    content is not preserved at all тАФ the opposite of verbatim.
  - **FR-009**: "Arc MUST preserve unrecognized attribute keys and
    unrecognized predicates verbatim (value and position), whether or not
    the current graph's schema has previously seen them." This is the
    sharpest violation: FR-009 makes no exception for body-list content that
    doesn't parse as a wikilink тАФ it unconditionally requires verbatim
    preservation of *unrecognized* predicates, and the current
    implementation instead unconditionally *drops* them.
- **Gap identified**: this isn't a case of the spec never having addressed
  the scenario тАФ spec 010's own plan.md and tasks.md explicitly *named* this
  exact gap and *deferred* it, rather than leaving it undiscovered:
  - plan.md's Complexity Tracking table: "`Texts` supports open predicate
    keys as a *type*, but this increment's parser only ever populates two
    (leading/trailing, per-`@type` labelled) тАФ spec FR-005/US1 Acceptance
    Scenario 2's 'several independently named prose sections' is not fully
    realized ... Recognizing an arbitrary `"## <PredicateName>"` heading
    immediately followed by prose (not a list) as a third+ named text block
    requires deciding how that heading is disambiguated from a link-block
    heading without schema role knowledge тАФ **exactly the Schema Index
    question spec 011 owns** ... deferred, not dropped."
  - tasks.md's own T023 GoDoc instruction: `textPredicateFor` must be
    flagged "as a temporary stopgap superseded by spec 011's Schema Index."
  - tasks.md's closing note: "do not expand `walkNodeBody` to recognize
    arbitrary named prose headings while implementing T024; that is
    explicitly deferred, not an oversight to fix inline."

  Spec 011 (`machine-readable-schema`) then built the Schema Index
  (`core.Index`, predicate `role`/`merge` vocabulary) that spec 010 said
  would own this тАФ but spec 011's own scope never routed that Index into
  parsing; it only used it for auto-registration defaults and schema-index
  loading. Spec 013 (`predicate-role-rendering`) then built schema-driven
  `role`-based dispatch тАФ but explicitly, a second time, excluded parsing
  from its own scope: **FR-011**: "This change MUST NOT alter how a node is
  parsed from Markdown into its in-memory Edges representation тАФ the
  existing unified parse-time behavior (spec 010) is unaffected; only the
  write/render path changes." No fourth spec ever closed this loop. Three
  specs each correctly identified that "the schema should drive parsing
  eventually" and each handed it to the next one; none of them did it.

### plan.md (010-predicate-node-model)

- **Affected section**: Complexity Tracking table (quoted above) and the
  Constraints line noting "spec FR-005/Acceptance Scenario 2's 'several
  independently named prose sections per node' is only partially satisfied
  by this increment."
- **Impact**: the plan's own justification for deferring this ("building an
  interim heuristic here risks producing behavior spec 011 then has to
  un-teach") is now moot тАФ spec 011's real schema-role machinery has existed
  for multiple features; the deferred work can finally be built on top of it
  rather than as a throwaway heuristic.

### tasks.md (010-predicate-node-model)

- **Affected tasks**: **T023** (`textPredicateFor`, explicitly documented as
  a temporary stopgap) and **T024** (`walkNodeBody`, explicitly instructed
  *not* to recognize arbitrary named prose blocks, "that is explicitly
  deferred, not an oversight to fix inline").
- **False completions**: none тАФ T023/T024 correctly implemented exactly what
  they were scoped to implement, with the limitation explicitly flagged
  rather than hidden. Nothing needs reopening as incorrectly-marked-done;
  the gap was honestly disclosed and simply never picked up by a later
  feature.
- **Missing tasks**: a task to thread `core.Index` into `ParsePatch`/
  `ParseNode`/`parsePatchBody`/`walkNodeBody`, mirroring the signature shape
  `RenderNode(n Node, index Index)`/`RenderPatch(p Patch, index Index)`
  already established (spec 013); a task to resolve a `**Label**` block's
  predicate identity via the inverse of `labelFor` (predicate тЖТ label,
  already used at render time) against `index.Predicates`; a task to
  dispatch block processing by the resolved predicate's `Role` instead of
  by the block's own structural shape; a task to extend the existing
  `RegisterPredicate` auto-discovery path (`internal/app/schema/service`) to
  default to `role: text, merge: append` when the observed content isn't
  wikilink-shaped, instead of always defaulting to `role: edge, merge:
  union` as it does today.

## Root Cause Analysis

`internal/core`'s patch-body parser (`walkNodeBody`/`blockTitle`/
`collectListLinks` in `internal/core/markdown.go`) is entirely schema-blind
тАФ `ParsePatch`/`ParseNode` take no `core.Index` parameter at all. A
`**Label**` immediately followed by a list is unconditionally treated as an
edge-producing block; `collectListLinks` runs every bullet through a
wikilink-shape regex (`parseListItemLink`) and silently skips (drops, no
error) any line that doesn't match.

This is not an isolated oversight тАФ it is the direct, foreseeable
consequence of a capability three consecutive specs each explicitly named
and each deferred to the next, without any of them ever closing the loop:
spec 010 named it and pointed at spec 011's Schema Index; spec 011 built the
Schema Index but never wired it into parsing; spec 013 built schema-role
dispatch but explicitly scoped it to rendering only (FR-011), again pointing
away from itself. The result is a real, silent-data-loss bug against
requirements (FR-005/FR-006/FR-009) that have been live and unconditional
since spec 010 shipped тАФ this graph's own `_schema/predicates/assumptions.md`
registration (`role: text, merge: append`) proves a real user already tried
to use the capability spec 010 promised and spec 011/013 never delivered.

There is also a real architectural asymmetry underlying why this was easy to
defer repeatedly: `RenderNode(n Node, index Index)`/`RenderPatch(p Patch,
index Index)` already accept `core.Index` and use it (`resolveRenderRole`/
`labelFor`) to drive schema-based layout decisions. `core.Index` is a
domain-native type (`internal/core/rules.go`), so threading it into the
parser crosses no new layering boundary ADR 001 wouldn't already permit for
rendering тАФ the fix is completing a symmetry that already half-exists, not
introducing a new one.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to update spec artifacts тАФ primarily spec 010
   (strike/amend the Complexity Tracking entry and Constraints line that
   deferred this, since the blocking condition they cited тАФ "without schema
   role knowledge" тАФ no longer holds), cross-referencing spec 011 (extend
   the auto-discovery mechanism) and spec 013 (revisit FR-011's "parsing is
   unaffected" scope boundary, since this fix necessarily changes parsing).
2. Run `/speckit.bugfix.verify` to confirm consistency across specs
   010/011/013.
3. Resume `/speckit.implement` to:
   - Thread `core.Index` into `ParsePatch`/`ParseNode`/`parsePatchBody`/
     `walkNodeBody`, mirroring `RenderNode`/`RenderPatch`'s existing
     signature shape. Callers (`internal/app/graph/service.Apply`) already
     have `index core.Index` in scope before calling `readPatch`/
     `core.ParsePatch` тАФ this is a mechanical threading change, not a new
     resolution step.
   - For a `**Label**`-prefixed block (list or paragraph), resolve the label
     to a predicate id via the inverse of `labelFor` тАФ scan
     `index.Predicates`, matching each entry's resolved display label
     against the block's literal label text.
   - If resolved, dispatch by the predicate's declared `Role`: `edge`/`link`
     тЖТ parse bullets as edges (unchanged target-extraction, but role no
     longer inferred from shape); `text` тЖТ aggregate the block's full
     content into `Texts[predicateID]`, so it participates in that
     predicate's own registered merge behavior (e.g. `assumptions`' `merge:
     append`) on later re-applies.
   - A bare list with no preceding label keeps today's per-line
     classification (explicit `predicate::` tag or wikilink shape), but a
     non-matching line now falls through to text instead of being dropped.
   - Extend `RegisterPredicate` (`internal/app/schema/service`) to default
     newly-discovered predicates to `role: text, merge: append` when the
     observed content isn't wikilink-shaped, instead of always `role: edge,
     merge: union` тАФ closing spec 011 research.md's own flagged gap: "today's
     auto-discovery path only ever observes edge-position predicates ...
     until a future feature broadens auto-discovery to other positions."
   - In the same pass, delete `textPredicateFor`'s dead `case "hypothesis":`/
     `case "aporia":`/`case "thought":` branches in `internal/core/markdown.go`
     тАФ lowercase, unreachable since spec 019 (CamelCase Node Class Names)
     made every real `@type` CamelCase, silently falling through to
     `default: "text"` already. Do **not** touch or generalize
     `Source`/`Entity`/`Resource`'s own cases (`abstract`/`definition`/
     `relevance`) тАФ `CoreTypeDefs.Required` and `arc lint`'s
     `checkTypeRequires` depend on those exact predicate names for the three
     built-in content types.
   - Add regression tests: a labeled text-role block with plain-prose or
     non-wikilink-list content round-trips through `arc apply` intact; a
     labeled edge-role block is unaffected; an unregistered label with
     non-wikilink content auto-registers as `role: text, merge: append` and
     its content survives; a mixed list (some wikilink lines, some not)
     preserves both the extracted edges and the remaining text.
