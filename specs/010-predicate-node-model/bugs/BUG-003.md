# Bug Report: BUG-002's data-loss fix preserves content but destroys its formatting — wikilinks, list shape, block labels, and per-block edge grouping are all lost on round-trip

**Type**: Implementation drift (the BUG-002 fix satisfies FR-019's letter — no content is dropped — while violating FR-006/FR-015's already-live verbatim/byte-stable round-trip guarantees, and spec 013's already-live label/grouping rendering guarantees)
**Severity**: Critical
**Feature**: 010-predicate-node-model (violates FR-006, FR-009, FR-015, and its own newly-added FR-019); 013-predicate-role-rendering (violates FR-001/FR-004/FR-014's label and grouping guarantees)
**Reported**: 2026-07-21
**Status**: Patched
**Patched**: 2026-07-21

## Description

BUG-002's fix (spec 010 Phase 6, `internal/core/markdown.go`'s schema-aware `walkNodeBody`) stops `arc apply` from *dropping* a `**Label**`-headed body block's content — but the content that survives comes out reformatted, not preserved. A real reported example, applying a Hypothesis node with `**Assumptions**`, `**Related Aporias**`, `**References**`, `**Assumes**`, and `**Derived From**` blocks:

**Original** (as authored in the source patch):
```
**Assumptions**
- Core graph structure can be meaningfully separated from domain-specific semantics
- Lightweight ontologies reduce friction compared to heavyweight formal models
- [[LLM]]s can be effectively trained on regulated node structures to maintain semantic consistency across extensions

**Related Aporias**
- [[Domain Overspecialization Limits Generalization]]

**References**
- [Smart Applications REFerence Ontology (SAREF)](https://saref.etsi.org) - SAREF Portal — ETSI, 2015
...

**Assumes**
- assumes:: [[LLM]]

**Derived From**
- derivedFrom:: [[dmitry-2026-article]]
```

**Written back** (after `arc apply`, then re-rendered):
```
Core graph structure can be meaningfully separated from domain-specific semantics
Lightweight ontologies reduce friction compared to heavyweight formal models
LLMs can be effectively trained on regulated node structures to maintain semantic consistency across extensions

[Smart Applications REFerence Ontology (SAREF)](https://saref.etsi.org) - SAREF Portal — ETSI, 2015
...

- [[Domain Overspecialization Limits Generalization]]
- assumes:: [[LLM]]
- derivedFrom:: [[dmitry-2026-article]]
```

Four distinct, independently-rooted defects, each severe on its own:

1. **Wikilink markup is stripped from text-role content.** `[[LLM]]s` (a wikilink immediately followed by an inflectional suffix, no whitespace) loses its `[[`/`]]` entirely and becomes plain `LLMs` — the link is gone, not just re-rendered differently.
2. **List shape is destroyed.** The three `- ` bulleted `Assumptions` items and three bulleted `References` items are flattened into bare newline-separated lines with no list markers at all — this is not the "byte copy" FR-006/FR-015 already require.
3. **`**Label**` headers vanish entirely.** Neither `**Assumptions**` nor `**References**` (nor any other label whose content became a `Texts` entry) reappears anywhere in the output — the reader has no way to tell what the surviving prose/list even *is* anymore.
4. **Per-block edge grouping collapses into one undifferentiated list.** The original input had three separate blocks — `**Assumes**`, `**Derived From**`, `**Related Aporias**` — each its own predicate with its own label. The output merges all three into one flat, unlabeled bullet list with no indication these were ever distinct groups.

## Artifact Traceability

### spec.md (010-predicate-node-model)

- **Affected user story**: User Story 1 — "Author a node with open, predicate-keyed identity and content," Acceptance Scenario 5: "a node produced by arc (round-tripped at least once) ... is read and re-written again, ... the result is byte-for-byte stable."
- **Affected requirements — currently violated**:
  - **FR-006**: "Arc MUST preserve each prose field's content verbatim under its own name across a read/write cycle." The Assumptions/References content is not verbatim — bullet markers and wikilink brackets are stripped.
  - **FR-009**: "Arc MUST preserve unrecognized attribute keys and unrecognized predicates verbatim (value and position)." Same violation, for predicates auto-discovered via BUG-002's own new path.
  - **FR-015**: "A node written by arc, when read back a second time and written again, MUST produce byte-for-byte identical output to the first write." Not testable as "stable" in any meaningful sense when the *first* write already diverges this far from the input's intent — and the first write is plainly not equivalent to the original content by any reasonable reading of "verbatim."
  - **FR-019** (added by BUG-002, 2026-07-20): correctly requires the resolved predicate's role to decide *how* content is captured, but says nothing about *preserving the block's own label for later rendering* or *preserving list-item source formatting* — the requirement's letter ("captured verbatim as prose") is satisfied at the `Texts` map level (the string is non-empty and contains the words) but not at the byte/markup level the rest of spec 010 already promises elsewhere.
- **Gap identified**: FR-019 closed the *data-loss* half of the deferred capability (spec 010's plan.md Complexity Tracking table, spec 011, spec 013 FR-011 each named and deferred it), but never addressed two things the original three-way deferral also implicitly required: (a) a `**Label**`-derived predicate must be able to render its own heading back on write, the same way a schema-declared `link`-role predicate already can; (b) list-item content captured into `Texts` must preserve its list-shaped, literal-markup nature rather than being pushed through the paragraph-prose inline-link-extraction pipeline that was designed for free-flowing sentences, not discrete bulleted items.

### plan.md (010-predicate-node-model)

- **Affected section**: Constraints line — "round-trip and idempotent-round-trip fidelity MUST hold except for the explicitly-permitted cosmetic edges-grouping normalization (spec FR-014/FR-015)." The grouping collapse described here (Issue 4) is not the *permitted* cosmetic normalization spec 013 describes (reordering groups, correcting an already-known predicate's flat-vs-grouped shape to match its declared role) — it is a *loss of distinctness* between three different predicates that had three different labels, caused by auto-registration defaulting all of them to the same flat `role: edge` regardless of which one had its own block.
- **Impact**: the Phase 6/BUG-002 closure note ("Closed (Bugfix BUG-002, 2026-07-20)") is premature — it closed the data-loss half of the deferred gap but reopens a *new*, narrower round-trip-fidelity gap that plan.md's own constraint already disallows.

### tasks.md (010-predicate-node-model)

- **Affected tasks**: **T085** (role-driven dispatch: correctly routes text-role content into `Texts[predicateID]`, but via a pipeline — `listItemLines` → `strings.Join(..., "\n")` → `extractInlineLinks`/`reconstructHRefs` — built for prose paragraphs, not list items, causing Issues 1 and 2) and **T087** (auto-discovery role defaulting: correctly distinguishes `text` vs `edge` observed content, but has no way to register a `label` attribute or to distinguish "edge content that came from its own `**Label**` block" — which should default to `role: link` to preserve grouping — from "edge content in the bare/ungrouped list" — which correctly stays `role: edge`, causing Issues 3 and 4).
- **False completions**: T085/T087/T089/T090/T091 were marked `[X]` based on tests and manual verification that checked *content survives* (per FR-019's letter) but did not check *formatting/shape survives* (per FR-006/FR-015's pre-existing, still-live letter) — this was a real gap in the verification coverage, not a spec misread.
- **Missing tasks**:
  1. A task to preserve list-item source formatting for text-role content: capture each list item's own raw markdown (including its wikilink brackets and, ideally, its list-marker shape) rather than routing bulleted items through the free-prose `extractInlineLinks`/`reconstructHRefs` heuristic pipeline.
  2. A task to fix `reconstructHRefs`'s `followedByBoundary` check (`internal/core/markdown.go`) so a wikilink immediately followed by an inflectional suffix (`[[LLM]]s`) is not silently dropped during reinsertion — or, given (1) above may make this path unreachable for list-shaped content, at minimum confirm it no longer fires for the fixed cases and document the residual limitation for paragraph prose.
  3. A task to have auto-registration (`RegisterPredicate`, `internal/app/schema/service`) populate a `label` attribute from the original block's literal label text (e.g. `"Related Aporias"`, not a `titleCaseType`-mangled `camelizeLabel` id) — required for spec 013 FR-004's label resolution to ever recover the original heading text for an auto-discovered predicate.
  4. A task to render a heading/bold-label for a `role: text` predicate's `Texts` entry on write, mirroring `renderEdges`'s existing `role: link` heading logic (`renderNodeBody`'s "other Texts keys" loop currently always renders a bare, unlabeled paragraph — this was explicitly scoped out of BUG-002 as "Gap 3," and this bug report is the escalation that gap's own exclusion note anticipated).
  5. A task to give `RegisterPredicate`/`distinctPredicates` a third observed-role signal — "edge content observed under its own `**Label**` block" → `role: link` (grouped, labeled) — distinct from "edge content observed in the bare/ungrouped list" → `role: edge` (flat), so `**Assumes**`/`**Derived From**`/`**Related Aporias**` each auto-register as their own grouped, headed block instead of collapsing into one undifferentiated flat list.

### spec.md (013-predicate-role-rendering)

- **Affected requirements**:
  - **FR-001/FR-004**: a `link`-role predicate's grouped rendering is driven by "that predicate's declared display label (falling back to the predicate's own name, capitalized, when no label is declared)" — this machinery already exists and works correctly for schema-declared predicates; BUG-002's auto-discovery path simply never populates the `label` attribute that would let it produce the right heading text for a predicate it just discovered.
  - **FR-014**: patch-format rendering of a `link`-role predicate group uses a bold-label paragraph — the identical mechanism issue 3/4's fix needs, already built and already correct for the schema-declared case.
- **Gap identified**: none — spec 013's rendering machinery is not broken; BUG-002's auto-registration simply never feeds it the two inputs (a populated `label`, and a `role: link` instead of `role: edge`) it needs to produce the original shape back.

## Root Cause Analysis

Each of the four symptoms traces to a distinct, narrow root cause, all four introduced by BUG-002's implementation (Phase 6, `internal/core/markdown.go` + `internal/app/schema/service/schema.go`):

1. **Wikilink loss** (`internal/core/markdown.go`, `reconstructHRefs`/`followedByBoundary`, ~line 1450): reinsertion of a stripped wikilink's brackets requires the display text to be followed by whitespace or punctuation; `[[LLM]]s` fails this check (`s` is neither), so the bracket markup is silently never reinserted. This heuristic pre-dates BUG-002 but was previously only exercised on free-flowing paragraph prose (where a mid-sentence wikilink followed by an inflectional suffix is rare); BUG-002's `walkNodeBody` now also routes **list-item** content (`listItemLines` → `strings.Join` → `extractInlineLinks`) through this same pipeline, where every bullet is a short, self-contained item — a pattern much more likely to end a wikilink with an inflected suffix, exposing a pre-existing but previously low-impact heuristic gap.

2. **List shape loss** (`internal/core/markdown.go`, `listItemLines`, line 761): each item is captured via `listItemText`, which already flattens the item's own inline block structure into a single space-joined string (existing helper, unchanged by BUG-002) with no list marker retained; `walkNodeBody`'s new text-role branch then joins these already-flattened strings with `"\n"`. The original bulleted-list *shape* is gone by the time it reaches `Texts`, not merely re-rendered differently.

3. **Heading loss** (two independent causes): (a) `RegisterPredicate` (`internal/app/schema/service/schema.go:407`) never sets a `label` attribute from the block's original literal text — an auto-discovered predicate like `relatedAporias` (from `camelizeLabel("Related Aporias")`) has no way to recover its original `"Related Aporias"` spacing/casing via `labelFor`'s `titleCaseType` fallback, which only capitalizes the first character; (b) independently, `renderNodeBody`'s "other Texts keys" loop (`internal/core/markdown.go:972-982`) unconditionally renders every non-leading/trailing `Texts` key as a bare paragraph with **no heading at all**, regardless of any `label` the predicate might have — this is "Gap 3," a scoping decision made explicitly and silently during BUG-002's implementation (documented only in that session's own internal working notes, never surfaced to the user or filed as a follow-up) rather than escalated.

4. **Grouping collapse** (`internal/app/graph/service/apply.go`, `distinctPredicates`/`predicateObservation`, and `RegisterPredicate`'s `observedRole` parameter): the auto-discovery signal introduced by BUG-002's T087 only distinguishes `"text"` vs `"edge"` observed content — it has no way to know whether an edge occurrence came from underneath its own `**Label**` block (which, per spec 013 FR-001, should register `role: link` to get grouped/headed rendering back) or from the bare, label-less list at the top of a body (which correctly stays `role: edge`, flat). Every auto-discovered edge-shaped predicate defaults to `role: edge` regardless of origin, so `assumes`/`derivedFrom`/the unresolved `relatedAporias` edge all render as one undifferentiated flat list.

**Common thread**: BUG-002 correctly identified and fixed the *data-loss* class of the deferred capability (spec 010 plan.md's Complexity Tracking table, spec 011, spec 013 FR-011), but its own scope and verification (T089's regression tests, T091's manual check) only asserted *content presence* — never *format/shape equivalence* — against spec 010's own pre-existing, still-live FR-006/FR-015 verbatim/byte-stable guarantees, or spec 013's own pre-existing label/grouping rendering guarantees. The fix solved the reported symptom (missing content) while introducing a narrower, but still severe, new violation of requirements that were already in force before BUG-002 ever started.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to update spec artifacts:
   - **spec.md (010)**: amend FR-019 (or add a new FR) to explicitly require that a `**Label**`-resolved `text`-role block's *list-item source formatting* (literal wikilink brackets, list-item boundaries) is preserved, not merely its word content; add an Edge Case entry for the `[[Target]]<suffix>` reinsertion-boundary limitation.
   - **spec.md (013)**: no FR text changes needed (the rendering machinery is correct) — add a note cross-referencing that auto-registered predicates must populate `label` and choose `role: link` vs `role: edge` correctly for FR-001/FR-004/FR-014 to apply to them the same way they already apply to schema-declared predicates.
   - **tasks.md (010)**: reopen T085/T087 with correction notes (not false completions — see above); add new tasks per the five items listed under tasks.md's "Missing tasks" above.
2. Run `/speckit.bugfix.verify` to confirm consistency across specs 010/013.
3. Resume `/speckit.implement` to:
   - Preserve text-role list-item content's own source formatting (raw wikilink brackets at minimum; ideally the list-item boundary itself) instead of running it through the free-prose `extractInlineLinks`/`reconstructHRefs` round-trip.
   - Fix or route around `reconstructHRefs`'s `followedByBoundary` gap for the `[[Target]]<suffix>` case.
   - Populate `label` on auto-registered predicates from the original block's literal text.
   - Render a heading/bold-label for `role: text` predicates on write (closing "Gap 3"), mirroring `renderEdges`'s existing `role: link` heading logic.
   - Add a third observed-role signal (edge-under-its-own-label → `role: link`; edge-in-bare-list → `role: edge`) to `distinctPredicates`/`RegisterPredicate`'s auto-discovery path.
   - Add regression tests that specifically assert *shape* equivalence (list markers present, wikilink brackets present, heading/label present, groups stay separate) — not just that the words survive — closing the verification gap that let this regression through T089/T090/T091 the first time.
