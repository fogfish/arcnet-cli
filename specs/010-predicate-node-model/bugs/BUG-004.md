# Bug Report: `walkNodeBody`'s "one bare list, then title+list pairs" grammar corrupts every block after a second, untitled list — pure/plain lists lose their shape and later `**Label**` blocks get split from their content

**Type**: Spec gap (the AST §6 grammar `walkNodeBody` implements only ever anticipated *one* bare list before the titled blocks begin; research.md 003's D3c revision explicitly claimed the trailing loop was "provably exhaustive" given that grammar — a real patch body with two consecutive plain lists falsifies that claim)
**Severity**: Critical
**Feature**: 010-predicate-node-model (violates FR-006/FR-014/FR-015's verbatim/round-trip guarantees and FR-019's role-driven dispatch); 003-apply-patch (research.md D3c's "provably exhaustive" trailing-loop claim, now falsified)
**Reported**: 2026-07-21
**Status**: Patched
**Patched**: 2026-07-21

## Description

`arc apply` mishandles a patch node body that contains a plain (non-wikilink) bulleted list — either as the "bare list" slot right after the leading paragraph, or anywhere a list is not immediately preceded by a `**Label**`/`## Label` title. Two independent defects compound:

1. **A pure list loses its list shape.** The bare `- xxx` / `- yyy` / `- zzz` list right after the leading paragraph is folded into the leading-prose slot and rendered back as three separate paragraphs, not a list.
2. **A second, untitled list permanently derails all later `**Label**` blocks.** Once `walkNodeBody`'s headed/labeled-blocks loop hits a `List` where it expected a title (`Paragraph`/`Heading`), it exits for good — it never resumes looking for further title+list pairs. Every subsequent well-formed `**Label**` + list pair in the same body (here: `**Mentions**`, `**Cites**`, `**Thoughts**`, each followed by its own list) falls into the flat trailing loop instead, which treats paragraphs and lists independently rather than as pairs. The result: the literal text `**Mentions**` / `**Cites**` / `**Thoughts**` survives as orphaned trailing prose (with no list under it), while each block's *list* is separately captured as edges and re-rendered as its own correctly-headed `## Mentions` / `## Cites` section — so the label and its content appear **twice**, disconnected from each other, in the output.

**Input** (patch node body, abridged — see full text in the user-supplied `<input-node>`):
```
The session examined how to represent durable schema metadata ...

- xxx
- yyy
- zzz

* A graph requires schema metadata independent of its content.
* Node types may define merge algorithms as schema metadata rather than per-node content.
... (6 more plain sentences)

**Mentions**

* mentions:: [[Knowledge Graph Schema]]
...

**Cites**

* cites:: [[Obsidian]]

**Thoughts**

* generatedThought:: [[Separate Schema From Content]]
...
```

**Written back** (after `arc apply`, per the user-supplied `<invalid-output>`):
- `xxx` / `yyy` / `zzz` render as three bare paragraphs, no list markers.
- The eight plain sentences (`A graph requires schema metadata ...` etc.) also render as bare paragraphs, no list markers, and are physically relocated to the *end* of the body — after the correctly-rendered `## Cites` / `## Mentions` sections rather than before them.
- `## Mentions`, `## Cites`, `## Thoughts` each appear correctly headed with their real edges (a rendering that already accounts for the *edges* that were extracted) — but the literal, disconnected label text `**Mentions**`, `**Cites**`, `**Thoughts**` *also* reappears, empty, at the very end of the body.

## Artifact Traceability

### spec.md (010-predicate-node-model)

- **Affected user story**: User Story 1, Acceptance Scenario 5 (round-trip fidelity); the Edge Cases entry added by BUG-002 ("A patch or node body has a `**Label**`-headed block ... the block's content must still be captured verbatim as prose ... never silently dropped").
- **Affected requirements — currently violated**:
  - **FR-006**: prose/list content must be preserved verbatim under its own name — a plain list's item markers are stripped and its items are reordered relative to labeled blocks that physically followed it in the source.
  - **FR-014/FR-015**: round-trip and idempotent-round-trip fidelity — the output is not even a *cosmetic* reordering of the input; content is duplicated (`**Mentions**` appears both as an orphaned label and as a correctly-rendered `## Mentions` heading) and physically relocated across block boundaries.
  - **FR-019**: "a `**Label**`-prefixed body block ... MUST have its predicate identity resolved against the graph's schema index." `**Mentions**`/`**Cites**`/`**Thoughts**` *do* resolve correctly when the headed-blocks loop reaches them — the bug is that the loop never reaches them at all once a second untitled list has already made it exit early, so their titles fall through to the paragraph-only trailing loop where no title+list resolution is even attempted.
- **Gap identified**: FR-019 (and the AST §6 grammar it's built on, `leading prose → one optional bare list → title+list blocks → trailing prose`) only ever anticipated **one** bare/untitled list appearing before the titled blocks begin. Real Markdown content — including the user's own worked example — routinely contains multiple consecutive plain lists (a short list, then a longer list of full-sentence bullets) with no title paragraph between or before them. The grammar has no slot for "more than one untitled list," and the current implementation's fallback (the trailing loop) is not actually a fallback for *this* case — it was designed and previously verified only for "a list left over *after* all titled blocks have already been matched" (research.md 003 D3c), not for "an untitled list encountered *before* titled blocks have all been matched," which is exactly what starves the headed-blocks loop of the chance to ever run past that point.

### research.md (003-apply-patch)

- **Affected section**: D3c's revised trailing-loop rationale — "remaining `**Label**`+list pairs are consumed by the (now bold-label-aware) headed-blocks loop first, so no unrecognized `List` should ordinarily reach the trailing loop at all ... this is now provably exhaustive." This claim assumed at most one untitled list occupies the "bare list" slot and that every list thereafter is either wikilink-shaped (bare list) or title-paired. A second untitled, non-wikilink-shaped list breaks that assumption: it reaches the headed-blocks loop *before* `**Mentions**`/`**Cites**`/`**Thoughts**` do, causes it to `break` immediately (its first check is `blockTitle(children[idx])`, and a bare `*ast.List` never satisfies `blockTitle`), and the loop never gets a second chance to resume once the trailing loop's flat, non-pairing scan takes over.
- **Impact**: the "provably exhaustive" claim needs to be revised or the grammar needs to change; the trailing loop's paragraph/list independence (case-by-case, not pair-by-pair) is the direct cause of `**Mentions**` splitting from its own list.

## Root Cause Analysis

Two distinct root causes, both in `internal/core/markdown.go`'s `walkNodeBody` (lines 590-721):

1. **List-shape loss for plain (non-wikilink) list content** (lines 615-622 and 674-678): `classifyListItems` correctly separates wikilink-shaped lines (→ edges) from plain lines (→ `textLines`), but both the bare-list slot and the trailing loop then do `leading = append(leading, textLines...)` / `trailing = append(trailing, textLines...)` and later `strings.Join(leading, "\n\n")` / `strings.Join(trailing, "\n\n")` — each plain list item becomes its own paragraph, with no `-` marker, indistinguishable from the node's genuine prose. `renderTextListLines` (lines 728-740) already exists to reconstruct a bulleted list from raw lines exactly like these, but only the named-block (`**Label**`-resolved) text-role path (line 703) calls it; the bare-list and trailing-list paths never do.

2. **The headed/labeled-blocks loop cannot resume after an untitled list** (lines 625-661, specifically the `if !matched || idx+1 >= len(children) { break }` at lines 627-629): the loop's *only* exit condition on a non-title child is a hard `break` — there is no "skip this list, keep scanning for the next title" branch. Once `idx` points at a `*ast.List` that isn't preceded by a matching title (here: the second, untitled list of eight plain sentences), the loop exits permanently at whatever `idx` it was on, and everything from there to the end of `children` — including the later, perfectly well-formed `**Mentions**`/`**Cites**`/`**Thoughts**` + list pairs — falls into the trailing loop (lines 669-680). That loop switches on each child's own type independently (`case *ast.Paragraph` vs `case *ast.List`) rather than attempting to re-pair a title paragraph with the list immediately after it, so `**Mentions**` (a `Paragraph`) is captured as orphaned trailing text while its list (a `List`, the very next child) is separately classified via `classifyListItems` and surfaces as edges with no link back to its own label — hence the label and its content reappearing disconnected, and the correctly-rendered `## Mentions` heading (built purely from the *edges*, at render time, independent of the orphaned label text) appearing to duplicate it.

Both root causes stem from the same design limitation: the grammar `walkNodeBody` implements (`leading prose → at most one bare list → title+list pairs, in one forward pass → trailing prose/lists`) does not model documents where an untitled, non-wikilink list can appear more than once, or anywhere other than immediately after the leading prose. The user's own suggested fix — separate the *parse* (classify each block as prose/list/title, decide pairing) from the *build* (turn classified blocks into the internal AST) — would remove this single-forward-pass constraint entirely, rather than patching the trailing loop with more special cases the way BUG-002/BUG-003 did.
