# Bug Report: `arc apply` rejects canonically-shaped patch documents that derive a node contribution's identity/type from its `## <ID>`/`# <Type>` section headings instead of duplicating `"@id"`/`"@type"` inside the node's own yaml fence

**Type**: Spec gap
**Severity**: High
**Feature**: 010-predicate-node-model
**Reported**: 2026-07-07
**Status**: Resolved
**Patched**: 2026-07-07
**Resolved**: 2026-07-07 — code fix implemented in `internal/core/markdown.go` (new `patchNodeIdentity`, replacing `identityFields` at `parsePatchBody`'s call site), covered by T081's unit tests and T082's E2E fixture, and manually verified against the actual `fogfish/bots/email-patch3.md` that originally triggered the report (`arc apply` now exits 0 and creates all 16 nodes).

## Description

`arc apply` fails with:

```
manifest is missing a mandatory field or uses the pre-0.5 node format: missing mandatory "@id" field
```

when applying `/Users/kolesnik/devel/go/src/github.com/fogfish/bots/email-patch3.md`, a real patch document produced by an existing, external patch-generating tool (`fogfish/bots`). The patch's node sections look like this (verbatim excerpt):

```
# Source

## dmitry-2026-graph

```yaml
authors:
    - Dmitry
published: "2026-07-03"
title: Ontologies, Graph Structures, and LLM-Based Knowledge Management
```
...
```

Neither `"@id"` nor `"@type"` appears anywhere in the node's own yaml fence. The `# Source` H1 heading names the node's type, and the `## dmitry-2026-graph` H2 heading names its identity — this is CORE §12.2's original, canonical patch convention (the same one this codebase's own `blockTitle`/`RenderPatch` doc comments cite as "CORE §12.2's canonical convention"), and the same shape every pre-existing patch fixture in this repository used before spec 010's implementation rewrote them.

The current `internal/core.parsePatchBody` (implemented under spec 010) instead requires `"@id"` and `"@type"` to be explicit keys duplicated inside each node's own yaml fence, treating the `# <Type>` / `## <ID>` headings as purely cosmetic/organizational labels with no bearing on identity or type. A patch that follows the pre-existing H1/H2-driven convention — like every patch `fogfish/bots` produces — is therefore rejected outright as if it were a pre-0.5 legacy file, even though it is not: it never used the legacy `kind` field, and its `"@id"`/`"@type"` are fully and unambiguously determinable from the section structure alone.

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 1 — "Author a node with open, predicate-keyed identity and content" (P1), and User Story 2 — "Operate the full toolchain against a predicate-first graph" (P2), Acceptance Scenario 1 ("a patch contributing to an existing node... merges into the node identified by its `@id`").
- **Affected requirements**:
  - FR-011: "A node contribution inside a patch document (the exchange format `arc apply` consumes) MUST declare its identity under the same `"@id"`-equals-basename rule as a standalone node file, with no fallback."
  - Edge Cases: "A patch document (the format `arc apply` consumes) references a node contribution — its identity declaration must follow the same `@id`-equals-basename rule as a standalone node file, with no fallback."
- **Gap identified**: FR-011 and the Edge Cases entry both say a patch node's identity declaration must follow "the same... rule as a standalone node file, with no fallback" — but neither ever states *where* that declaration must live syntactically for a patch. A standalone node file has only one place to put it (the front matter); a patch section has two candidate places (the `## <ID>` heading itself, which is the section's structural identity marker, or a duplicated `"@id"` key inside the per-node yaml fence). The spec never says whether the heading text *is* the declaration (matching CORE §12.2's own convention, where the heading *is* the identity, nothing else needed) or whether it must be *redundantly restated* as a yaml key. "No fallback" was written to rule out the old `id`/`title`/`period` chain for a standalone file's front matter — it was never clearly extended (or explicitly declined to extend) to settle this patch-specific structural question.

### plan.md / research.md / data-model.md

- **Affected sections**:
  - `data-model.md`'s `Node` table, `ID` row: "From `"@id"`. MUST equal the file's basename... for a standalone node file; MUST equal the patch section's heading text for a patch-document node contribution." — this row gives `ID` an explicit patch-heading carve-out (the heading text is what `"@id"` must equal), consistent with the heading itself being the declaration.
  - `data-model.md`'s `Node` table, `Type` row: "From `"@type"`. Open vocabulary..." — this row has **no** equivalent patch-heading carve-out at all, unlike `ID`'s row immediately above it. Read literally, `Type` must always come from an explicit `"@type"` key, even inside a patch, with the `# <Type>` heading treated as non-authoritative — but this asymmetry with `ID`'s own row is never explained or justified anywhere in `data-model.md`, `research.md`, or `plan.md`.
  - `data-model.md`'s `Patch` table, `Nodes` row: "each node contribution's `"@id"` MUST equal its `"## <ID>"` section heading, with no fallback — same rule as a standalone file (spec FR-011)." — "MUST equal... section heading" is consistent with reading the heading itself as authoritative, but does not foreclose requiring a separately-declared `"@id"` key that is then cross-checked against the heading (the interpretation actually implemented).
  - `research.md` D1 (drops the `Kind` named type), D2 (`Predicate` shape), D3 (scalar↔list normalization), D5 (`Edges` unification), D7 (old-format detection) — none of these research decisions discuss patch per-node yaml-fence content vs. section-heading derivation for identity/type at all.
  - `plan.md`'s Project Structure section describes `markdown.go`'s planned changes only in terms of `deriveNodeID`/`ParseNode`/`parsePatchBody`'s `"kind"`→`"@type"` field rename and the removal of the `id`/`title`/`period` fallback chain — it does not describe, one way or the other, whether `parsePatchBody` should keep deriving identity/type from the existing H1/H2 structural walk (as `internal/core/markdown.go`'s pre-spec-010 `parsePatchBody` did: `currentKind := Kind(strings.ToLower(linesText(h1, source)))`, `id := linesText(h2, source)`) or require the same two fields to also be declared inside the per-node yaml fence.
- **Impact**: Because none of these documents settle the question precisely, the implementing pass (spec 010's rewrite of `internal/core/markdown.go`) had to make a judgment call under genuine ambiguity. It chose to require `"@id"`/`"@type"` explicitly inside every patch node's yaml fence (and to source `Type` from the fence rather than the `# <Type>` heading at all), which is defensible against the `Patch`/`Nodes` row's literal text but diverges from `ID`'s own row (which explicitly ties `"@id"` to the heading text for a patch) and from CORE §12.2's own canonical convention this codebase's other comments cite approvingly. `quickstart.md`'s Scenario 1 worked example (which *does* show `"@id"`/`"@type"` duplicated inside the fence) reflects this same implementation-time choice rather than an independently-arrived-at spec decision — it was authored as part of the same Phase 1 design pass and does not resolve the ambiguity, it just documents one resolution of it.

### tasks.md

- **Affected tasks**:
  - T021 — "Read `"@type"` instead of `"kind"` for `Node.Type` in `ParseNode`/`parsePatchBody`, `internal/core/markdown.go`" — this task's own wording ("instead of `"kind"`") only asked for a field-name substitution, not a decision about whether `parsePatchBody` should stop deriving `Type` from the `# <Type>` H1 heading structurally. The mechanical substitution reading was extended, during implementation, into "stop deriving from headings entirely, require an explicit key" — a bigger behavioral change than the task describes.
  - T025 — "Implement old-format detection... a front-matter `kind` key present at all, or `"@id"`/`"@type"` absent or empty, or (for `ParseNode`) `"@id"` ≠ the file's basename, each returns `ErrManifestInvalid`..." — this task is scoped to `"@id"`/`"@type"` being *absent from the manifest entirely*, which is the correct rejection condition for a standalone file (no other place to look) but is being applied to patch documents where the heading structure is a legitimate alternative source, causing a false-positive rejection of a well-formed, non-legacy patch.
  - T029 — "Unit tests in `internal/core/markdown_test.go`: `ParseNode`/`RenderNode`/`RenderPatch` round-trip for every ARCNET-CORE §11 worked example..." — this task's round-trip tests were written and verified only against the implementation's own chosen fence-duplication convention, so they never exercised (and could not have caught) a patch shaped like `fogfish/bots`' output, which is the actually-affected real-world case.
- **False completions**: T021, T025, and T029 are all marked `[X]` complete and their own stated acceptance criteria are technically satisfied by the current code — this is not a case of incomplete task execution, but of the tasks themselves being under-specified relative to the real requirement, per the gap above.
- **Missing tasks**: A task to explicitly decide and document (in `data-model.md`/`contracts/ast-contract.md`) whether `parsePatchBody` derives `"@id"`/`"@type"` from the `## <ID>`/`# <Type>` section headings (optionally cross-validated against an explicit yaml-fence key *if one happens to be present*, but not required), plus corresponding fixture coverage using a patch shaped like CORE §12.2's canonical convention (heading-only, no duplicated identity keys in the fence) — this is the shape every pre-existing patch fixture in this repository, and at least one real external tool, actually produce.

## Root Cause Analysis

This is a genuine spec gap, not a straightforward implementation error: `spec.md`'s FR-011 and Edge Cases entry both state a patch node's identity must follow "the same... rule... with no fallback" as a standalone file, but a standalone file's identity has exactly one possible source (its front matter), while a patch section's identity structurally has two candidate sources (the `"## <ID>"` heading, or a yaml-fence key) — the spec never disambiguates which one is authoritative, or whether both must agree. `data-model.md`'s own `Node.ID` row explicitly ties a patch node's `"@id"` to "the patch section's heading text," which is inconsistent with also *requiring* a separately-declared `"@id"` key — and its `Node.Type` row has no equivalent heading carve-out at all, an asymmetry the spec never explains.

Under this ambiguity, the implementation (spec 010) resolved the question by requiring both `"@id"` and `"@type"` explicitly inside every patch node's yaml fence, matching (and validated against) `quickstart.md`'s own worked example — but that worked example was produced during the same design pass, not derived from an independent, pre-existing requirement, so it does not actually resolve the ambiguity; it just reflects the same judgment call. This diverges from CORE §12.2's original, canonical H1(type)/H2(id)-driven convention, which every patch fixture in this repository used before spec 010 and which at least one real external patch-generating tool (`fogfish/bots`) still produces — causing `arc apply` to reject a well-formed, non-legacy patch as though it were a pre-0.5 file.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to update `spec.md` (FR-011 and its Edge Cases entry), `data-model.md` (reconcile the `Node.ID`/`Node.Type` row asymmetry), and `contracts/ast-contract.md` to explicitly state that a patch node contribution's `"@id"` and `"@type"` are established from its `"## <ID>"`/`"# <Type>"` section headings (mirroring the pre-existing, CORE §12.2 canonical mechanism, just reading `"@type"`'s vocabulary the same open way `"kind"` used to), with an explicit yaml-fence `"@id"`/`"@type"` key treated as optional, redundant, and — if present — cross-checked for agreement with the heading rather than required.
2. Run `/speckit.bugfix.verify` to confirm the patched artifacts are internally consistent (in particular, that `Node.ID`'s and `Node.Type`'s rows in `data-model.md` no longer disagree in shape).
3. Resume `/speckit.implement` to apply the corresponding code fix in `internal/core/markdown.go`'s `parsePatchBody` (derive `Type` from the `"# <Type>"` heading when no explicit `"@type"` key is present in the fence; keep the existing heading-vs-`"@id"` equality check, but stop requiring `"@id"` to be redundantly declared when the heading alone already establishes it unambiguously) and add fixture coverage using a CORE §12.2-canonical, heading-only patch shape (no yaml-fence `"@id"`/`"@type"` keys) in `internal/core/markdown_test.go` and `cmd/arc/graph/apply_test.go`.
