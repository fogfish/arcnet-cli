# Bug Report: `--verbose` reports "appended" for an append/union predicate even when nothing was actually appended

**Type**: Implementation drift
**Severity**: Medium
**Feature**: 012-predicate-merge-policies (branch `012-predicate-merge-policies`; discovered while working from `016-arc-revert`, unrelated to that feature)
**Reported**: 2026-07-12
**Status**: Patched
**Patched**: 2026-07-12

## Description

Reported by the user as a suspected `arc apply` regression:

> It seems there is a regression in `arc append`. Building the graph from `dmitry-2026-graph.md` and then `dmitry-2026-article.md` does not append the text predicate. Specifically `arc apply -v` says `definition: append -> appended`; but the definition is not expanded from definitions of these patches. Either there is a bug in append and then serialization feature or apply has stopped reporting skip due to similarity in the text.

Reproduced end to end with a fresh graph and the two named patch files:

```text
$ arc apply --verbose dmitry-2026-article.md
...
  LLM: merged
    category: firstWriteWin -> unchanged
    definition: append -> appended
    indexed: immutable -> unchanged
    published: immutable -> unchanged
    scoreC: validatedOverwrite -> unchanged
    scoreZ: validatedOverwrite -> unchanged
...
```

Inspecting `entities/LLM.md` before and after confirms the node's `definition` text is **byte-identical** across both patches (both source documents contribute the exact same leading paragraph for the `LLM` entity) — the file does pick up an `updated:` timestamp and a new `mentionedIn:: [[dmitry-2026-article]]` edge, but the `definition` prose itself never changes. `mergeText`'s own near-duplicate detection (`internal/core/merge.go`'s `mergeParagraphs`/`paragraphAlreadyPresent`, Jaccard similarity > 0.8) is working correctly here — the incoming paragraph is a 100%-similarity duplicate of the existing one, so it is correctly *not* appended. The bug is that `--verbose`'s per-predicate report says `appended` regardless, even though `mergeText` added zero new content. The user's own second hypothesis — "apply has stopped reporting skip due to similarity in the text" — is exactly the root cause: the merge/serialization pipeline itself is not broken; the outcome label is wrong.

## Artifact Traceability

### spec.md (specs/012-predicate-merge-policies)

- **Affected requirement**: **FR-017** — "`arc apply --verbose` MUST report, for every predicate present on either side of a merged node..., that predicate's name, the `MergeOp` it resolved to..., and its per-predicate outcome (e.g. **unchanged**/filled/overwritten/**appended**/flagged)..." FR-017 itself already lists `unchanged` and `appended` as distinct, mutually exclusive outcomes — the spec's own vocabulary anticipates a no-op case separate from a genuine append.
- **Gap identified**: FR-017 requires the outcome label to describe what actually happened per predicate, but the implementation (`internal/core/merge.go`'s `mergeTexts`, `internal/core.OutcomeAppended`/`OutcomeCreated`/`OutcomeUnchanged` constants) never checks whether the append/union dispatch branch actually changed the merged value — it unconditionally labels any list-merge (`append`/`union`) predicate that already existed as `OutcomeAppended`, and any that didn't exist as `OutcomeCreated`, with no `OutcomeUnchanged` path ever reachable for a list-merge key. This is not a spec gap so much as an implementation that doesn't fully realize FR-017's own stated intent — classified here as implementation drift.

### plan.md / research.md

- Spec 012's own historical **BUG-001** (`specs/012-predicate-merge-policies/bugs/BUG-001.md`) is the direct predecessor of this issue: it added FR-017's per-predicate `--verbose` reporting and fixed `text`-role predicates to seed `append` by default. That fix is exactly what makes this new bug user-visible — before BUG-001, `definition` seeded `firstWriteWin` (would have shown `overwritten`/`flagged`/`unchanged`, all of which *do* correctly branch on whether the value changed via `mergeScalar`); after BUG-001, `definition` seeds `append`, whose own outcome derivation (`mergeTexts`'s `isListMerge` branch) was never given the same "did it actually change" check `mergeScalar`'s branch already has.

### tasks.md

- **Affected task**: T035 (`internal/core/merge.go`'s `Merge` widened to return `[]PredicateOutcome` — the `isListMerge` branch inside `mergeTexts` is the exact code this bug is in) and T037 (wires `PredicateOutcome` into `--verbose`'s `Reporter.Step` output, which is where the misleading label surfaces to the user) — both `[X]` complete, but neither's acceptance covered "list-merge dispatch when the incoming contribution is a full near-duplicate."
- **Missing task**: a new task to compare `mergeText`'s output against the existing value (mirroring `nodeContentChanged`'s own byte-comparison approach, `internal/app/graph/service/apply.go`) before choosing between `OutcomeAppended`/`OutcomeCreated` and a new `OutcomeUnchanged` result for the list-merge branch, plus regression coverage (a re-contribution whose paragraph is a Jaccard near-duplicate of the existing one must report `unchanged`, not `appended`).

## Root Cause Analysis

`internal/core/merge.go`'s `mergeTexts` (and its `Attrs` counterpart, `mergeAttrs`, for the same `isListMerge` branch) derive a predicate's reported outcome purely from *which merge operation ran*, not from *whether the operation changed anything*:

```go
if isListMerge(op) {
    merged[k] = mergeText(ev, iv)
    outcome := OutcomeAppended
    if !existed {
        outcome = OutcomeCreated
    }
    outcomes = append(outcomes, PredicateOutcome{Name: k, Op: op, Outcome: outcome})
    continue
}
```

Every other dispatch class in the same file (`mergeScalar`, via its own `diverges`/equality checks) already derives its outcome from the *result* of the merge — `OutcomeUnchanged` is reachable there. The list-merge branch is the one place `OutcomeUnchanged` was never wired in, even though `mergeText`'s own near-duplicate detection (BUG-004, `paragraphAlreadyPresent`) can legitimately produce a merged value byte-identical to the existing one. This was very likely an oversight at BUG-001's implementation time: BUG-001's own fix focused on *adding* the per-predicate report and the `append`-by-default seeding, not on auditing every dispatch branch's outcome accuracy against a true no-op.

## Recommended Fix

1. Run `/speckit.bugfix.patch` (against `specs/012-predicate-merge-policies`) to add a requirement clarifying FR-017's "outcome" must reflect the actual merge result even for list-merge (`append`/`union`) predicates, and add a task to implement + test it.
2. Run `/speckit.bugfix.verify` to confirm consistency.
3. Resume `/speckit.implement` to fix `internal/core/merge.go`'s `mergeTexts` (and, for consistency, `mergeAttrs`'s `isListMerge` branch) to compare the merged value against the existing one and report `OutcomeUnchanged` when nothing actually changed, plus add regression coverage for the exact near-duplicate-paragraph case reported here.
