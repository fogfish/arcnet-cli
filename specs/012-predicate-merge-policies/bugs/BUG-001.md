# Bug Report: Verbose apply reporting stayed node-level after per-predicate dispatch; text-role predicates default to firstWriteWin instead of append

**Type**: Spec gap (verbose reporting) / Spec conflict (text-role default)
**Severity**: Medium
**Feature**: 012-predicate-merge-policies (branch `012-predicate-merge-policies`)
**Reported**: 2026-07-08
**Status**: Resolved
**Patched**: 2026-07-08
**Resolved**: 2026-07-08 — implemented via Phase 6 (T035-T042); `go build`/`go test ./...`/`go vet`/`staticcheck` all green; manually re-verified against a built `arc` binary (definition/abstract/notes now append silently, `--verbose` shows per-predicate lines).

## Description

Reported directly by the user after exercising `arc apply -v` against a real graph:

> existing "arc apply -v" in verbose mode reports merge at node level only. However, the new functionality is the predicate-based merge. Therefore, the verbose report must report the status per node and per each predicate in the node together with used algorithms. Secondly, all predicates with "text" role should use "append" merge strategy.

This bundles two distinct findings discovered together while validating the just-shipped per-predicate dispatch:

1. **Reporting granularity gap**: `arc apply --verbose`'s per-node progress line (`Reporter.Step`) reports one outcome per node (e.g. `Widget: merged (conflict flagged)`), same as before this feature. It does not say which predicate(s) were touched, which `MergeOp` each resolved to, or which specific predicate(s) caused a conflict flag. Since spec 012's entire premise is that reconciliation is now per-predicate rather than per-node, a node-level-only report hides the one thing an operator most needs to see when diagnosing a merge outcome (as happened live: the user had to read raw conflict markers in `entities/LLM.md`/`entities/Graph OS.md` and ask why, rather than seeing it in `-v` output).
2. **`text`-role default disagreement**: `internal/app/schema/kernel/schema.go`'s `CorePredicateDefs` seeds `abstract`, `definition`, `notes`, `relevance`, and `description` (all `Role: "text"`) as `MergeFirstWriteWin`, not `MergeAppend` — only the generic `"text"` predicate itself is seeded `MergeAppend`. This is exactly what produced the conflict-marker surprise in the same live session: `definition` (an `entity`'s leading prose) is `role: text` but declared `firstWriteWin`, so a second, genuinely different definition froze-and-flagged instead of appending. The user's stated expectation is that `role: text` alone should be sufficient to predict `append` dispatch.

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 1 ("A node's fields each reconcile by their own rule, not one rule for the whole node") and User Story 2 ("Human-review conflict flagging fires only where the rule calls for it") — both describe *outcomes* but neither commits to any *visibility* contract for how an operator observes which rule fired.
- **Affected requirements**:
  - **FR-016** ("A graph's built-in predicate vocabulary... MUST declare each predicate's genuinely correct behavior... distinguishing a predicate that must never change... from one that legitimately changes over time") — this was written to justify *per-predicate-name* semantic assignment (research.md D2/data-model.md's seed table), not a role-based rule. It does not currently require `role: text` ⇒ `merge: append`; the finding's second half asks for exactly that new constraint, which FR-016 as worded does not cover and the current seed table actively contradicts.
  - No FR currently addresses `arc apply --verbose`'s reporting granularity at all — this is a pure gap, not a violated requirement.
- **Gap identified**: (a) no functional requirement specifies what `--verbose` must surface post-per-predicate-dispatch; (b) no functional requirement ties a predicate's `role` to its permitted/default `merge` value — `role` and `merge` are documented as independent, freely-combined fields (data-model.md "PredicateDef / TypeDef / Index").

## Root Cause Analysis

Both findings trace back to the same cause: spec 012's scope was deliberately bounded to *reconciliation behavior* (research.md's own framing: "wiring already-declared per-predicate rules into reconciliation, not extending the node file format" / D5a) and explicitly marked observability (Principle X) and the predicate-vocabulary's exact per-name assignments (FR-016, decided per research.md D2 against CORE §9.3's own distinctions, not by `role`) as settled, non-negotiable inputs going into implementation. Once the feature actually shipped and was exercised against a real graph, two implicit assumptions turned out to need revisiting:

1. Verbose output's *existing* contract (spec 003 FR-021) was written when merge was whole-node; nobody re-examined whether it still communicated enough once the unit of reconciliation became the predicate. This is a spec gap, not an implementation defect — the code faithfully does what spec 012 asked, spec 012 just didn't ask for richer reporting.
2. The seed table's choice to assign `firstWriteWin` to several `role: text` predicates was a deliberate, documented decision (research.md D2, data-model.md) grounded in CORE's per-predicate semantics (a short human-authored `abstract`/`definition`/`notes` genuinely differs from freely-growing `text`). The user's new ask — role alone should imply the strategy — is a **policy change** request, not a bug in the sense of "code doesn't match spec"; it is a spec conflict that needs a deliberate decision (and likely a spec.md/data-model.md revision) before implementation should change, since it would silently alter merge behavior for five existing predicates across every graph already using this schema.

## Recommended Fix

1. Run `/speckit-bugfix-patch` to update `spec.md` (new FR for verbose per-predicate reporting; revised FR-016 or a new FR establishing the `role: text` → `append` default/override rule) and `data-model.md` (repoint `abstract`/`definition`/`notes`/`relevance`/`description` to `MergeAppend`, or introduce an explicit override mechanism if any of them should stay `firstWriteWin` by exception).
2. Run `/speckit-bugfix-verify` to confirm the patched spec artifacts are internally consistent (FR-016 vs. the seed table vs. TN15's "no node content change is silent" compliance note, since re-seeding these five predicates as `append` **will** change on-disk output for any graph re-running `arc init`/`arc apply` against updated schema documents).
3. Resume `/speckit-implement` to: (a) widen `core.Merge`'s return shape (or introduce a companion type) to carry a per-predicate `{name, op, outcome}` trail, thread it into `internal/app/graph/service/apply.go`'s reporter calls, and add a per-predicate `Reporter.Step` line under `--verbose`; (b) update `internal/app/schema/kernel/schema.go`'s `CorePredicateDefs` per the patched data-model.md; (c) update `internal/core/merge_test.go`, `internal/app/graph/service/apply_test.go`, and `cmd/arc/graph/apply_test.go` fixtures/assertions that currently assert `firstWriteWin` behavior for the affected predicates (notably the `LLM`/`Graph OS`-style `definition`-conflict scenarios exercised live).
