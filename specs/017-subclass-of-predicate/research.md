# Phase 0 Research: Type Inheritance via `rdfs:subClassOf`

All unknowns below were resolved either from the existing codebase (`internal/core`, `internal/app/schema`) or from direct user direction during `/speckit-plan` (including two targeted clarifying questions asked before this document was written). No open `NEEDS CLARIFICATION` markers remain.

## D1 — On-disk representation of `rdfs:subClassOf`

**Decision**: Register `subClassOf` as an ordinary new predicate in `kernel.CorePredicateDefs` — `Role: "edge"`, `Merge: core.MergeUnion`, `Aligned: "rdfs:subClassOf"`. A type's base-type references are recorded as ordinary edges on its Class node: `core.Link{Predicate: "subClassOf", Target: "<base-type-name>"}`, one edge per declared base type. Because `Role` is `"edge"` (not `"link"`), these render as flat bullets (`- subClassOf:: [[Node]]`) in the node's body, not grouped under their own heading — consistent with how other structural/semantic RDF-aligned predicates (`broader`, `isPartOf`, `requires`, …) already render (per `internal/core/markdown.go`'s existing edge-role/link-role partition, spec 013).

**Implementation-time correction**: The predicate's own *name* (the bullet's `<name>::` key, and `kernel.CorePredicateDefs`'s map key) is the plain camelCase `subClassOf`, not the literal colon-bearing `rdfs:subClassOf` string this document's examples originally showed — `internal/core/markdown.go`'s `listItemPattern` regex only accepts `\w+` for a bullet's predicate name, so a colon there fails to parse entirely (silently dropping the edge) and would require a change to `internal/core` to support, contradicting this document's own "zero changes to `internal/core`" decision. Every other RDF/schema.org-aligned predicate already in `CorePredicateDefs` follows exactly this split (plain local name, `Aligned` carries the prefixed vocabulary term) — `subClassOf`/`Aligned: "rdfs:subClassOf"` simply follows that existing precedent. Discovered and corrected during `/speckit-implement`; every other document in this feature (data-model.md, contracts/, quickstart.md) has been updated to match.

**Rationale**: `internal/core`'s Edges/Link model is already fully generic — any registered predicate can appear as a node edge, and `renderEdges`/`walkNodeBody` already round-trip arbitrary edges without any predicate-specific code. Registering a new predicate in `kernel` (a subpackage of `internal/app/schema`) is the only change needed to make `rdfs:subClassOf` a first-class, machine-readable relationship — zero changes to `internal/core`. This was confirmed against the user's explicit direction (chose "dedicated `rdfs:subClassOf` edge" over reusing the `## Requires`/`## Optional` target-name lists) and against the stated constraint that the feature "impacts only `internal/app/schema`."

**Alternatives considered**:
- *Overload the `## Requires`/`## Optional` target lists* (a base-type reference is just another entry in `Required`/`Optional`, disambiguated at resolve time by checking whether the name matches a known type instead of a known predicate): rejected — the user explicitly preferred a dedicated edge; it also introduces a real (if rare) ambiguity if a predicate and a type ever share a name, and buries a structural relationship (type hierarchy) inside a list whose meaning today is "predicates an instance must/may carry."
- *A new `## SubClassOf` heading (`Role: "link"`)*: rejected in favor of flat `Role: "edge"` rendering — RDFS `subClassOf` is conventionally a direct triple, not a labeled group, and every other structural/semantic predicate already registered (`broader`, `isPartOf`, `requires`, `replaces`, …) uses `Role: "edge"`, so this keeps the new predicate consistent with its closest existing peers rather than inventing a third rendering shape.

## D2 — Where inheritance resolution happens, and how it stays "impacts only `internal/app/schema`"

**Decision**: `core.TypeDef`'s public shape (`Merge`, `Required`, `Optional`, `Description`) is unchanged. `internal/app/schema/service.resolveTypes` (and `Seed`) decode each type document's `rdfs:subClassOf` edges into a private, package-local intermediate (a raw type record carrying `merge`/`required`/`optional`/`subClassOf`/`description`, not exported and never exposed to `core.Index`). A second, purely in-memory pass recursively flattens each type's effective `Required`/`Optional` by unioning in every (transitively resolved) base type's own effective sets, memoizing already-resolved types and tracking the current recursion stack to detect cycles. Only the fully flattened result is converted into `core.TypeDef` and placed into the `core.Index.Types` map that `Resolve` returns.

**Rationale**: Every existing consumer of `core.Index.Types` — most importantly `arc lint`'s `checkTypeRequires`/`checkTypeOptional` (`internal/app/lint/service/rules_type_conformance.go`) — reads `def.Required`/`def.Optional` directly off one `TypeDef` with no notion of hierarchy. Because those fields already contain the complete, inherited contract by the time `Resolve` returns, lint (and any other schema-contract consumer, satisfying spec FR-009) needs zero code changes. This is a direct implementation of the user's instruction: "resolve `rdfs:subClassOf` relation during the schema indexing to keep the flat structure in memory... this approach impacts only `internal/app/schema`."

**Alternatives considered**:
- *Add a `SubClassOf []string` field directly to `core.TypeDef`, and make lint (or every consumer) walk it*: rejected — this leaks the hierarchy concept into `internal/core` and every consumer package, directly contradicting "impacts only `internal/app/schema`" and forcing a change to `internal/app/lint`.
- *Lazy/on-demand resolution (compute effective contract each time a consumer asks)*: rejected — "keep the flat structure in memory" plus "may require multiple passes to build an index" both point at eager, one-time resolution during `Resolve`/`Seed`, not repeated recursive walks at check time.

## D3 — Multi-pass resolution algorithm

**Decision**: Pass 1 parses every `_schema/types/*.md` document (order-independent) into the raw intermediate record described in D2, keyed by type name. Pass 2 resolves each type's effective contract via memoized depth-first recursion: to resolve type `T`, start from `T`'s own raw `Required`/`Optional`; for each of `T`'s `subClassOf` targets, look it up in the raw map — if absent, record an unresolved-base-type failure for `T`; if present and currently on the active recursion stack, record a cycle failure; otherwise recursively resolve it (or reuse its memoized result) and union its effective `Required`/`Optional` into `T`'s. A predicate that is required anywhere in the ancestor chain is required in the effective result even if some other ancestor (or `T` itself) lists it as merely optional (spec FR-007) — implemented by always adding required predicates to the required set and only adding a predicate to the optional set if it is not already in the required set.

**Rationale**: Directly satisfies FR-003 through FR-007 (union, transitivity, de-duplication, required-wins) and FR-010/FR-011 (cycle / unresolved-reference detection) in a single linear-time pass over the type set (each type resolved once, memoized), matching the user's explicit "may require multiple passes to build an index" framing (decode pass + resolve pass) without unbounded re-computation.

**Alternatives considered**: A naive resolver that re-walks the full ancestor chain from scratch for every type (no memoization) — rejected as unnecessary work; still correct, but the memoized version is no more complex to write and scales better for wide hierarchies (SC-003's ≥3-base, ≥4-level bar).

## D4 — Cycle and unresolved-base-type reporting

**Decision**: A cycle or unresolved base-type reference causes `Resolve` to return an error, via two new `faults` types in `internal/app/schema/service/errors.go`: `ErrSchemaCycle` (names the type where the cycle was detected) and `ErrSchemaUnresolvedBase` (names the type and the unresolved base-type reference) — following the exact pattern already established by `ErrSchemaInvalid`/`ErrSchemaMissing`. `Seed`'s own built-in data is exempt in practice (it is fixed, valid-by-construction data, exactly as today's `Seed` comment already documents for other fields).

**Rationale**: Matches the existing precedent (spec 011: "the absence of a valid schema causes failing") and requires no change to `internal/app/lint`'s violation model — a broken hierarchy fails schema loading before any command (lint or otherwise) gets a chance to run, satisfying FR-010/FR-011 and SC-004 without inventing a second, lint-specific reporting path for a problem that is really "this graph's schema is invalid," identical in kind to a malformed `role`/`merge` field today.

**Alternatives considered**: Surfacing cycles/unresolved references as `arc lint` violations instead of load-time errors — rejected; `arc lint`'s violations are about a *node's* non-conformance to an otherwise-valid schema, whereas a cyclic or dangling `rdfs:subClassOf` chain means the schema itself has no well-defined meaning to check nodes against, which is exactly the class of problem `Resolve` already refuses to proceed past for other schema-validity issues (spec FR-014 precedent).

## D5 — The `Node` universal base type and its scope

**Decision**: `kernel.CoreTypeDefs` gains a new `"Node"` entry: `Required: ["published", "created"]`, `Optional: ["tags", "text", "updated", "scoreZ", "scoreC"]`. The resolution pass in D3 always treats every type *except* `Node`, `Property`, and `Class` as if it declared `rdfs:subClassOf → Node`, whether or not that edge is actually present in the type's own document — i.e., `source`, `entity`, `resource`, `timeline`, and any future custom type a graph maintainer registers, all implicitly inherit `Node`'s contract. `Property` and `Class` (the schema meta-types) are excluded: their contracts (`role`/`merge`/`description`; `merge`/`description`) describe predicate/type schema documents, not content, and have no natural relationship to `tags`/`text`/`published`/`created`/etc. `Seed`'s generated `source`/`entity`/`resource`/`timeline` documents additionally carry an *explicit* `rdfs:subClassOf → Node` edge (redundant with the implicit rule, written for the same self-describing-schema reason spec 011 already establishes: readable without running any tool or consulting source code), and each of their `Required`/`Optional` lists drops every predicate now supplied by `Node` (`tags`, `text`, `published`, `created`, `updated`, `scoreZ`, `scoreC` — never `indexed`, `mentions`, or `mentionedIn`, which stay type-specific and are not part of `Node`'s list).

**Rationale**: Directly implements the user's explicit instruction, resolved against the two clarifying questions asked before this document was written (dedicated edge; content types only, not `Property`/`Class`). Dropping the now-redundant entries from each content type's own `Required`/`Optional` is the literal point of the feature (spec SC-001: "writes that shared contract exactly once").

**Consequence flagged for implementation** (not a new question — a call-out for the eventual `tasks.md`/implementation review): `entity` and `resource` currently list `published`/`created` only under `Optional`; after this change both become `Required` for those types via `Node`. `timeline` does not list `published` at all today; after this change it becomes newly `Required` for `timeline` too, even though a period-index node's own "publication date" is not an obviously meaningful concept the way it is for `source`. This is implemented exactly as instructed ("for all core types," confirmed as the four content types) — any existing graph's `timeline` nodes will need a `published` value (or a value convention decided at implementation time, e.g. mirroring `created`) to keep passing `arc lint` once this ships. Flagged here so it is visible at review time rather than discovered only when `lint` fixtures start failing.

**Alternatives considered**: Applying `Node` to all six `CoreTypeDefs` entries including `Property`/`Class` — rejected per the user's explicit answer to the Node-scope clarifying question.

## D6 — `Node`'s own `Merge` value

**Decision**: `Node.Merge: core.MergeUnion`, matching `entity`'s and `Property`'s existing convention for types whose own instances are expected to accumulate rather than stay fixed. In practice this value is inert: no node's own `@type` is ever `"Node"` (it exists only to be inherited from), so `arc apply`'s whole-node merge dispatch (keyed by `index.Types[node.Type].Merge`) never looks it up. The field is populated only because `core.TypeDef`/the `Class` node format requires a `merge` value on every type document, per `decodeTypeDef`'s existing validation.

**Rationale**: Keeps `Node`'s schema document conformant with the existing, unmodified `Class` type contract (`Required: ["merge", "description"]`) with the least surprising default, without implying anything about how a (non-existent) direct `Node` instance would merge.

**Alternatives considered**: Leaving `Merge` unset/zero-value — rejected, since `decodeTypeDef`'s existing validation already requires a valid `MergeOp` string on every type document; `Seed`'s own output must satisfy the same contract it seeds, or `arc init` followed immediately by `arc lint`/`Resolve` would fail against the tool's own seeded schema.
