# Bug Report: Extraction-boundary edges become dangling references when applied to a graph that lacks the excluded targets

**Type**: Spec gap
**Severity**: High
**Feature**: 007-arc-subgraph (`arc subgraph`)
**Reported**: 2026-07-05
**Status**: Patched
**Patched**: 2026-07-05

## Description

`arc subgraph` renders each included node's body verbatim, including its
full `Edges`/`Links` list. When an included node's edge points at a target
that exists in the source graph but falls *outside* the extraction
boundary (excluded by `--depth`, a truncation cap, or
`--kind`/`--tag`/`--attr`), that edge is still present in the rendered
body — it belongs to the included node's own content, not to the excluded
target, so nothing in the current design strips or substitutes it.

`arc apply` performs no link-resolution validation (only `arc lint` does),
so the patch is accepted with no structural error — but the target graph
now contains a dangling reference: a node whose declared `Edges`/`Links`
name an id with no corresponding file. This is invisible when re-applying
into the *same* graph the subgraph was extracted from (the boundary node's
file already exists there), but maximally visible when applying into an
empty or otherwise different graph, since nothing exists there to resolve
against.

This is distinct from the dangling-link case FR-006 already covers: FR-006
is about a target that does not exist *anywhere in the source graph*
(genuinely broken data, correctly excluded with nothing to extract). This
bug is about a target that **does** exist in the source graph's index —
it was simply not selected for inclusion in this particular extraction.

**Agreed resolution** (captured for `/speckit.bugfix.patch`): introduce an
opt-in `--stubs` flag. When passed, for every edge/link from an included
node whose target exists in the source graph's index but was not itself
selected for inclusion, `arc subgraph` emits a minimal stub node section
for it — `kind` and `id` only, no other attributes, empty body (no
`Text`/`Notes`/`Edges`/`Links` of its own, and critically **no
recursion** — a stub's own connections are never traversed, or the
`--depth` bound becomes meaningless). This guarantees every link in the
patch resolves to a real file the moment it is applied, even to an empty
graph. Without `--stubs`, today's behavior (boundary edges rendered
verbatim, boundary targets excluded) is unchanged — this is additive, not
a behavior change to the default path.

## Artifact Traceability

### spec.md

- **Affected user stories**: User Story 1 (P1) — "turning 'everything
  relevant to node X' from a manual, error-prone file hunt into a single
  command" implicitly assumes the extracted document is self-contained
  enough to be useful standalone; User Story 2 (P2) — depth control is the
  mechanism that creates the boundary this bug crosses.
- **Affected requirements**:
  - **FR-006** — covers only the "target does not correspond to an
    existing node" case; does not address a target that exists in the
    source graph but was excluded by the extraction boundary itself
    (depth/cap/filter). No requirement currently governs referential
    integrity across that boundary.
  - **FR-007/FR-008** — "body content unmodified" and "structurally
    complete... applies... without a structural parsing failure" are both
    satisfied today, which is exactly why this gap was not caught by
    existing acceptance criteria: structural validity and referential
    integrity are different properties, and only the former is specified.
  - **FR-015** — the truncation-cap mechanism is one of the three ways a
    real, existing node ends up excluded from the extraction while still
    being referenced by an included node's edges; the spec documents the
    truncation itself (count, "most-connected kept") but not its
    referential-integrity consequence.
- **Gap identified**: No functional requirement addresses what happens to
  a structural link whose target is real but extraction-excluded. FR-006's
  wording ("does not correspond to an existing node") reads narrowly
  enough that it was reasonably interpreted at implementation time as not
  covering this case (confirmed against research.md D3's "dangling target
  (absent from the index) is silently skipped" — the BFS traversal itself
  only ever consults the index, so a target present in the index was never
  treated as "dangling" by the implementation, correctly per the letter of
  FR-006, but leaving this scenario unspecified).

## Root Cause Analysis

The spec (FR-006) and its implementation (research.md D3, `service.Subgraph`)
correctly handle "target absent from the whole graph" but never considered
"target present in the graph but absent from *this extraction*" as a
distinct case requiring its own rule. This is a scope gap in the original
`/speckit.specify`/`/speckit.clarify` pass, not an implementation error —
the code does exactly what FR-006/FR-014/FR-015 asked for. It surfaced now
because the only round-trip testing performed (automated and manual) always
re-applied extracted output back into the graph it came from, where the
excluded-but-referenced nodes already exist as files and mask the dangling
reference. Applying into a graph that does not already contain those nodes
— the exact "iterative discoverability" workflow this feature exists to
support — was never exercised until now.
