# Bug Report: `RenderPatch` reuses `RenderNode`'s `## Label` heading grouping, corrupting the patch document's fixed H1=type/H2=id structure

**Type**: Spec gap (with a direct Implementation drift consequence against the external ARCNET-CORE spec)
**Severity**: Critical
**Feature**: 013-predicate-role-rendering (branch `013-predicate-role-rendering`)
**Reported**: 2026-07-09
**Status**: Patched
**Patched**: 2026-07-09

## Description

This feature introduced schema-driven `## Label` heading grouping for `link`-role predicate occurrences,
implemented once in `internal/core/markdown.go`'s `renderNodeBody`/`renderEdges` and shared unconditionally
by both `RenderNode` (a standalone graph node file) and `RenderPatch` (a patch-exchange document).

That sharing is correct for `RenderNode` — ARCNET-CORE §5 explicitly specifies that a `link`-role predicate's
occurrences render as their own `## Predicate` block within a graph node file's body. But it is **wrong** for
`RenderPatch`: ARCNET-CORE §14.2 fixes a patch document's structure as `H1 = node's @type`, `H2 = node's @id`,
and states explicitly:

> "Markdown headings are reserved for type and identity; node bodies use bold labels, never headings."

Because `RenderPatch` calls the same `renderNodeBody` as `RenderNode`, `arc subgraph`/`arc serve`'s
`subgraph_get` now emit `## Mentions`-style headings *inside* a patch's per-node body — indistinguishable,
to a human or a patch-aware tool not written by this codebase, from the `## <ID>` headings that carry the
document's actual node-identity structure. This corrupts the patch format's own fixed, compact structural
contract (confirmed live: `arc subgraph MixedShapeEntity --depth 0` on a real `arc init`-seeded graph emits
`## Mentions` under the `## MixedShapeEntity` node section, both at the same heading level).

`arc`'s own parser (`parsePatchBody`/`isSectionBoundary`) happens to tolerate this without literally breaking
round-trip, because it only treats an H2 as a new-node boundary when immediately followed by a yaml fence.
That parser leniency masks the structural violation rather than fixing it — an external, spec-conforming
patch consumer is not obligated to apply the same heuristic, and a human/LLM reader loses the "H2 = one node"
invariant CORE §14.2 promises.

The fix requires the graph and patch codecs to diverge for link-role rendering: `RenderNode` keeps `## Label`
headings (CORE §5, unchanged); `RenderPatch`'s per-node body must render link-role groups under a bold label
(`**Label**`) instead, never a heading — matching CORE §14.2 and this codebase's own pre-existing bold-label
parsing precedent (`blockTitle`/`boldLabel`, BUG-003).

  distinguishing `RenderNode` (graph node file, CORE §5) from `RenderPatch` (patch-exchange document, CORE

- **Acceptance Scenario affected**: US1 Acceptance Scenario 1 — "those occurrences render under a

- **Gap identified**: spec.md 013 never acknowledges that `RenderNode` and `RenderPatch` are two distinct
  serialization targets with different body-formatting rules per the external ARCNET-CORE spec (§5 vs
  §14.2). It implicitly assumed one rendering algorithm suffices for both, which was true before this
  feature (both formats rendered every predicate as an identical flat bulleted list) and stopped being true
  the moment `## Label` grouping was introduced.

## Root Cause Analysis

The `/speckit-plan` design phase (research.md D1/D2) chose to share `renderNodeBody` unconditionally between
`RenderNode` and `RenderPatch` because both already called the identical function before this feature, when
every predicate rendered as one uniform flat bulleted list regardless of format — a shape that genuinely was
format-agnostic. Introducing `## Label` heading grouping for `link`-role predicates broke that pre-existing
invariant without re-deriving whether the shared-function assumption still held against the external
ARCNET-CORE spec. CORE §5 (graph node files) and CORE §14.2 (patch documents) diverge specifically on how
`link`-role predicate groups render — a divergence that was invisible pre-feature and became load-bearing the
moment heading-based grouping was introduced. Neither spec.md, research.md, nor
contracts/render-shape-contract.md re-checked this constraint against ARCNET-CORE during design; the plan
treated `internal/core`'s Markdown codec as a single, format-agnostic concern rather than two distinct
serializations sharing only their front-matter/predicate-schema machinery.

   and the corrected `RenderPatch` bold-label shape.
