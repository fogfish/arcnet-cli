# Bug Report: `arc init` over-reports progress and renders it with broken alignment

**Type**: Implementation drift (formatting/hash) + Spec/plan gap (default verbosity, hint text)
**Severity**: Medium
**Feature**: `002-arc-init` (branch `002-arc-init`)
**Reported**: 2026-07-02
**Status**: Fixed
**Patched**: 2026-07-02
**Fixed**: 2026-07-02

## Description

Running `arc init` in default (non-quiet, non-verbose) mode prints a per-step
progress line (`▶ <label>` then `✅ <label> (<elapsed>)`) for all four
internal steps — checking git availability, `git init`, `git add -A`, `git
commit` — every one styled green with a checkmark emoji. The final one-line
success confirmation and the `PostRunE` next-step hint are also miscolored
and, critically, visually broken: each line after the first appears
indented by an amount matching the length of the previous line, because the
line breaks are corrupted.

Reported (verbatim) actual output:

```text
▶ Checking git availability
✅ Checking git availability (14ms)
                                   ▶ Initializing git repository
✅ Initializing git repository (21ms)
                                     ▶ Staging graph files
✅ Staging graph files (16ms)
                             ▶ Committing initial graph
✅ Committing initial graph (33ms)
                                  ✅ Initialized empty knowledge graph at /Users/kolesnik/devel/go/src/github.com/fogfish/gxx (commit de9aae8533ae41b26622ef1c0e2c223d21a91e79)
                                                                                                                                          (use "arc list" to see what's in your new graph)
```

User-reported defects, verbatim:

1. Overused emoji and green color.
2. The default report is verbose (progress is shown even without `-v`).
3. Wrong formatting: the report is not aligned and hard to read.
4. Too many sub-sections in the report; it should be concise.

User-reported desired behavior:

1. Default `arc init` output is a single line, in plain/white text (no
   per-step progress): `✅ Initialized empty knowledge graph at <path>
   (commit <short-hash>)`.
2. Always use **short** commit IDs, not the full 40-character SHA.
3. The `PostRunE` next-step hint should suggest `arc apply <patch.md>`, not
   `arc list`.
4. Under `--verbose`/`-v`, show the per-step progress in **gray/faint**
   text, for: git availability status, git repository status, and
   committing the empty graph.

## Root Cause Analysis

Four distinct root causes, three of them genuine implementation bugs and one
a design decision that no longer matches the desired UX:

### 1. Alignment/formatting corruption (implementation bug, CONFIRMED)

`internal/bios/reporter.go`'s `stderrReporter.Done`/`Error`, and
`cmd/arc/ctrl/init.go`'s `humanInitPrinter.Show`, all build a string with an
embedded/trailing `"\n"` and then pass that whole string into
`lipgloss.Style.Render(...)`:

```go
text := fmt.Sprintf("%s%s (%s)\n", SCHEMA.IconOK, label, elapsed.Round(time.Millisecond))
fmt.Fprint(r.w, SCHEMA.StatusOK.Render(text))
```

Reproduced in isolation: `lipgloss.NewStyle().Foreground(...).Render("hello
(14ms)\n")` returns `"hello (14ms)\n            "` — lipgloss treats
multi-line input as a block and pads every line to the block's width
(including the phantom empty line produced by the trailing `\n`), so the
rendered value ends in trailing **spaces**, not a newline. The next
`Start()`/`Done()` call's own leading text is then written immediately
after those spaces, on the same terminal line, which is exactly the
"indented by the previous line's length" pattern in the reported output.

**Locus**: `internal/bios/reporter.go` (`stderrReporter.Done`, `.Error`),
`cmd/arc/ctrl/init.go` (`humanInitPrinter.Show`).

### 2. Reporter runs unconditionally regardless of `--verbose` (spec/plan gap)

`cmd/arc/ctrl/init.go`'s `RunE` always constructs `bios.NewReporter(bios.Quiet, false)` — the *only* switch is `--quiet`; there is no branch on
`--verbose` at all, so every step's `Start`/`Done` prints by default. This
matches what was actually decided in `research.md` D2 at the time (`silentReporter` used for `--quiet`/`--silent` only), so this is not a
divergence from an existing written decision — the decision itself needs
revising to gate progress behind `--verbose` instead of showing it by
default.

**Locus**: `cmd/arc/ctrl/init.go` (`RunE`, `reporter := bios.NewReporter(...)`); `specs/002-arc-init/research.md` D2.

### 3. Full commit hash instead of short hash (implementation drift, CONFIRMED)

`contracts/cli-contract.md`'s own stdout example already shows a **short**
hash — `(commit a1b2c3d)` — and `data-model.md`'s `VCS.Commit` contract says
"returns the resulting commit's hash (equivalent to `git rev-parse HEAD`
immediately after committing)" without specifying short vs. full, but the
CLI contract's literal example is short. The implementation uses the full
hash:

```go
// internal/app/ctrl/adapter/git/git.go, VCS.Commit
out, err := run(ctx, dir, "rev-parse", "HEAD")
```

This is implementation drift against `contracts/cli-contract.md`'s own
documented example output, not a spec gap.

**Locus**: `internal/app/ctrl/adapter/git/git.go` (`VCS.Commit`).

### 4. Hint text suggests `arc list`, not `arc apply` (spec/contract gap)

`contracts/cli-contract.md` hard-codes the `PostRunE` hint text as
`(use "arc list" to see what's in your new graph)`, which `cmd/arc/ctrl/init.go` reproduces verbatim. The user now wants this to read
`arc apply <patch.md>` instead, since a freshly initialized graph is empty —
`arc list` on it is not a useful next step, but `arc apply` (loading a
patch) is. **Caveat for the patch phase**: neither `arc list` nor `arc apply`
exist yet as implemented commands in this codebase (only `arc init` is
implemented so far; both are future VISION.md commands) — a hint that
suggests a not-yet-existing command in a way that looks runnable should be
weighed against DS-12's rule that a hint MUST be "a real runnable
invocation." This is a contract-level decision, not something this report
resolves.

**Locus**: `contracts/cli-contract.md` ("stdout / stderr contract" section);
`cmd/arc/ctrl/init.go` (`PostRunE`).

## Artifact Traceability

### spec.md

- **Affected user story**: US1 (Bootstrap a new graph in the current
  directory) — Acceptance Scenario 2 references "the git history" but
  spec.md does not constrain terminal presentation.
- **Affected requirements**: None directly — spec.md is intentionally
  technology/UI-agnostic (constitution: "Success criteria are
  technology-agnostic"). **SC-004** ("A user can confirm initialization
  succeeded from the command's own output alone") is still technically
  satisfied even with the broken formatting, but the broken alignment
  undermines its spirit.
- **Gap identified**: spec.md has no functional requirement governing
  default vs. verbose progress output — that responsibility was correctly
  deferred to `plan.md`/ADR 002, which is where this bug actually needs to
  be traced and fixed.

### plan.md / research.md

- **Affected sections**: `research.md` D2 ("Reporting git progress to the
  user") — currently states Start/Done framing is shown for every git
  invocation by default (gated only by `--quiet`/`--silent`), not by
  `--verbose`. This decision needs revising.
- **Affected contract**: `contracts/cli-contract.md` "stdout / stderr
  contract" section — the literal `(commit a1b2c3d)` short-hash example and
  the `arc list` hint text both need reconciling with the adapter/command
  implementation (short-hash) and the user's new hint-text request
  (`arc apply`).
- **Impact**: D2 needs an explicit rule that per-step Reporter output is
  shown only under `--verbose`/`-v` (styled faint/gray, per ADR 002 DS-05's
  `Hint`-style treatment), and is otherwise fully suppressed in default
  mode (not just under `--quiet`). The cli-contract.md stdout/stderr
  section needs the hint text and hash-length decision reconciled.

### tasks.md

- **Affected tasks**:
  - **T026** — `internal/app/ctrl/adapter/git/git.go`: implemented with
    `git rev-parse HEAD` (full hash) instead of a short hash.
  - **T035** — `cmd/arc/ctrl/init.go`: wires the `Reporter` unconditionally
    (only `--quiet`-gated) and renders both the reporter lines and the final
    printer output via a newline-corrupting `lipgloss.Render()` call.
  - **T015-T017** (`internal/bios/reporter.go`, `theme.go`) — `stderrReporter.Done`/`.Error` embed a trailing `\n` inside the
    `Render()`-ed string, corrupting line breaks.
- **False completions**: T026, T035, and the `internal/bios` reporter
  portion of T015-T017 are marked `[X]` complete but do not match the
  now-clarified UX expectations and contain a genuine rendering bug; they
  need reopening.
- **Missing tasks**: A task to gate `Reporter` visibility behind
  `--verbose` (currently no such task exists — D2 never called for it), and
  a task to reconcile the `PostRunE` hint text with the user's
  `arc apply`-suggestion request.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to update `research.md` D2 and
   `contracts/cli-contract.md` (verbose-gated Reporter output, short commit
   hash, revised hint text) and reopen tasks T015-T017, T026, T035 in
   `tasks.md`.
2. Run `/speckit.bugfix.verify` to confirm consistency across the updated
   spec artifacts.
3. Resume `/speckit.implement` to apply the code fix:
   - Stop calling `lipgloss.Style.Render()` on strings that already contain
     `\n`; render the styled span and the newline separately.
   - Gate `bios.NewReporter` selection on `--verbose` (faint/gray style) vs.
     default (silent), independent of `--quiet`.
   - Switch `VCS.Commit` to `git rev-parse --short HEAD`.
   - Update the `PostRunE` hint text per the reconciled contract.
