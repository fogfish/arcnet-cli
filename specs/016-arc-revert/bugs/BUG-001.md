# Bug Report: `arc revert` refuses after a revert‚Üíreapply‚Üírevert cycle ("more than one ingest commit found")

**Type**: Spec gap
**Severity**: High
**Feature**: 016-arc-revert (branch `016-arc-revert`)
**Reported**: 2026-07-12
**Status**: Patched
**Patched**: 2026-07-12

## Description

After a document has been reverted once (whole-commit path) and then re-applied via `arc apply` (a normal, unremarkable operation ‚Äî the source node no longer exists, so `arc apply`'s own idempotency check does not block it), a second `arc revert` of the same `source-id` refuses outright:

```text
$ arc revert dmitry-2026-graph
Retract dmitry-2026-graph's contribution from the graph? [y/N] y

 ‚ùå more than one ingest commit found for dmitry-2026-graph ‚Äî run `arc lint` to check the Source-Id: trailer invariant
   Run `arc help revert` for guidance.
```

Reported git history at the time of failure:

```text
39a111e (HEAD -> main) graph(ingest): dmitry-2026-graph ‚Äî
aa2a551 Revert "graph(ingest): dmitry-2026-graph ‚Äî"
ef64149 graph(ingest): dmitry-2026-graph ‚Äî
ceacc7c graph(init): empty knowledge graph
```

Both `ef64149` and `39a111e` are genuine `arc apply` ingest commits for `dmitry-2026-graph`, each carrying its own `Source-Id: dmitry-2026-graph` trailer ‚Äî `ef64149` from the original apply, `39a111e` from the re-apply after `aa2a551` (`git revert`) removed the first ingest's contribution. `service.Revert`'s `CommitsMatching(dir, "Source-Id: dmitry-2026-graph")` call correctly finds both, and ‚Äî per its own D1 design ‚Äî treats "more than one match" as an unrecoverable integrity anomaly and refuses. It is not an anomaly here: it is the expected, mechanical result of retract-then-reapply, a sequence nothing in the codebase prevents and that `arc apply`'s own re-apply behavior actively permits.

## Artifact Traceability

### spec.md

- **Affected requirements**: FR-001 ("locate its original ingest commit from the graph's git history") and FR-002 ("refuse... when the given identifier does not correspond to any ingest commit") both implicitly assume at most one ingest commit exists per `Source-Id` for the lifetime of the graph. Neither FR anticipates more than one *valid* match.
- **Key Entities > Ingest Commit**: defined as "*The* single git commit `arc apply` produced when the patch was originally applied" ‚Äî singular, with no accounting for a later re-apply producing a second one.
- **Out of Scope > "Re-applying a retracted patch"**: explicitly scopes out *restoring content* after a retraction, but does not address the knock-on effect on a *subsequent* `arc revert` invocation's own lookup ‚Äî re-applying was assumed to be a dead end for this feature's own concerns, not a state that a later revert must still cope with.
- **Gap identified**: the spec never states what `arc revert` should do when a `source-id` has more than one ingest commit in history because it was reverted and re-applied at least once. The current implementation (research.md D1) collapses this legitimate case into the same refusal path as genuine corruption (e.g. a hand-edited or duplicated `Source-Id:` trailer), which the format is not actually supposed to allow to happen in the "several *simultaneously live* ingest commits" sense ‚Äî but multiple *historical* ingest commits, where all but the newest have already been fully retracted, is not corruption at all.

### plan.md / research.md

- **Affected section**: research.md **D1 ‚Äî Locating the ingest commit**, specifically: *"More than one match is an integrity anomaly the format guarantees cannot happen (spec 003 FR-011: exactly one commit per successful apply) ‚Äî treated as a hard refuse..."* This reasoning is only valid for a *single* apply lifecycle; spec 003 FR-011 guarantees uniqueness "per successful apply" at a point in time, not across a graph history that this very feature (`arc revert`) introduces the ability to rewind and repeat.
- **Impact**: D1's decision needs a third case ‚Äî more than one match, but only one of them corresponds to the *currently active* ingest (i.e., every earlier match has itself already been retracted) ‚Äî that resolves to the newest match rather than refusing. The genuinely-anomalous case (two or more *simultaneously un-retracted* ingest commits for the same `source-id`) should still refuse, but that is a narrower, rarer condition than "more than one match at all."

### tasks.md

- **Affected tasks**: T017/T020 (implement `service.Revert`'s D1 ingest-commit lookup in `internal/app/graph/service/revert.go`), T023 (unit tests for D1's decision branches in `internal/app/graph/service/revert_test.go`) ‚Äî all marked `[X]` complete, but T023's coverage only exercises "zero matches" and "two matches ‚Üí refuse," never "two matches, older one already retracted ‚Üí succeed."
- **False completions**: T020/T023 should be considered incomplete with respect to this newly-identified branch of D1's own decision logic; they satisfied the D1 as originally written, but D1 itself needs revision.
- **Missing tasks**: a new task to (1) revise D1's decision rule to disambiguate "currently active" ingest commits from "historical, already-retracted" ones among multiple `CommitsMatching` results, using each candidate's own `sources/<id>.md` tracked-state history (or equivalent) rather than refusing on count alone, and (2) unit tests covering the retract‚Üíreapply‚Üírevert cycle end-to-end (both whole-commit-then-reapply and per-node-then-reapply starting points).

## Root Cause Analysis

This is a spec gap compounded by an implementation assumption that outlived its premise. Spec 003 (`arc apply`) established "exactly one ingest commit per `Source-Id`" as an invariant when no command could ever cause a document to be untracked and reintroduced. Spec 016 (`arc revert`) itself breaks that premise ‚Äî retracting a document is the whole point of the feature ‚Äî yet D1 was designed by directly reusing spec 003's uniqueness invariant (research.md D1's own rationale cites spec 003 FR-011 verbatim) without re-examining whether it still holds once `arc revert` exists in the same codebase. The "Out of Scope: Re-applying a retracted patch" note in spec.md shows the interaction was considered at a shallow level (whether *content restoration* is this feature's job ‚Äî correctly, no) but not followed through to its consequence for `arc revert`'s own future invocations against the same `source-id`.

2. Run `/speckit.bugfix.verify` to confirm the patched spec, plan, and tasks stay consistent.
